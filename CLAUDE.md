# Vitta

Health companion app: food, workouts, water, sleep, body weight and reminders. Flutter, SDK ^3.10.7.

## Backend

Supabase (Postgres). Chosen over Firebase because the diet domain is inherently relational — a day's macro totals are a `SUM()`/`GROUP BY` over `food_logs` joined to `foods`, which Postgres does natively; Firestore would push that aggregation into the client. `supabase_flutter` talks to it directly from the app (no custom backend server).

- `supabase/schema.sql`: source of truth for the `foods` and `food_logs` tables, run manually in the Supabase SQL editor (no migration tooling yet). Both tables have row-level security scoped to `auth.uid()`, so each user only ever sees their own rows.
- Auth is anonymous (`signInAnonymously`, wired in `lib/app/bootstrap.dart`) so RLS has a stable `user_id` without building a login UI yet. Swapping in real accounts later means adding email/OAuth sign-in on top of the same anonymous-session-per-device bootstrap — existing anonymous data can be linked via Supabase's `linkIdentity`.
- `main()` only calls `bootstrap(appRunner: () => runApp(const VittaApp()))` — infrastructure bootstrap (dotenv, Sentry init, Supabase init, anonymous sign-in, `setupDependencies()`) is deliberately kept out of `AppCubit`/widget code so it stays a single, synchronously-ordered sequence and `AppCubit` stays pure presentation state. `appRunner` is threaded through rather than `main()` calling `bootstrap()` then `runApp()` as two separate statements, so every init step (not just the widget tree) runs inside Sentry's zone — see Error tracking below.
- Credentials live in a gitignored `.env` (`SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEY`, `SENTRY_DSN`), loaded via `flutter_dotenv` and read through `lib/app/core/env/env.dart`. Copy `.env.example` to `.env` and fill in a real project's values — the placeholder committed values won't connect to anything.

## Error tracking

`sentry_flutter`, initialized inside `bootstrap()` (`lib/app/bootstrap.dart`) via `SentryFlutter.init((options) => options.dsn = Env.sentryDsn, appRunner: ...)` — the package's own `runZonedGuarded`/`FlutterError.onError` wiring is what actually captures uncaught exceptions, so nothing else in the app needs to touch `runZonedGuarded` directly. `appRunner` wraps the rest of bootstrap (Supabase init, Hive init, `setupDependencies()`) and finally calls the `appRunner` passed in from `main()`, so a startup crash in any of those steps is captured too, not just runtime errors after the widget tree is up.

**Errors surfaced through `Result`/`VTError` (network failures, validation, ...) are not reported to Sentry.** Those are expected, already-handled failure paths shown to the user via an error toast — forwarding every one to Sentry would mostly report noise (e.g. a user opening Diet with no internet connection). Sentry is scoped to genuinely unexpected crashes only. If a specific `VTError` case turns out to be worth tracking, capture it explicitly at that call site (`Sentry.captureException(error.cause)`) rather than wiring every `Failure` generically.

## Logging

Navigation and user-action logging goes through one centralized `LoggingService` (`core/services/logging/`, see issue #46) so a new log sink is added in exactly one place. The service holds a `List<LogDestination>` and fans every `LogEntry` (a `category` like `'navigation'`/`'action'`, a `message`, a `LogLevel`, and a `data` map) out to all of them — currently `ConsoleLogDestination` (`debugPrint`, gated on `kDebugMode`, so log lines show in the `flutter run` **terminal** — `dart:developer`'s `log` was tried first but only surfaces in DevTools/the IDE console, not the terminal) and `SentryLogDestination` (a Sentry **breadcrumb**, so the last actions/navigations ride along on any crash report — this is the one place breadcrumbs are added, complementing the crash-only `Sentry.captureException` above). Adding a remote/analytics sink later means one new `LogDestination` implementation added to the list, nothing else. `LogDestination` is an abstract class — the one other deliberate interface besides the vendor adapters, because "send logs to different places" is a real polymorphism point.

**Call sites never see `LoggingService` — they use the static `Log` facade** (`core/services/logging/log.dart`): `Log.action('<snake_case_event>', data: {...})` and `Log.navigation(...)`, so nothing has to inject a logger through its constructor. `Log` holds one swappable `LoggingService` (defaulting to a no-op with zero destinations, so code that logs is safe in unit tests that don't care); `setupDependencies` wires the real one once via `Log.service = LoggingService(destinations: const [...])`. It's the one sanctioned global-static in the app — deliberately not a GetIt registration, since a logger is a pure cross-cutting sink with no swap boundary worth a `G<>()` lookup.

**Navigation is logged automatically** by `LoggingNavigatorObserver` (a `NavigatorObserver` wired into `GoRouter(observers: ...)` in `app_router.dart`), which calls `Log.navigation(action: 'push'|'pop'|'replace', route: route.settings.name)` on every route change — no per-page code. **User actions are logged explicitly** from the cubit that performs them, calling `Log.action('<snake_case_event>', data: {...})` on the success path — the write actions across the app are instrumented this way: `AuthCubit` (`sign_in`/`sign_up`/`sign_out`), `FoodSearchCubit` (`food_logged`), `DietCubit` (`food_log_deleted`/`food_log_updated`), `WaterCubit` (`water_logged`/`water_log_deleted`/`water_goal_changed`), `SleepCubit` (`sleep_logged`/`sleep_log_deleted`), `MacroGoalsCubit` (`macro_goals_saved`), `OnboardingCubit` (`onboarding_completed`), and `AppCubit` (`theme_changed`/`locale_changed`/`unit_system_changed`). Instrument a new notable action the same way — event name in `snake_case`, on the success branch, never from the widget layer. Tests that assert a log fired swap in a mock with `useMockLog()` (`test/fixtures/logging_fixture.dart`, which resets `Log.service` on teardown) and `verify` on it; tests that don't care need no setup because the default `Log.service` is a harmless no-op.

## Food data

`foods` is a catalog shared across every user, not a per-user table — RLS allows any authenticated user to read every row (see `Anyone can read foods` in `supabase/schema.sql`), while insert/update/delete stay scoped to the row's own `user_id`. `DietRepository.searchFoods` tries the shared catalog first (`SupabaseDietDataSource.searchCatalog`, an `ilike` on `name`) and falls back to the [Open Food Facts](https://world.openfoodfacts.org) public API (`OpenFoodFactsDataSource`, `lib/app/data/diet/datasources/http/`) whenever the catalog is **sparse**, not only when it's empty: below `_sparseCatalogThreshold` (5) catalog hits it also queries OFF and merges the two, deduping by `barcode` (catalog rows first since they're popularity-ranked, then OFF rows not already present) — a Portuguese term the OFF *import* barely covers (its country filter is US-heavy) still gets augmented by OFF's live search, which spans the whole OFF database. An empty catalog returns the OFF result directly (including its error); a non-empty-but-sparse catalog whose OFF augmentation fails still returns the catalog hits rather than erroring — there's still no proprietary/mirrored copy of the full OFF export (several GB, would dwarf Supabase's free-tier storage), so the catalog only grows organically from what real users actually log. A search result (from either source) is only persisted (into the shared `foods` table) once a user actually logs it; `LogFoodUseCase` does the save-then-log in one step for any `Food` without an `id`, and `SupabaseDietDataSource.saveFood` looks up an existing row by `barcode` first so the same product logged by different users reuses one row instead of duplicating it (see the unique partial index on `barcode`). Users can also add a fully custom food (name + optional brand + macros per 100g, optionally a photo) through the same flow — see the Custom food page below; custom foods become part of the shared catalog immediately, with no moderation step. A photo uploads to the `food-images` Supabase Storage bucket (public read, authenticated write — see `supabase/schema.sql`) via `SupabaseService.storage(bucket)`, the one place a datasource touches Storage directly instead of `.from(table)`; the resulting public URL is what gets stored in `foods.image_url` and rendered by the design system's `VTFoodImage` (falls back to a placeholder icon when there's no image or it fails to load).

Catalog search ranks by **popularity** (see issue #56): `searchCatalog` orders its `ilike` matches by `foods.times_logged desc` (then `name` as a stable tiebreak), so the most-logged products surface first. `times_logged` is a denormalized counter on the shared `foods` row — it has to live there because `food_logs`' RLS scopes each user to their own rows, so "how many times *anyone* logged this" can't be aggregated client-side. A `security definer` trigger on `food_logs` (`bump_food_times_logged` in `supabase/schema.sql`) keeps it in sync (+1 on insert, −1 on delete, floored at 0) — deliberately not on update, since editing a log's quantity or meal (`SupabaseDietDataSource.updateFoodLog`) leaves it the same one log, and the food it points at can't be swapped; it's `security definer` because the logging user doesn't own the (often imported, unowned) food row, so `foods`' update policy would otherwise block the bump. The signal is in-app logging only — deliberately *not* seeded from OFF's `unique_scans_n` — so it starts at 0 for the bulk-imported catalog and grows purely with real usage.

Macros (calories/protein/carbs/fat/fiber) are dedicated `numeric` columns since they're mandatory and used in checks, but **vitamins and minerals are a keyed `jsonb` blob (`foods.micronutrients`), not one column per nutrient** — there are dozens, coverage is sparse (most OFF rows only carry macros), and a new nutrient shouldn't need a schema migration. Each is a case on the `Nutrient` enum (`lib/app/domain/diet/entities/nutrient.dart`) pairing its `wireKey` (the JSONB/DB key), its `offKey` (the OFF `nutriments` key its per-100g value is read from — like `SupabaseTable`/`MealType` pair a case with its wire value), and a `NutrientUnit` (mg/µg) that only governs display. Everything is stored per-100g **in grams**, the same base unit as macros (`NutrientUnit.fromGrams` converts for display); `Food.micronutrientsPer100g` is a `Map<Nutrient, double>` parsed from the blob (unknown keys ignored, so the app tolerates nutrients it doesn't model yet), `FoodLogEntry`/`MacroTotals` scale and sum it exactly like a macro. `MacroSummaryCard` has two versions (see issue #54): it defaults to the compact macros-only view and, only when some micronutrient is actually present (`> 0`), shows an `unfold_more`/`unfold_less` icon button to the right of the calorie total that toggles the complex version — expanding a "Vitamins & minerals" section that renders only the present nutrients, which keeps the sparse data from showing as a wall of blanks. Per-nutrient **goals are deliberately not modelled yet** — `MacroGoals` still covers only calories + the four macros. Custom foods (`CustomFoodPage`) capture macros only; their micronutrient map is empty until OFF data fills it. Add a nutrient by adding an enum case plus its `nutrientLabel` l10n key — no new column, request field, or parse branch.

The diet calendar colors each logged day's dot by how close that day's calories are to the goal (see issue #51), not just "has a log". `DailyMacros.adherenceTo(MacroGoals)` returns a `GoalAdherence` (`met`/`close`/`off`) — it ratios the day's `totalCalories` against `calorieGoal` and classifies it via `GoalAdherence.forRatio` (green band ±10%, yellow band ±25%, else red — the bands are the tunable constants). **Adherence is calories-only by design** — the four macros are shown on `MacroSummaryCard` but don't factor into the day's color (a `combineWorst` fold across all macros was tried and dropped; per-macro goals aren't a strong enough signal to gate the color on). `GoalAdherence.color` maps `met`/`close`/`off` to `VTColors.green`/`warning`/`error` (used by both the calendar dots and `MacroSummaryCard`'s calorie ring — the ring falls back to `colorScheme.primary` on an empty day so a day with no log isn't painted red). This is why the calendar loads full per-day macros for the month: `DietCubit.loadMonthMacros` → `GetMacrosInRangeUseCase` → `DietRepository.getMacrosInRange` (groups `SupabaseDietDataSource.getLogsInRange`'s joined rows into a `Map<DateTime, DailyMacros>`). That chain is named for the range it takes (`from`/`to`), not for a month — the history page (see below) reuses it for a rolling 7/30/90-day window, so anything month-shaped is the caller's choice of arguments.

**The calorie goal is derived, not stored (issue #116).** `MacroGoals.calorieGoal` is a getter — `protein*4 + carbs*4 + fat*9`, fiber excluded (it's part of carbs, so counting it double-counts) — so editing a macro moves the calorie target with it and the two can never disagree. `DietGoalsLocalDataSource` therefore persists only the four macro grams; there's no calorie key. Everything downstream (`adherenceTo`, the ring, trend charts, the calendar) reads the same getter, unchanged. `calorieMin`/`calorieMax` expose the "on target" band as `calorieGoal ×` `GoalAdherence.metLowerBound`/`metUpperBound` (the same ±10% green band the calendar uses — the bounds were made public for exactly this). The goals page (`MacroGoalsPage`) is **sliders, not text fields**: one `VTLabeledSlider` per macro in its `VTColors.macro*` accent, and a `CalorieTargetCard` hero showing the derived target, the min/max band, and a protein/carbs/fat energy split (`VTDistributionBar`). Its calorie slider calls `MacroGoals.withScaledCalories`, which rescales the three energy macros proportionally to hit a new total (fiber untouched; a balanced 30/40/30 fallback when the macros are all zero and there's no ratio to keep). `MacroSummaryCard` gained an optional `onEditGoals` (the "pass no callback to disable" convention — null on the read-only history day view) rendering a `tune` button that opens the editor from the diet page.

**Diet modalities are a preset energy split, derived not stored (issue #122).** A modality — `DietModality` (`domain/diet/entities/`), cases `balanced`/`highProtein`/`lowFat`/`lowCarb`/`keto` — pairs each case with a protein/carbs/fat split (fractions of calories, e.g. balanced 0.30/0.40/0.30), the way `MealType` pairs a case with its wire value; the label lives in l10n via `dietModalityLabel` (the `nutrientLabel` pattern) so the domain stays Flutter-free. `modality.applyTo(goals)` re-splits the *current* `calorieGoal` by its ratios (fiber preserved) through `MacroGoals.fromEnergySplit`, and `DietModality.matching(goals)` recovers the modality from the grams by comparing each macro's energy share (`MacroGoals.proteinCalorieRatio`/etc.) within a 3-point tolerance — so a modality is **never persisted**: it's inferred from the same four macro grams `DietGoalsLocalDataSource` already stores, and nudging a slider off a preset simply reads back as no match (a "Custom" split). This mirrors the calorie-goal-is-derived decision above. The picker is a `DietModalitySelector` (a horizontal row of `DietModalityCard`s, each a leading icon avatar via `dietModalityIcon` over a mini `VTDistributionBar` + percentages) at the top of `MacroGoalsPage` — the issue's "diet modality page" is the enhanced goals page, reached by the diet intro's existing "set goals" CTA; picking a card snaps the sliders to that split, and the sliders below tweak from there. The selector **starts collapsed and is toggled** by a tappable header with an expand chevron (`AnimatedSize` so the cards actually unmount rather than staying in the tree like `AnimatedCrossFade` would); collapsed, the header summarises the current pick — the modality's label, or `macroGoalsModalityCustom` for a custom split. Selection reads from the card's primary border/tint alone (no check glyph). Add a modality by adding an enum case plus its `dietModalityLabel` and `dietModalityIcon` — no storage, request field, or migration.

## Logging food by unit

A countable food can be logged as "2 eggs" rather than "100 g" (issue #114). The whole feature turns on one modelling call: **`foods.grams_per_unit` is a property of the food, not of a quantity**, so it lives on the shared catalog row and logging by unit is a client-side multiply. Nothing below the sheet changed — `FoodPortion`, `MacroTotals`, `DailyMacros`, adherence, the calendar and every history chart still see only grams, exactly as `amountMl`/`weightKg` are their features' base units.

**Null `grams_per_unit` means not countable**, which is most of the catalog: rice is a bulk grain, milk is a liquid. `Food.isCountable` is what makes `FoodQuantityInput` render its weight/units switch at all, so a food nobody counts shows exactly the plain weight field it always had.

**A countable food leads with the unit, not the weight (issue #143).** `FoodQuantityInput` puts the `VTStepper` (the unit count) on the *left* and the equivalent-weight field on the right, and `LogFoodSheet` seeds a new log at **1 unit** (grams = `gramsPerUnit`) rather than the 100 g default a non-countable food still gets — logging "an egg" is one tap, and the weight field just reads back the equivalent. The edit sheet is unchanged: it seeds from the existing log's own count. The unit-first ordering is layout only; `_lastEdited`/`gramsFor` still make grams the source of truth, so editing either field stays consistent.

**A unit is not a third `UnitSystem`.** metric/imperial is a g↔oz choice; two eggs are two eggs in either. `FoodQuantityMode.units` therefore *bypasses* `WeightConversion` rather than extending it — the mode enum owns `gramsFor`/`unitsFor` (total, so no call site needs a `!` on `gramsPerUnit`) and both sheets share it. Only the *equivalent-weight hint* under the field is converted, since that's a weight being read.

**`food_logs.quantity_units` is recorded, not authoritative.** `quantity_grams` stays not-null and is the source of truth for every macro; the unit count only exists so the day view can say "2 un" back instead of answering "100 g" to someone who typed "2 eggs". It is deliberately never used to recompute grams — a later converter re-run that revises a food's `grams_per_unit` must not move yesterday's calories. That's the opposite of the recipe-edit wart, and it's free here because grams were always the truth. Switching an edit back to weight mode nulls the column; `EditFoodLogSheet` also refuses to open in unit mode for a food that has since lost its unit weight, which would otherwise strand the sheet in a mode whose switch isn't rendered.

**The catalog is populated in bulk, not at runtime** — `tool/populate_food_unit_weights.dart` asks [food-gram-converter](https://food-gram-converter.onrender.com/docs) what one item of each food weighs. Calling it live from the app was rejected: it would make an LLM behind a free-tier host a hard dependency of the log sheet. `foods.grams_per_unit_checked_at` is what makes re-runs cheap — **null means never asked**, and non-null with a null weight means "asked, and it isn't countable"; without it null would conflate the two and every run would re-ask about every bulk food forever. A food the converter couldn't be *reached* for stays unstamped, so it's retried. Rows are processed `times_logged desc` so a bounded run (`UNIT_MAX_FOODS`) still covers what people actually eat, and a low-confidence answer is discarded — a wrong unit weight silently misreports macros for everyone, the same reasoning the scan prompt's "return null rather than guess" follows.

Reading Open Food Facts' own `product_quantity` instead was considered and dropped: it's free and right for a 350ml can, but a 1kg bag of rice has one too, and believing it would make rice countable at 1000g per unit. The converter's `not_convertible` verdict is exactly the judgement that field lacks.

Custom foods capture their own unit weight (`CustomFoodPage`, optional). It's a plain `String gramsPerUnitText` on `CustomFoodState` rather than a `double?` — emptying the field has to be expressible and `copyWith` can't tell a null meaning "cleared" from one meaning "unchanged", which is the same problem `nutrients` uses a map to sidestep. It deliberately isn't a `CustomFoodNutrient` case: that would force it into `isComplete`, crash `_buildFood`'s `!` assertions, need a `ScannedNutritionFacts` mapping and get coloured into the energy-split card.

## Architecture

Clean Architecture, without repository/datasource interfaces and without a separate data-model layer — one concrete class per responsibility, no `Impl` suffixes:

```
lib/app/
  core/            DI (get_it), navigation extensions, error (Result/VTError), http, env, services/
  data/<feature>/   Repository (concrete class), datasources/
  domain/<feature>/ Entities, use cases (plain classes with a `call` method)
  design_system/    VT-prefixed tokens, themes and components
  presentation/     Pages (Cubit + VTPage), routing
```

`data/diet/` and `domain/diet/` are the first feature built out this way, composing two datasources behind one `DietRepository`: `SupabaseDietDataSource` (persistence) and `OpenFoodFactsDataSource` (remote search, via `core/http`'s `VTHttpClient`). Entities double as the wire format for Supabase rows via a `fromMap(Map<String, dynamic> row)` factory constructor on the entity itself (`Food.fromMap`, `FoodLog.fromMap`, ... — no `@JsonSerializable`/`build_runner`, Supabase's client already returns decoded maps, so generated (de)serialization would just add ceremony); the datasource calls `EntityName.fromMap(row)` and otherwise just orchestrates the Supabase/HTTP call, with no parsing logic of its own. `data/water/`/`domain/water/` follow the same shape with a single `SupabaseWaterDataSource`/`WaterRepository` — no remote catalog to search, so it's simpler: log an amount directly, no save-then-log step. Follow this same shape for the next feature (workout): one repository per feature composing whatever datasources it needs, entities mapped by hand via their own `fromMap`.

A feature's `datasources/` folder is flat (single file, e.g. `data/auth/datasources/supabase_auth_datasource.dart`) when there's only one datasource. Once a feature composes more than one kind, split into one subfolder per kind — `supabase/`, `http/`, `local/` — so the datasource type is visible from the path: `data/diet/datasources/{supabase/supabase_diet_datasource.dart, http/open_food_facts_datasource.dart}`, `data/water/datasources/{supabase/supabase_water_datasource.dart, local/water_local_datasource.dart}`. A vendor-specific `requests/` folder (wire-format request bodies, e.g. `CreateFoodRequest`) nests under that vendor's subfolder (`data/diet/datasources/supabase/requests/`), not directly under `datasources/`. This doesn't apply to a local-only feature with no other datasource kind (`OnboardingLocalDataSource`, `SettingsLocalDataSource`) — those keep the flatter `data/<feature>/<feature>_local_datasource.dart` shape from the Local storage section below, since there's nothing to disambiguate from.

A domain use case depends directly on the concrete repository class from `data/`, e.g.:

```dart
class LogFoodUseCase {
  LogFoodUseCase({required DietRepository dietRepository}) : _dietRepository = dietRepository;
  final DietRepository _dietRepository;
  Future<Result<VTError, FoodLog>> call({required Food food, ...}) => _dietRepository.logFood(...);
}
```

One deliberate exception to "no interfaces": the thin adapter that directly wraps a third-party SDK gets its own concrete class under `core/services/<vendor>/`, so datasources depend on it rather than on the vendor's client type directly (`core/services/storage/local_storage_service.dart`'s `LocalStorageService`; `core/services/supabase/supabase_service.dart`'s `SupabaseService`). Repositories/datasources/use cases above that boundary stay concrete-only as usual. `core/services/image_picker/`'s `ImagePickerService` is the one such adapter a **cubit** depends on rather than a datasource (picking a photo is a device interaction, not persistence) — it returns a plain `PickedImage` (path + bytes + extension) and takes the app's own `ImagePickerSource` enum, which pairs its case with `image_picker`'s `ImageSource` the same way `SupabaseTable` pairs a case with its wire name, so nothing above `core/services/` imports `package:image_picker`. Choosing camera-vs-gallery stays UI (`showImageSourceSheet`); the cubit only receives the chosen source.

**A Page/widget never resolves a datasource, repository, or use case itself** — no `G<XLocalDataSource>()`, `G<XRepository>()`, or `G<XUseCase>()` inside a `build()` method or a callback. The only thing a widget resolves via `G<...>()` is its own Cubit (in a `BlocProvider`'s `create:`), and the only thing it calls on that Cubit is one of its methods. The Cubit is what depends on the use case(s), which depend on the repository, which depends on the datasource(s) — every hop stays in that order, with nothing skipped. `domain/onboarding/`'s `OnboardingPage → OnboardingCubit → CompleteOnboardingUseCase → OnboardingRepository → OnboardingLocalDataSource` is the reference shape, including for a feature with no remote datasource to justify a repository: it's still a thin single-datasource wrapper, kept because nothing above `data/` is allowed to reach past the repository boundary, not because it composes anything. The one deliberate exception is routing-level infrastructure — `HomeRoute`'s redirect (the onboarding gate, see Navigation below) depends on `HasSeenOnboardingUseCase` directly via `G<>()`, the same way `bootstrap.dart` calls services directly; neither is a Page, both sit outside the widget tree as wiring. (`WaterCubit` predates this rule and still depends on `WaterLocalDataSource` directly for its daily-goal read/write, with no use case/repository in between — don't copy that shape for a new feature; `OnboardingCubit` is the one to follow.)

## Dependency injection

`GetIt.instance` is aliased as `G` (`lib/app/core/di/dependencies.dart`). Register dependencies in `setupDependencies()`, called once from `bootstrap()`. Resolve with `G<Type>()`.

## Local storage

`hive_ce` (+ `hive_ce_flutter` for `Hive.initFlutter()`) for anything that must survive an app restart but doesn't belong in Supabase (device-local preferences, not user data), sitting behind `core/services/storage/local_storage_service.dart`'s `LocalStorageService` (`get<T>`/`put<T>`/`delete`, backed by a Hive `Box<dynamic>`) — datasources never touch `package:hive_ce` directly, only `LocalStorageService`'s constructor does. One shared `app` `Box<dynamic>`, opened once in `bootstrap()` and wrapped in a single `LocalStorageService` registered in DI (opening a box is async, DI registration isn't) — not one box/service per feature. Each feature still gets its own local datasource class (concrete, no interface, same shape as a feature's Supabase datasource) depending on `LocalStorageService`; keys are prefixed with the feature name (e.g. `SettingsLocalDataSource`'s `settings.locale`, `settings.themeMode` in `data/settings/settings_local_datasource.dart`) so datasources can't collide with each other in the shared box. Primitives only (`String`, `int`, `bool`, ...) — no `TypeAdapter`/`build_runner`. `SettingsLocalDataSource` sits behind `SettingsRepository`/`GetAppSettingsUseCase`/`SaveAppSettingsUseCase` (see App-level state below) — nothing above `data/` touches it directly. A new feature that needs local-only storage gets its own `data/<feature>/<feature>_local_datasource.dart` depending on `LocalStorageService`, with its own key prefix — don't add unrelated state to `SettingsLocalDataSource`. Route it through a repository and use case(s) per the Architecture section's layering rule (`data/onboarding/`'s `OnboardingLocalDataSource` behind `OnboardingRepository` is the current example) rather than a Cubit depending on the local datasource directly.

Supabase gets the same treatment: `core/services/supabase/supabase_service.dart`'s `SupabaseService` wraps `SupabaseClient`, exposing `auth`, `currentUserId`, `hasSession`, and `from(table)` — datasources depend on `SupabaseService`, never on `package:supabase_flutter`'s `SupabaseClient` directly (`SupabaseDietDataSource` is the example). This isn't a full abstraction of Postgrest's fluent query builder (that's not a real swap boundary and would just add ceremony) — it centralizes construction and the one piece of logic every datasource needed (`currentUserId`), consistent with `LocalStorageService`.

**Always pass `ascending:` explicitly to `.order(...)`.** postgrest-dart's `order()` defaults to **`ascending: false`** — descending — which is the opposite of both SQL's default and everyone's assumption. A bare `.order('position')` silently returns the list backwards; the symptom is never "sorted wrong", it's a feature that looks broken for an unrelated reason (routine reordering read as "my drag didn't save" while the database held the correct order the whole time). The default is invisible at the call site, so the only defence is to always spell the direction out, even when ascending is what you want — `test/app/data/order_direction_test.dart` scans `lib/` and fails on any `.order(...)` that doesn't. It has to be a source scan rather than a real test: datasources are tested against a mocked `SupabaseService`, so PostgREST's ordering is never exercised and this class of bug is invisible to the suite by construction.

Table and Storage-bucket names are never raw strings: `from(...)` takes a `SupabaseTable` enum and `storage(...)` takes a `SupabaseBucket` enum (`core/services/supabase/supabase_table.dart`, `supabase_bucket.dart`), each pairing its Dart case with the DB `wireName` the same way `AppRoute` pairs a name with its `path` (`foods('foods')`, `foodLogs('food_logs')`, `foodImages('food-images')`, ...). A new table/bucket means adding an enum case, not hardcoding its name at the call site — the signatures make a raw string impossible to pass. Column names and embedded-resource selects stay plain strings, except where a select re-references a table name it reads it off the enum too (`getDailyLog`'s `'*, ${SupabaseTable.foods.wireName}(*)'`).

Tests exercise `LocalStorageService` against a real Hive box in a temp directory (`Hive.init(tempDir.path)`, no platform channels needed) rather than mocking it — see `test/app/core/services/storage/local_storage_service_test.dart`, `test/app/data/settings/settings_local_datasource_test.dart`, and `test/widget_test.dart` (all via the `test/fixtures/local_storage_fixture.dart` fixture). `SupabaseService` is mocked instead (`test/mocks/services_mocks.dart`) since standing up a real Supabase client in tests isn't practical.

## State management

`flutter_bloc` Cubits. A Cubit that loads data extends `PresentationCubit<S, P>` (`lib/app/presentation/general/presentation_cubit.dart`) instead of `Cubit<S>` directly — it mixes in `bloc_presentation`'s `BlocPresentationMixin<S, P>`, giving it `emitPresentation(...)` for one-off side effects that aren't part of state. `P` is **not shared globally** — every cubit gets its own `sealed class XPresentationEvent` next to its state/cubit files (`diet_presentation_event.dart`, `water_presentation_event.dart`, ...), so a cubit is free to grow presentation events beyond loading (a snackbar, a one-off navigation) without touching anyone else's type. Pages that own such a cubit use `VTPage<C, S, P>` (`lib/app/presentation/general/vt_page.dart`), which does three things at once: resolves the cubit via `G<C>()` (a page never calls `G<C>()` itself, and never passes a `create:` callback — `VTPage` is the one place that does DI resolution for pages), wires `BlocBuilder` (handing cubit + state to `builder:`, so pages never call `context.read` for their own cubit), and wires `BlocPresentationListener`, forwarding every event to an **optional** `onPresentation: (context, event) => ...` (see Loading below for the one thing every current page uses it for). A cubit with no async load at all (`AppCubit`) doesn't need `VTPage` — `SettingsPage` just uses a plain `BlocBuilder<AppCubit, AppState>` since `AppCubit`'s state is available synchronously from construction.

**`XState` is a single, immutable, `Equatable` class per cubit holding only real data — never a `sealed class` with one subtype per "mode", and never an error field.** What used to be separate variants (`XLoaded`/`XError`, or `FoodSearchIdle`/`FoodSearchLoaded`/`FoodSearchError`) are now just data fields on one class (`DietState.dailyMacros`, `SleepState.logs`, ...) — an error is never part of state, see Errors below for where it actually lives. `FoodSearchState.results` is nullable — `null` means idle/no search yet, a non-null empty list means a search ran and found nothing; the two are different states with the same field. Every `XState` gets a `copyWith({...})` for updating one or two fields without reconstructing the rest. Pages read `state.field` directly, or pattern-match with `switch (state) { XState(results: final results?) => ..., ... }` when there's more than a plain if/else worth of cases (see `FoodSearchPage`) — never a `switch` over subtypes. This only applies to **Cubit state**; a domain entity that's genuinely a closed set of mutually exclusive shapes (`User` — `AuthenticatedUser` vs `AnonymousUser`, `lib/app/domain/auth/entities/user.dart`) still models that as a `sealed class`, since that's a real sum type, not a "same data plus an error flag" shape — the distinction is state (one cubit's evolving snapshot, data only) vs. entity (a domain concept with fundamentally different shapes).

Any cubit `VTPage` creates that needs to kick off work on mount (typically an initial load) overrides `PresentationCubit.onInit()` — e.g. `DietCubit.onInit() => loadToday();`. Nothing calls `onInit()` itself; `VTPage` calls it automatically, but only from a widget that's a **descendant** of its `BlocPresentationListener`, not from `BlocProvider`'s `create:`. This matters: `emitPresentation` writes to a plain broadcast stream with no replay, so if `onInit()` ran before the listener subscribed, the very first `showLoading` a cubit emits on mount would be emitted into a stream nobody's listening to yet and silently dropped — the overlay would just never appear for the initial load. Flutter guarantees a widget's `initState` completes before any descendant mounts, so calling `onInit()` from a small private `StatefulWidget` placed as `BlocPresentationListener`'s `child` (see `_VTPageOnInit` in `vt_page.dart`) is what makes the ordering safe.

## Loading

No cubit models loading as a state field — it's a pair of presentation events instead, shown as one global overlay instead of each page switching on a loading state. Every feature's `XPresentationEvent` declares its own plain `XShowLoading`/`XHideLoading` classes (no shared base class to extend — `VTPage` doesn't auto-detect loading events centrally, each page wires it explicitly). A cubit emits these around an async load: `emitPresentation(DietShowLoading())` before the await, `emitPresentation(DietHideLoading())` after, then `emit(...)` the real state — see `DietCubit.loadToday()`/`WaterCubit.loadToday()`/`FoodSearchCubit.search()`. Because there's no loading state, a `PresentationCubit`'s **initial state is a "loaded but empty" value** (e.g. `DietCubit`'s initial state is `DietState(dailyMacros: const DailyMacros(entries: []))`, not a null/loading placeholder). Every page that owns such a cubit passes `onPresentation` to `VTPage` and switches on its own event type to call `context.showLoading()`/`context.hideLoading()` explicitly (`lib/app/core/loading/loading_extensions.dart`, wrapping `loader_overlay`'s `context.loaderOverlay`) — see `DietPage`/`WaterPage`/`SleepPage`/`FoodSearchPage`/`AuthPage`/`ProfilePage`'s `onPresentation:` switch. `main.dart` wraps `MaterialApp.router`'s `builder:` in a single `LoaderOverlay` for the whole app — pages/cubits never touch `loader_overlay` directly, only through `context.showLoading()`/`hideLoading()`. The overlay's visual is `VTLoadingOverlayIndicator` (design system, see below).

## Errors

Like loading, an error is never part of `XState` — it's a presentation event, shown as a toast instead of each page switching on an error state. Every feature's `XPresentationEvent` declares its own `XError` class (`DietError`, `WaterError`, `SleepError`, `FoodSearchError`; `AuthCubit` reuses its existing `AuthActionFailed` for the same role) carrying `message`. A cubit emits it in place of a state change: `emitPresentation(DietError(message: error.message))` — the state stays whatever it already was, so the page keeps showing the last-good (or empty) content underneath the toast. Every page's `onPresentation` switch adds a case for its error event calling `context.showErrorToast(message: message, onRetry: context.read<XCubit>().loadX)` (`lib/app/core/toast/toast_extensions.dart`) — `onRetry` is omitted where there's nothing sensible to retry (`FoodSearchPage`, since the cubit doesn't hold onto the last query). `DietError` carries a `date` alongside `message` since `DietCubit` can be loading any day, not just today (`DietPage`'s `DietDateSelector` lets the user page through history) — `onRetry` calls `cubit.goToDate(date)` with the date that actually failed, not whatever `state.date` happens to be by the time the user taps retry. **Every one of these is a toast, not a dialog** (issue #80) — `VTErrorDialog` is gone, along with `showErrorDialog`. A modal whose only action is OK blocks the screen to ask nothing, which is why 10 of the 18 error call sites were pure ceremony; the other 8 had a real choice (retry or not), and that choice now rides on the toast as an action instead of behind a barrier. The only `showDialog` left in the app is the water and sleep goal editors, which are forms — they take input, so they can't be toasts.

Two things this deliberately trades away, both real: a toast **can be missed**, and a load failure now leaves the page showing its empty state, which reads as "no data" rather than "didn't load". The mitigation is duration, not modality (`_successDuration` 4s, `_failureDuration` 6s, `_actionableFailureDuration` 8s — a failure is not free to miss the way an acknowledgement is). If a specific failure ever proves too costly to miss, the fix is that call site, not bringing the dialog back.

**Validation is not a crash and doesn't wear a crash's colours.** `context.showWarningToast(...)` is the sibling of `showErrorToast` for a failure the user can fix from where they already are — the incomplete custom food, the unnamed recipe, the unnamed routine, a scan that read nothing. It picks `VTSeverity.warning` (amber, an info glyph) over `.error` (red), and **deliberately takes no retry**: running the same incomplete form again fails the same way. Both default their title from l10n (`errorTitle` / `warningTitle`) and take an override where the caller has something better to say (`dietNutritionScanNoDataTitle` — its title states the problem so the message is free to state the action).

## The accent-avatar shortcut has a contrast floor — use `VTColors.inkOn`

The "circular icon avatar tinted with an accent at ~16% alpha, with the accent itself as the icon colour" convention **fails WCAG's 3:1 non-text floor for almost every accent the app owns**, in one theme or the other. Measured on the app's own card surfaces:

| accent | on light card | on dark card |
|---|---|---|
| `water` | **2.11** | 4.30 |
| `coral` | **2.54** | 3.66 |
| `green` | 3.57 | **2.63** |
| `sleep` | 3.47 | **2.72** |
| `success` | **2.87** | 3.23 |

This is the same trap the `VTSeverity` section already tells a story about, and the Home redesign walked straight into it before the numbers were run. **A solid accent disc with the right ink clears it every time** (worst case 4.45:1), so `VTColors.inkOn(accent)` returns whichever of `onGreen`/`onSurfaceLight` actually contrasts better — computed, not thresholded. A luminance cutoff was tried first and mispicks `water`, whose best ink is the dark one despite a mid luminance.

`vt_colors_ink_test.dart` asserts the 3:1 floor across every accent, asserts `inkOn` beats the alternative ink each time, and pins the 16%-tint failure so nobody reintroduces it. **Reach for `inkOn` whenever an accent carries an icon**; the 16% tint is still fine as a *background* behind neutral ink, which is what the meal and badge tints do.

## Food search: the idle state is the feature

The screen every logged meal passes through, reworked in issue #163. Four things were wrong and they compound:

**The hint named the fallback.** "Search Open Food Facts" — but `DietRepository.searchFoods` queries **our catalog first** and only augments with OFF when results are sparse (see Food data). It advertised the wrong source, and one the user should never have to know exists. It is now just "Search foods".

**Search was submit-only.** Results came from `onSubmitted`, so every lookup cost a keyboard return. `FoodSearchCubit.queryChanged` now debounces (350ms, minimum 2 characters) and `VTSearchField` gained an `onChanged` to feed it. The debounce lives in the cubit, not the widget, so the timer is cancelled in `close()`. It is listed in `motion_tokens_test.dart`'s `_allowed`: **an input delay is not a transition**, the same category as the HTTP retry backoff.

**The idle state remembered the wrong thing.** It listed past *queries* — text you typed. What a food tracker is actually asked for is **the food you logged last time**, because eating is repetitive.

**The `+` means the same thing in both modes: accept the last amount.** Tapping a recent food's row opens a sheet to adjust; tapping its `+` takes the quantity you used last time — logging it when adding food, popping a `RecipeIngredient` when picking for a recipe. The first cut opened the quantity sheet on `+` while picking, which made the same button mean "accept" in one flow and "adjust" in the other. Pinned by a test that asserts the picked ingredient carries the last amount.

**Recently-logged foods are their own tab, not the Search tab's idle state.** Putting them above the recent searches was tried first and buried them — on the Search tab, the thing you want is your recent *searches*. So the strip is now **Search / Recent / Favourites**: Search idles into recent-search chips, Recent lists the foods you've logged with the amount you used last time so one tap re-logs it. The tabs carry no icons, because three labelled tabs across a phone is already tight.

Recent searches are `InputChip`s, not rows. A row per past query ate the whole tab for something that is one word long; a chip wraps, and its built-in `onDeleted` is the ×.

`DietRepository.getRecentlyLoggedFoods` is the new read behind it. It is **deduped in the repository, not the query**: PostgREST has no usable `DISTINCT ON`, so the datasource takes the newest `_recentEntryLookback` (80) rows and the repository keeps the first entry per food id. It returns `FoodLogEntry` rather than `Food` precisely because the entry carries the quantity — that is what makes the one-tap re-log possible.

**Nothing tonal may default to `secondaryContainer`.** `IconButton.filledTonal` and a selected `ChoiceChip` both do, and the app's secondary is coral — which in dark resolves to `coralContainerDark` (#7A3A18) and reads as **brown against an otherwise green app**. `VTQuickAddButton` wraps the add affordance in `primaryContainer`/`onPrimaryContainer`, and `VTTheme` sets `chipTheme.selectedColor` to the same, so the meal chips in the log sheet follow. Reach for the quick-add component rather than a bare `filledTonal`.

**Rows were ragged.** The catalog/OFF merge means some rows have a brand and some do not, and the name could wrap to two lines — so a list had visibly uneven row heights. `FoodSearchResultTile` now gives the name **one line** and always renders **one** meta line (`brand · kcal`, or just kcal), which is pinned by a test asserting a branded and an unbranded row measure the same height.

## Picking a day is not judging a day

`CopyMealsPage`'s calendar used `GoalAdherence.color`, so a day you ate badly on showed **red** — in a picker whose only question is "does this day have meals I can copy?". Adherence is the diet *history* calendar's job; here every logged day is equally valid, so they are all `VTColors.success`.

**Empty days are pickable too.** They used to be inert, which reads as a broken tap — the user has no way to tell "nothing here" from "this control is dead". `VTCalendarDayCell`/`VTCalendarMonthGrid` gained `allowsEmptySelection` (opt-in, so the diet/water/sleep history calendars keep refusing empty days as before), and picking one swaps the "No day picked" prompt for "Nothing logged that day". Both pinned, both mutation-verified.

## Adding food is one place, not four

Scanning a meal photo used to be an icon in the **diet** app bar, next to recipes, copy and history — isolated from the flow it belongs to, and part of why that bar carried four actions. `MealScanAction` now lives on `FoodSearchPage` beside the custom-food action, so the three ways to add food (**search it, type it, photograph it**) sit together in the one place a user goes to add food. The diet bar is down to three actions. A fourth way to add food goes there too, not back on the diet page.

## Swipe to delete

`VTSwipeToDelete` (`design_system/components/general/`) wraps a row in a `Dismissible` with an error-tinted background, `endToStart` only, and a `VTHaptics.warning()` on dismissal. Used by `FoodLogTile`; reminders are the next call site.

**It does not replace the visible delete button, it joins it.** A swipe is invisible to VoiceOver and undiscoverable to anyone who has not been taught it, so removing the `IconButton` would trade a small gain in tidiness for a real accessibility regression — the opposite of what the Accessibility section is for. Swipe is the shortcut; the button is the affordance.

**`endToStart` only** is deliberate and pinned by a test: a bidirectional `Dismissible` turns a mis-aimed horizontal scroll into a deletion.

## Celebration

`VTCelebration` (`design_system/components/general/`, issue #163) wraps a widget and bursts confetti when its `trigger` flips **false → true**. It is hand-rolled (a `CustomPainter` over an `AnimationController`) rather than a Lottie/Rive dependency — the same "own the look" call the charts and `VTWeightPicker` make, and it avoids shipping a third-party animation asset whose licence would have to be cleared for App Store submission. Every call site goes through the one component, so swapping the visual for a Lottie file later is a single-widget change.

**It fires on the edge, never on the state.** `didUpdateWidget` requires `widget.trigger && !oldWidget.trigger`; triggering off the state alone would re-fire on every unrelated rebuild, which is the difference between a reward and an irritation. Pinned by a test, verified by mutation.

**It resets itself.** `forward()` leaves the controller *completed*, not dismissed, so gating the painter on `isDismissed` alone leaves an invisible burst in the tree forever. `_celebrate()` awaits the run then `reset()`s (guarded on `mounted`).

**It honours reduce-motion.** `MediaQuery.disableAnimationsOf(context)` returns the child untouched — a celebration is exactly the kind of motion someone turns that setting on to avoid. Also pinned by mutation.

**Sparingly, and sized to the moment.** Four milestones: finishing a workout (`WorkoutState.isFinished`), closing the calorie ring (`GoalAdherence.met`), hitting the water goal (`WaterProgressCard`'s `reached`), and ticking a reminder. The first three are all **once-a-day, goal-met** moments on a headline metric, which is the bar — the reminder tick is the exception and uses `VTCelebrationSize.small` because it is the most repeated action in the app and a full burst there would go stale within a day. **Do not add a fifth without asking what it costs the others** — the value is in scarcity, and anything that can fire more than once a day needs the small variant at most. The burst carries `VTHaptics.success()` itself, so a call site that celebrates must *not* also fire its own success haptic (`ReminderTile` now only fires `.selection()` on un-completing).

## Toast copy is the app's voice

Audited end to end (issue #163). It is more consistent than expected — titles are uniformly short noun phrases ("Meals copied", "Sleep synced", "Profile updated") and that is the pattern to keep. The one real drift was **passive voice**: "The detected items were logged to your day" and "Your changes were saved" sat against an otherwise active, warm voice ("You're all set", "Saved as a food you can log"). Both were rewritten active. `profileAccountDeletedMessage` stays passive deliberately — the gravity suits it.

## Skeletons, and the loaded-vs-empty distinction

The global overlay stays for **writes**; an initial **read** shows a `VTSkeleton` instead (issue #163). The overlay is right for a blocking write, but for a read it hides structure the app already knows it is about to draw, and makes the app feel slower than it is.

**The reason this mattered urgently is a regression the Phase 1.4 empty states introduced.** A `PresentationCubit`'s initial state is deliberately a "loaded but empty" value (see State management), so on first mount `hasData` is false and the full-page empty state rendered *before the first read resolved* — a user with months of history saw "No meals logged yet / Log a meal" every time they opened it. Proven with a probe (`emptyVisible=true` on frame 1), fixed, and now pinned by a regression test that pumps a deliberately slow use case and asserts skeleton-not-empty-state on that frame.

**`XHistoryState.isLoaded` is not a loading flag**, and the "no cubit models loading as a state field" rule still stands. It records whether the first read has *ever resolved* — a fact about whether the data is **known**, the same distinction `FoodSearchState.results` draws by being nullable (`null` = idle, `[]` = searched and found nothing). It never goes back to false, so it is not modelling an operation in flight; that is still the overlay's job via presentation events. The three states are: **not known yet** → skeleton, **known and empty** → `VTEmptyState`, **known** → content.

It is folded into the existing data emit (`copyWith(xInMonth: ..., isLoaded: true)`) rather than emitted separately, so the happy path costs no extra rebuild. The trailing `if (!state.isLoaded)` guard exists only for the **failure** path — a failed first read must fall through to the empty state, never leave a skeleton pulsing forever.

`VTRefreshable` gained `isLoaded` + `skeleton` mirroring `hasData` + `emptyState`, with an assert for each pairing.

**A skeleton pulses; it does not transition.** `VTSkeleton` is in `motion_tokens_test.dart`'s `_allowed` alongside the loading spinner, and its `Curves.easeInOut` is deliberate — a pulse breathes symmetrically, where `VTMotion.curve` eases *out of* a change.

## An empty state's CTA replaces the FAB, it doesn't join it

Once `VTEmptyState` carries a CTA, a screen showing that empty state must **hide its FAB** — otherwise "Log sleep" appears twice, once mid-screen and once in the corner, which is what shipped in #171. Six screens gate it (`floatingActionButton: state.x.isEmpty ? null : ...`): diet, sleep, body weight, recipes, routines and reminders.

**Gate on the same emptiness the CTA is gated on, which is not always the visible list.** `ReminderPage` hides the FAB on `state.reminders.isEmpty`, *not* `visibleReminders` — when a filter hides everything, the `.noResults` state shows and that one has no CTA, so the FAB has to stay or the screen offers no way to add anything.

## Pull to refresh

Every screen that loads remote data is refreshable through `VTRefreshable` (`design_system/components/general/`, issue #163) — 12 screens, up from 4. **There are no raw `RefreshIndicator`s left**; once the component exists, reaching for the Material widget is the "never add a raw Material widget where a `VT` component exists" rule being broken.

The API is one widget with `hasData` + `emptyState` rather than a ternary between two constructors, so `onRefresh` is named once:

```dart
VTRefreshable(
  onRefresh: cubit.refresh,
  hasData: state.hasData,
  emptyState: VTEmptyState(...),
  children: [...],
)
```

**The reason the component exists rather than a bare `RefreshIndicator` is the empty state.** A `RefreshIndicator` only fires if its child is a *scrollable* — and a `VTEmptyState` is a `Center`, so pull-to-refresh silently did nothing on exactly the screens a new user would most want to retry on (a regression the Phase 1.4 empty states introduced). `VTRefreshable` wraps a non-scrollable `emptyState` in a `SingleChildScrollView` with `AlwaysScrollableScrollPhysics` and a viewport-height `ConstrainedBox`, so the pull works and the state stays centred. `vt_refreshable_test.dart` pins it, verified by deleting the wrapper and watching the test fail.

Worth knowing, because the obvious guess is wrong: a **short `ListView` refreshes fine** under default physics. Measured, not assumed — the only case that genuinely fails is a child that isn't a scroll view at all.

**`RoutinesPage` is the one deliberate omission.** It is a `ReorderableListView`, so pulling down at the top competes with dragging the first row upward, and routines are user-created and only ever change from this device — there is no remote update a pull could fetch. If it ever needs one, it needs a gesture decision first, not a wrapper.

## Empty states

**An empty state that only apologises wastes the moment** (issue #163) — so `VTEmptyState` takes an `actionLabel`/`actionIcon`/`onAction` and renders the CTA itself. Before this it *couldn't*: `MealScanPage` was stacking a separate `VTPrimaryButton` under it, which is the workaround that proved the gap, and none of the other ~22 empty states offered anything at all while a FAB with exactly the right action sat in the corner.

**"No data yet" and "no results" are different states and read differently.** `VTEmptyState.noResults` is the search-shaped one (it fixes the `search_off` icon, which used to be the *default* — so every first-run screen was wearing a search metaphor until it overrode it). No-data-yet gets an onboarding tone and a CTA; no-results suggests a fix and usually has no action worth offering. `ReminderPage` is where both live on one screen: nothing logged at all invites a first reminder, while a filter hiding everything says so and points at the filter — those used to share one message.

**Never name a control glyph in a message.** Six messages said "Tap the + button" — and *none of them was accurate*: water has no FAB at all (its quick-add chips are the action, and they sit right above the empty state), while diet, sleep, reminders and recipes use a **labelled** `FloatingActionButton.extended`, not a bare `+`. Directions to a control go stale the moment the layout moves and are redundant once the empty state carries its own CTA. Describe what the feature gives the user and let the CTA carry the verb.

**A CTA is only added when the action isn't already on screen.** Water's chips are visible directly above its empty state, so a duplicate button there would be noise — the fix was the wording alone.

**The five history screens now distinguish two levels of empty**, and both are real: `XHistoryState.hasData` (any data in the displayed month *or* the trend range) routes a brand-new user to a full-page `VTEmptyState` with a CTA back to the logging screen, while a month that *has* data but an empty trend range keeps the existing per-card "Nothing logged in this period yet." messages. Diet history already had that per-card treatment — the issue's claim that these screens render flat charts with no data was only half right — so the full-page state is layered above it, not in place of it. `diet_history_page_test.dart` pins both, including the partial case (stub the range use case per-range: the month query starts on day 1).

## Motion

Motion is a shared language, not per-screen taste. Every duration and curve comes from `VTMotion` (`design_system/tokens/vt_motion.dart`, issue #163) and `test/app/design_system/motion_tokens_test.dart` **scans `lib/` and fails on any raw `Duration(milliseconds:)` or `Curves.`** — the same source-scan shape as `order_direction_test.dart`, and for the same reason: the drift is invisible at any single call site.

Four durations, chosen by **what the motion is doing**, not by how it feels in isolation:

- **`micro` (150ms)** — a control settling under the finger (`VTWeightPicker`'s snap).
- **`transition` (250ms)** — one state becoming another: expand/collapse (`AnimatedSize`), a sliding indicator, a route's reverse transition. **This is the one that had drifted worst** — 200/240/250/260/300ms were all doing the same job on different screens.
- **`entrance` (400ms)** — content arriving (`VTAppearEffect`, `VTPhotoHeader`).
- **`data` (800ms)** — a value growing to its figure (`VTMacroRing`, `VTProgressBar`, `VTBarChart`, `VTLineChart`). Deliberately the slowest: it is the one animation carrying information rather than decorating a change.

`VTMotion.curve` is `easeOutCubic` everywhere — it was already 12 of 14 uses, so this codified what the app had mostly settled on.

**A stagger is an index, never a millisecond count.** `VTAppearEffect` takes `index:` and derives the delay through `VTMotion.staggerFor` — because the old `delay:` API produced **17 different stagger formulas across 22 call sites** (per-item steps of 40/50/60/80ms, caps of 8, 10 or none, leading offsets of 60/80/200ms). No one designed that; each screen invented its own and none was wrong on its own. Passing an index makes the inconsistency unrepresentable. A list passes the item's own index; a fixed layout passes its ordinal position (`index: 1`, `index: 2`).

**The stagger is capped at `maxStaggerSteps` (6).** Uncapped it delays the very content the user is waiting for — `DietPage` used a bare `index * 80`, so the tenth meal section waited **800ms**, longer than the entrance animation itself. Past the cap every remaining item appears together, which is what "staggered" should have meant all along. The cap is asserted (`staggerFor(50) == staggerFor(maxStaggerSteps)`, and shorter than one `entrance`).

Two things are deliberately **not** motion tokens and are listed in the scan's `_allowed`: `VTHttpClient`'s retry backoff (a network delay) and `VTLoadingOverlayIndicator`'s 1100ms (a continuous loop, not a transition between two states). Toast dwell times (4s/6s/8s, see Errors) are not motion either — they are how long a message stays readable.

## Accessibility

**An `IconButton`'s `tooltip` *is* its accessibility label** — the framework says so in `icon_button.dart` ("is used for accessibility") and `Tooltip` sets `semanticsTooltip` on the node. It lands in the semantics **`tooltip` field, not `label`**, which is why `find.bySemanticsLabel` won't match it — that's a testing gotcha, not a missing label. So **never wrap a tooltipped `IconButton` in a `Semantics(label:)`**: that double-labels it and VoiceOver reads the button twice. An icon-only button needs a `tooltip`, and that is the whole fix.

**A custom-painted widget whose child is already text should `MergeSemantics`, not carry a hand-written label.** All four `VTMacroRing` call sites wrap a `Column` of localized `Text` (the calorie readout, "of 2000 kcal", "150 left") — so the ring was never silent, it just announced as **three separate VoiceOver stops**. `MergeSemantics` collapses them into one, which is the actual improvement, and it **cannot drift out of sync with what's on screen** the way a duplicated `semanticLabel` string would. The ring only takes an explicit `semanticLabel` for a caller that genuinely has no text child.

`VTSemanticSummary` (`components/general/`) is the shared piece: given a label it replaces the subtree's semantics (`excludeSemantics: true, container: true`), given null it leaves the child alone. `VTMacroRing`, `VTProgressBar`, `VTBarChart`, `VTLineChart` and `VTDistributionBar` all take an optional `semanticLabel` through it.

**The charts are deliberately left unlabelled for now.** Every chart in the app sits in a card that already announces its title, its average badge and its logged-day count as ordinary `Text` — what the painting adds is the *shape*, and a single label can't linearize 90 bars into anything useful. Restating the average the card already says would be redundant noise, not accessibility. The real fix for per-point values is the chart **interactivity** item in the issue's Tier 3 (tap a bar to hear its value); the `semanticLabel` hook exists so a chart that ever *is* the only source of its data can be labelled without a component change.

**Tap targets: `VTSpacing.minTapTarget` (44) is the floor**, and it is measured, not eyeballed — `vt_tap_target_test.dart` asserts a rendered `getSize`. `VTDragHandle` shipped at **30×38** and is the reason the token exists; a bare `IconButton` is already 48×48, so the delete buttons that looked suspicious were fine. Measure before "fixing" a tap target.

Adding localized tooltips to a design-system component makes it need `AppLocalizations` delegates — an existing widget test that pumped it in a bare `MaterialApp` will start throwing a null-check on `AppLocalizations.of`. Add the delegates to the test rather than pulling l10n back out of the component.

## Haptics

Every physical feedback goes through `VTHaptics` (`design_system/components/general/vt_haptics.dart`, issue #163) — call sites express **intent** (`.selection()`, `.success()`, `.warning()`, `.error()`, `.drag()`), never an impact strength, so the mapping is tuned in one place rather than re-decided per screen.

**Flutter does not expose iOS's `UINotificationFeedbackGenerator` types**, so success/warning/error cannot be told apart semantically — only by weight. The mapping is therefore an **escalation**: success is `lightImpact`, warning `mediumImpact`, error `heavyImpact`. The worse it is, the harder it hits. `selection` is `selectionClick` (the discrete tick) and `drag` is `mediumImpact`.

**Wire it at the choke point, not per call site.** Every toast's haptic comes from `_showToast` switching on the `VTToast`'s own `VTSeverity`, so all three toast helpers — and therefore every error, warning and success in the app — are covered without a single page changing. The same idea puts the tick inside `VTSegmentedTabs`, `VTStepper`, `VTLabeledSlider`, `VTWeightPicker` and `VTCalendarDayCell` rather than at the screens that use them. A new screen built from `VT` components gets its haptics for free; **if you find yourself adding `VTHaptics` to a page, ask whether it belongs in the component instead.**

Two rules that are easy to get wrong and are pinned by `vt_haptics_test.dart`:

- **A continuous control ticks per division, never per event.** `VTLabeledSlider` is continuous, so firing on every `onChanged` machine-guns; it quantizes the range into `_hapticDivisions` (100) and only ticks when the bucket changes, so one full sweep is a ruler regardless of whether the slider spans 300g or 3000 kcal. `VTWeightPicker` gets this free — `_onScroll` already gated on crossing a step.
- **A no-op action gets no feedback.** Re-tapping the already-selected segment doesn't tick. Feedback for something that didn't happen is worse than none.

**A toggle is not symmetric**: completing an exercise or ticking a reminder is `.success()`, undoing it is `.selection()` — un-completing is a correction, not an achievement.

The test **records real platform channel calls** (`SystemChannels.platform`, `HapticFeedback.vibrate`) rather than asserting a `VTHaptics` method was called, so it pins the actual mapping. Both behavioural tests were verified by mutating the component and watching them fail — the same discipline `vt_toast_test.dart`'s contrast assertions follow.

## Toasts

**A toast is how the app says anything one-off** — it worked, it failed, you left a field empty. All three follow the same presentation-event shape: a cubit emits a one-off event (`FoodLogged`, carrying what the toast renders — `foodName`, `mealType`) and the page's `onPresentation` switch calls `context.showToast(...)` / `showErrorToast(...)` / `showWarningToast(...)` (`lib/app/core/toast/toast_extensions.dart`). Each shows a floating, transparent-background `SnackBar` whose `content` is `VTToast` (design system) — a rounded elevated card with a leading circular icon avatar, a title, a subtitle, and an optional action, so the toast fully owns its look rather than the Material `SnackBar` chrome. `FoodSearchCubit.logFood` still returns its `Result` to `_LogFoodSheet` (which pops on success) *and* emits `FoodLogged` on success — the sheet's inline error handling and the page's toast are two different concerns on the same call.

`VTSeverity` (`success`/`warning`/`error`) pairs the case with its icon and its container/onContainer pair, the way `SupabaseTable` pairs a case with its wire name. **`accentColor` is the one escape hatch** and it overrides the disc with a non-severity colour (a meal's accent on the food-logged toast); it keeps the older 16%-tint-with-the-accent-on-top look, which is safe only because those accents happen to be dark enough — see the contrast section below for why a severity can't be drawn that way.

## Design system: the accent-avatar shortcut has a contrast floor

The "leading circular icon avatar tinted with an accent at ~16% alpha, with the accent itself as the icon colour" convention below is **not safe for every accent**, because it uses one colour as both sides of the pair. Measured on the app's own surfaces: `warning` on light is **1.85:1** and `error` on dark is **2.52:1** — an icon that reads washed out and misses even the 3:1 floor for non-text. It happens to be fine for `error` on light (4.70:1), which is why it survived.

`VTSeverity` therefore uses a **container tone with its own ink** (`container`/`onContainer`), the pattern the palette already had in `errorContainerLight`/`onErrorContainerLight` — 6.6:1 or better in all six combinations, and it stays light. `warningContainer*`/`onWarningContainer*` were added to complete the set. A solid accent disc also clears contrast but reads as a blood-red blob, which was tried and rejected on sight.

`vt_toast_test.dart` pins this by **computing the contrast at runtime** (`Color.computeLuminance` is already WCAG's relative luminance) across every severity × brightness and asserting ≥ 4.5:1. Asserting the colours equal their tokens is what the test did first, and it was tautological — it passed while the token was deliberately broken, because both sides of the comparison moved together. Assert the property, not the constant; verify a colour test by mutating a token and watching it fail.

## Design system

Everything under `lib/app/design_system` is prefixed `VT` (`VTColors`, `VTSpacing`, `VTRadius`, `VTTextStyles`, `VTTheme`, `VTGap`, `VTCard`, `VTPrimaryButton`, `VTEmptyState`, `VTLoadingOverlayIndicator`, `VTFeatureTile`, `VTProgressBar`, `VTMacroRing`, `VTToast`, `VTSearchField`, `VTAppearEffect`, `VTBadge`, `VTBarChart`, `VTDistributionBar`, `VTLegendDot`, `VTPhotoHeader`, `VTImageSourceSheet`) so design-system pieces are always identifiable at a glance. Never use raw Material spacing/colors/text styles in a page when a `VT` token or component exists — add one if it's missing instead of reaching for `SizedBox`/`Colors.*` directly. Component set is sized to what's actually rendered — add a new one only once a page needs it (e.g. a network image or search field component once the diet feature has food images/search).

Palette: forest green primary (health, nutrition), coral-orange secondary (energy, warmth), warm-neutral surfaces. Typography: Poppins for headings, Inter for body text (both via `google_fonts`).

The screens are meant to feel modern and visually appealing, not just functional — the diet page (see below) is the reference bar for that polish; new features should aim for the same level. Beauty comes from the design system, never from raw Material widgets in a page: when a screen needs a richer visual (a progress ring, a colored badge, a gradient header), add a reusable `VT`-prefixed component for it rather than one-off styling inline, the same rule as spacing/color tokens. The visual vocabulary already established — favor these over inventing a parallel look: circular progress (`VTMacroRing`) for a single headline metric against a goal, `VTProgressBar` (now colorable via its `color`) for secondary/list metrics, soft rounded cards (`VTCard`), a leading circular icon avatar tinted with a category's accent color at ~16% alpha for section headers, and pill-shaped badges (rounded-full container, accent color at ~16% alpha with the accent as text/icon color) for compact figures like a meal's calorie total.

## Charts

Charts are hand-rolled `VT` components under `design_system/components/charts/`, not a charting package (`fl_chart` was considered and dropped for issue #55) — `VTMacroRing` already set the precedent that a `CustomPainter` we own beats a dependency whose look isn't ours, and the two shapes the app needs are simple enough that the package's tooltips/axes would be paid for in theming fights.

- **`VTBarChart`** — vertical bars over time. Every bar is a `VTBarChartBar` holding a list of `VTBarChartSegment`s (`value` + `color`), so a plain bar (one segment) and a stacked bar (protein/carbs/fat) are the same widget on one scale; `VTBarChartBar.empty()` is a day with no data and paints a faint stub rather than a zero-height bar, which keeps gaps legible. An optional `referenceValue` draws a dashed goal line, and the scale always includes it so the goal can't fall off the top.
- **`VTDistributionBar`** — one rounded horizontal bar of shares (segments again), for "what is this made of". Used by `CustomFoodEnergySplitCard` (protein/carbs/fat energy split), `MealSplitCard` (calories per meal) and `RecipeTotalsCard`.
- **`VTLegendDot`** — the colour swatch + label that goes under a chart (`isDashed` for a reference line rather than a series).
- **`VTLineChart`** — a line over time for a single metric (issue #101, body-weight trend), a list of `VTLineChartPoint`s (`value` + optional `label`). Unlike `VTBarChart` it **auto-scales the y-axis to the data's own [min, max]** with a little padding rather than anchoring at zero: a weight going 75→73 kg is the whole story, and a zero baseline would flatten it to a featureless line near the top. Only the first and last point labels are drawn, so a sparse series doesn't crowd the axis.

`VTBarChart`/`VTDistributionBar` take `VTBarChartSegment`, so a colour+value pair is the one chart primitive. Bars animate in via `TweenAnimationBuilder` like the rest of the design system. Reach for these before writing a one-off `CustomPainter` in a page.

## Recipes

A recipe is a set of foods eaten together (issue #63). The modelling decision that makes everything else small: **a recipe is stored as an ordinary `foods` row** with `source: 'recipe'`, whose per-100g macros are the roll-up of its ingredients. Logging a recipe is therefore the plain `logFood` path — search, the log sheet, editing, the calendar, adherence and the history charts all keep working untouched, and "add a recipe to my day" needs no new write path at all. The two alternatives were dropped for good reasons: expanding a recipe into one `food_log` per ingredient loses the recipe's identity in the day view, and hanging a nullable `recipe_id` off `food_logs` would force `food_id` nullable and give `FoodLogEntry`/`DailyMacros`/`MacroTotals`/history two cases to handle forever.

`recipes` (user_id + food_id) and `recipe_ingredients` (recipe_id + food_id + quantity_grams) only record **what the recipe was built from**, so it can be listed and its ingredients shown; they carry no macros of their own — the foods row is the single source of truth for the name and the numbers. `recipe_ingredients` has no `user_id`: its RLS policy checks the parent recipe's owner instead of duplicating the column.

**Recipes are personal**, unlike the shared catalog they read from: `searchCatalog` adds `.or('source.neq.recipe,user_id.eq.<me>')`, so a row is kept when it isn't a recipe *or* it's mine. That one predicate is the whole visibility rule — recipes still live in the shared `foods` table, they're just filtered out of everyone else's search.

**Deleting a recipe deliberately leaves its foods row behind.** Past `food_logs` reference it and must keep resolving; the leftover food is no different from any other food you logged once and stopped using.

`SaveRecipeUseCase` is the one place with real orchestration, and it covers **both create and edit** (`recipe: null` means create) — the form is one page doing both, and the ingredient-saving below would otherwise be duplicated across two use cases. An ingredient picked from an Open Food Facts search has no `id` yet, so each id-less ingredient is `saveFood`ed first (the same save-then-log idea as `LogFoodUseCase`); then creating saves the rolled-up `RecipeDraft.toFood()` and writes `recipes` + `recipe_ingredients`, while editing `updateFood`s the recipe's existing foods row in place and replaces its ingredient rows. These are sequential writes, not a transaction — there's no RPC in the project and this matches the existing save-then-log shape, so a mid-way failure can leave an orphan foods row (harmless: it's an unreferenced catalog entry).

**Editing a recipe rewrites what past days show.** Because the recipe *is* its foods row, changing its ingredients changes that row's per-100g macros, and every `food_log` pointing at it — including yesterday's — is re-read through the new numbers. This is the same wart comparable trackers have, and it's the direct cost of the model that makes everything else free. The alternative (a fresh foods row per edit, old logs frozen against the old one) was rejected: the superseded row would still be a recipe-food owned by you, so it would sit in your own search as a duplicate with no recipe behind it.

`RecipeDraft` (name + ingredients, no id) is what the form edits and what computes the roll-up; `Recipe` (id + food + ingredients) is what comes back persisted. Both mix in `MacroTotals`.

**`MacroTotals` was generalised for this**: it now folds over `FoodPortion` (a mixin pairing a `Food` with `quantityGrams`, and deriving calories/protein/carbs/fat/fiber/micronutrients from it) rather than `FoodLogEntry` specifically, so a recipe ingredient and a logged food total up through the exact same code. `FoodLogEntry` and `RecipeIngredient` both mix in `FoodPortion`; `DailyMacros` still narrows `entries` back to `List<FoodLogEntry>` via a covariant override. `MacroTotals` also gained `totalGrams`, which is what the issue's "somar ... gramas" asks for and is meaningful for a day too.

UI: a book icon in the diet `AppBar` opens `RecipesPage` (`/diet/recipes`, list + delete + FAB); tapping a recipe or the FAB opens `RecipeFormPage` (`/diet/recipes/new`), which creates or edits depending on the `Recipe?` in its `RecipeFormExtra`. The form's "add ingredient" opens `AddFoodPage` in `AddFoodMode.pickIngredient` (`/diet/recipes/new/ingredient`), which reuses `FoodSearchCubit` and `FoodSearchResultTile` but pops a `RecipeIngredient` instead of logging — `FoodSearchPage` stays single-purpose rather than growing a "pick" mode. `RecipeTotalsCard` shows the live roll-up as the draft grows. A recipe can carry a photo, picked and uploaded exactly like a custom food's: the cubit uploads the bytes and puts the resulting URL on the draft, so the domain only ever sees `RecipeDraft.imageUrl` and the raw `Uint8List` stays in `RecipeFormState`.

**`VTPage` gained an optional `cubitParam`** for this: `RecipeFormCubit` needs the recipe being edited at construction, and it's handed to `G<C>(param1: ...)` against a `registerFactoryParam` registration. Passing `param1: null` to a plain `registerFactory` is a no-op, so every other page is unaffected. Reach for this rather than a `create:` callback or seeding a cubit from a widget's `initState` — `VTPage` stays the one place that resolves a page's cubit.

## Favorite foods

A heart on `FoodSearchResultTile` favourites a food, and `FoodSearchPage` splits into **Search / Favorites tabs** (issue #78). `FoodSearchState.tab` holds the selection and `changeTab` sets it — cubit state, not a `TabController`, the same shape as `DietHistoryState.trendRange`. Both tabs stay alive across a switch: results and favourites are separate fields, so flipping tabs never re-queries.

**Favouriting a `Food` is the whole feature — recipes came free.** A recipe *is* a `foods` row (`source: 'recipe'`, see Recipes above), so it favourites through the same insert and lands in the same list as any other food. No second mechanism, no `favorite_recipes` table.

`food_favorites` is a join table (`user_id` + `food_id`, unique together) rather than a flag on `foods`, because `foods` is a shared catalog — "is this a favourite" is per-user and can't live on the row everyone reads. It's private like `recipes`: RLS scopes it to `auth.uid()`, and the `on delete cascade` on both FKs means deleting a food (or a user) takes its favourites with it.

**The heart is optimistic**: `toggleFavorite` flips `favorites` before the request and `_revert`s to the previous list (plus an error dialog) if it fails. Favouriting is cheap and reversible, so making the user watch a spinner — two round-trips for an unsaved Open Food Facts result — buys nothing.

**`FavoriteFoodUseCase` does save-then-favourite** and returns the persisted food: a result straight out of an Open Food Facts search has no `id` yet, and the id is what the favourite points at — the same save-then-act step `LogFoodUseCase` does before logging. That id-less window is why `FoodSearchState.isFavorite` matches an id-less food **by value** and everything else **by id** (`FoodSearchCubit._sameFood`), and why `_favorite` swaps the saved copy into both `results` and `favorites` once it lands. It also creates one real race: un-hearting *during* the save leaves `_unfavorite` with no id to delete by, so it no-ops and `_favorite` settles the server when its id arrives.

`FoodSearchState.favorites` is the single source of truth for both the favourites tab and every result's heart; `loadFavorites()` only runs from `onInit`.

The heart is **optional on the tile** (`onToggleFavorite` nullable, the "pass no callback to disable" convention `MealSectionCard` and `DietCalendarDayCell` already use): `IngredientPickerPage` reuses the same tile to build a recipe, where favouriting isn't the job.

`VTSegmentedTabs` (design system) is the tab strip: a pill track with an indicator that slides between segments via `AnimatedAlign`, rather than Material's `SegmentedButton`, which snaps and brings its own chrome. It's generic over the value type, so `TrendRangeSelector` — still on `SegmentedButton` — could move onto it later.

## Recent searches

The Search tab's idle state lists past queries instead of the "search for a food" prompt, which now only shows for someone who has never searched (issue #86). Tapping one replays it; each row has an × and the header a Clear.

**Device-local, no table** — they're a typing shortcut, not user data worth a round-trip. `RecentSearchesLocalDataSource` (`data/diet/datasources/local/`) sits on `LocalStorageService` under the `diet.recentSearches` key, a sibling of `DietGoalsLocalDataSource`. Reads go through `get<List<dynamic>>(...)?.cast<String>()`: Hive stores the element type but hands the list back untyped, so asking it for a `List<String>` directly throws.

**Only a search that found something is kept.** A typo or a dead end is exactly what the list should spare you from replaying, so `FoodSearchCubit.search` records the query after a hit and skips it on an empty result or an error.

The ordering policy — most recent first, case-insensitive de-duplication (the newly typed spelling wins), capped at `_maxRecentSearches` — lives on `DietRepository`, next to `searchFoods`'s catalog/OFF merge, since both are read-then-decide over a datasource. Each mutating method **returns the updated list** so the cubit doesn't re-read to find out where the query landed.

`onInit` emits recent searches (a synchronous local read) *before* awaiting favourites, so the list is there instantly rather than behind a network call. That's two emits, and the first one is delivered even when it equals the initial state — `Cubit.emit` only de-duplicates after its first emission.

`VTSearchField` gained a `text` property synced in `didUpdateWidget` (the `CustomFoodForm` pattern): replaying a recent search has to fill the box, or you'd see results for "banana" over an empty field. Only a *change* to `text` is pushed in, so typing is never clobbered — `FoodSearchState.query` is set when a search runs, never on keystrokes.

## Copying meals

The diet `AppBar`'s copy icon opens `CopyMealsPage` (`/diet/copy`, issue #61): pick a source day on a calendar, tick which of its meals to bring over, and they're appended to the day the diet page is currently showing. The target day rides in on `CopyMealsExtra` and reaches the cubit through `VTPage`'s `cubitParam` (a `registerFactoryParam`, same as `RecipeFormCubit`) — the page never picks the date itself, it copies into whatever day you were already looking at.

**Copying appends, it never replaces.** Meals already logged on the target day stay put and the copied entries land alongside them, so nothing is destroyed by a mistaken tap; deleting the surplus is the user's job. This is also why there's no confirmation dialog.

A copy is **plain `food_logs` inserts**, not a new kind of row: `SupabaseDietDataSource.copyFoodLogs` rebuilds a `CreateFoodLogRequest` per entry against the target date and bulk-inserts them in one round-trip. The copies therefore reference the same `foods` rows, and the `bump_food_times_logged` trigger counts each one — logging via copy is still logging. Nothing marks a log as "copied": once inserted it's indistinguishable from a hand-logged one, which is what makes editing/deleting/adherence keep working untouched.

`CopyMealsState.macrosByDate` **accumulates across months** rather than being replaced per month (`{...state.macrosByDate, ...loaded}`), which is what lets a day picked in June survive browsing to July — the button names the source date, so an off-screen selection still reads clearly. It also means the source day's meals come free from the month load already in memory; picking a day fires no extra query. The target day is deliberately not tappable as its own source (`CopyMealsMonthGrid` passes a null `onTap` for it).

The calendar reuses `MonthNavigator` and `DietCalendarDayCell` from `pages/diet_history/widgets/` — the cell gained an optional `isSelected` (filled primary circle) and a **nullable** `onTap`, following `MealSectionCard`'s "pass no callback to disable" convention. Its existing "not tappable unless the day has a log" rule is exactly what a source picker wants, so days with nothing logged can't be chosen. The weekday header is the one piece not reused: history's carries a week-average column this page has no use for.

## History pages (diet, water, sleep)

All three features have the same history surface: a month calendar over a trends section, reached from a calendar icon in the feature's `AppBar` (`/diet/history`, `/water/history`, `/sleep/history`). Diet's came first (issue #55); water and sleep followed (issues #89, #90) and are what forced the shared pieces below out of `pages/diet_history/`.

**They all reduce to one shape: a day's value against a goal.** Diet is `totalCalories` vs `calorieGoal`, water `totalMl` vs `water.dailyGoalMl`, sleep `totalHours` vs `sleep.durationGoalHours`. That's why `GoalAdherence` (`core/goals/`, moved out of `domain/diet/entities/` once it stopped being diet's) and `DailyGoalAverage` (which replaced `CalorieAverage`) serve all three unchanged. Add a fourth tracker and it should fall out the same way.

**The calendar is a design-system component** (`design_system/components/calendar/`) and knows nothing about goals: `VTCalendarMonthGrid` takes a `dayColor(day)` callback where `null` means nothing was logged — which is what makes a cell plain and untappable — plus an optional `weekBadge` builder and an `isDayEnabled` veto. What counts as a good day is the feature's business, not the calendar's. `VTCalendarWeekBadge` renders `-` for a week with no data. `VTMonthNavigator` and `VTCalendarWeekdayHeader` moved there too, and `CopyMealsPage` uses the same grid (`isDayEnabled` is how the target day is barred from being its own source).

**Averages count only days that have data** (`DailyGoalAverage.fromValues`) — an untracked day means "no data", not "zero", and averaging zeros in would slander every day the user forgot to log. This is why every caller filters to logged days before handing values over.

`TrendRange` and `TrendRangeSelector` live in `presentation/general/`, shared by the three pages. The trend window is a rolling 7/30/90 days ending today and is **independent of the displayed month** — progression is the point, and paging month-by-month buries it — so each history cubit runs two separate range queries against the same use case.

Per-feature specifics: water and sleep calendars are read-only (`isDayEnabled: (_) => false`) because there's no per-day page to open, unlike diet's `DietDayPage`. Sleep is the one feature whose goal didn't exist: `sleep.durationGoalHours` (default 8) is new, stored device-local like water's and edited from a dialog on `SleepPage`. `DailySleep` sums a date's logs rather than assuming one night, since nothing stops a nap being logged too. `SleepQualitySplit` counts **only rated nights** — `qualityRating` is optional, so an unrated night is missing data, not a bad night.

## Body weight

Body weight (issue #101) is a tracker like water/sleep — `body_weight_logs` (user_id + logged_date + `weight_kg`, RLS scoped to the user) behind `BodyWeightRepository`/`SupabaseBodyWeightDataSource`, kept in **kilograms** the same way water is milliliters and workout load is kilograms; the display unit (kg/lb) is a presentation concern handled by `LoadConversion` (the same kg↔lb extension the workout load uses, *not* `WeightConversion`'s g↔oz). It is the "fourth tracker" the History-pages section anticipated, but it deliberately **breaks the shape**: weight has no daily goal, so there's no `GoalAdherence`/`DailyGoalAverage`, no goal-coloured calendar, and no month grid. It's a **trend**, not a day-vs-goal — so `BodyWeightHistoryPage` is a `TrendRangeSelector` over a `VTLineChart` (auto-scaled, see Charts) with min/max/average/change stats, and `BodyWeightPage` is a current-weight hero (`BodyWeightSummaryCard`: latest weight + a change pill + a mini `VTLineChart`) over a recent-entries list. No unique constraint on the date — a user may weigh more than once a day, and "current weight" is simply the most recent row by `logged_date` then `created_at` (`getLatest`, `order` on both descending).

**Logging is a ruler, not a text field.** `LogBodyWeightSheet` uses `VTWeightPicker` (design system, `components/inputs/`) — a horizontal ruler the user scrolls to dial in a weight, with a fixed centre pointer and a big read-out. Owned rather than the `vertical_weight_slider`/`animated_weight_picker` packages the owner linked, the same "own the look" call the charts make. It **opens at the last logged weight** (or 70 kg with no history), and is unit-agnostic: the sheet feeds it bounds/step/label in display units (metric 30–250 @ 0.1, imperial 60–550 @ 0.2) and converts the picked value back to kg. Two ruler traps encoded there and pinned by `vt_weight_picker_test.dart`: each tick lives in a ~10px slot, so a major label must be a `Positioned` box wider than the slot centred on the tick (a non-positioned `Stack` child is squeezed to the slot width and clips its own text; a `Column`/`Flex` per tick overflows by Infinity), and `_snap`'s `animateTo` must bail when already on a tick or its own `ScrollEndNotification` recurses into a stack overflow.

**The workout uses the latest weight for bodyweight exercises.** `GetLatestBodyWeightUseCase` feeds `WorkoutState.latestBodyWeightKg`, loaded in `WorkoutCubit.loadDate` non-blocking like the routine cycle and last sets (a failure just leaves the prefill unset). `WorkoutPage` passes it as `LogSetSheet`'s `defaultLoadKg` **only for `Equipment.bodyOnly` exercises**, pre-filling a new set's load with the user's body weight (editable — add a vest, subtract for assisted). This changes what a bodyweight set logs from 0 to the body weight, so volume (reps×load) finally counts bodyweight work — a deliberate revision of the "weight_kg is 0 external load for bodyweight" note in the Workout section. Consequence for tests: any workout test that loads a day must now also stub `GetLatestBodyWeightUseCase` (a `_noBodyWeightUseCase()` helper returning `Success(null)`), alongside the cycle and last-sets stubs.

## Diet history page

The diet page's `AppBar` calendar icon opens `DietHistoryPage` (`/diet/history`, issue #55) — a **read-only** analysis surface, deliberately separate from `DietDateSelector`'s `showDietCalendarSheet`, which is a date *picker* for choosing which day to edit. Two calendars, two jobs: one navigates, one explains.

It's one scrolling page (tabs were tried and dropped), stacking a calendar card over a trends section:

- **Calendar** — a month grid built as `DietCalendarWeekRow`s rather than a `GridView`, because each week row ends in a `WeekAverageBadge`: the week's mean calories, tinted by `GoalAdherence` like every other calorie figure. The average counts **only days that actually have food logged** (`CalorieAverage.fromLoggedDays`) — an untracked day means "no data", not "ate nothing", and averaging zeros in would slander every week you forgot to log. A week with nothing logged shows `-`. Day cells are tappable only when that day has a log.
- **Trends** — a `TrendRange` (7/30/90 days) drives `CaloriesTrendCard` (daily bars tinted by adherence against a dashed goal line), `MacrosTrendCard` (stacked protein/carbs/fat), and `MealSplitCard` (`MealCalorieSplit`, calories per meal as a `VTDistributionBar` plus per-meal share and daily average). The range is independent of the displayed month — progression is the point, and paging month-by-month buries it. `DietHistoryCubit` therefore runs two separate range queries (month for the calendar, rolling window for the trends) against the same `GetMacrosInRangeUseCase`.

`DietDayPage` (`/diet/history/day`) is what a calendar day opens. It takes the day's `DailyMacros` and `MacroGoals` through `DietDayExtra` and **has no cubit at all** — the history page already holds every day of the month in memory, so re-fetching one of them would be ceremony. It reuses `MacroSummaryCard` and `MealSectionCard` from `pages/diet/widgets/` so the day reads identically to the live diet page; `MealSectionCard`/`FoodLogTile` take **nullable** `onEditEntry`/`onDeleteEntry`/`onAddFood`, and passing none is what makes them read-only. `initiallyExpanded: true` opens every meal, since the point of the page is seeing everything at once. Importing another page's widgets is the deliberate exception here — the macro/meal cards are diet-domain widgets shared by the diet pages, and neither duplicating them nor hoisting domain-specific cards into the design system would be an improvement.

## Diet page visual language

The diet page (`DietPage`) is the app's showcase screen — the intended look is a modern nutrition tracker (MyFitnessPal / Lifesum family), and it sets the polish bar every other feature should match. Its structure:

- **Macro hero (`MacroSummaryCard`)** — a `VTMacroRing` calorie ring (progress = calories / goal, colored `colorScheme.primary`) sits left with the consumed total, `dietCaloriesOfGoal`, and a `dietCaloriesLeft`/`dietCaloriesOver` delta stacked in its center; the four macros sit right as color-coded `_MacroProgressRow`s (colored dot + label + `progressLabel` + a tinted `VTProgressBar`). Each macro has a fixed accent from `VTColors` — `macroProtein` (coral), `macroCarbs` (amber), `macroFat` (blue), `macroFiber` (teal); reuse these exact colors anywhere macros are shown so the color↔macro mapping stays learnable. The "Vitamins & minerals" toggle is a centered `TextButton.icon` (`unfold_more`/`unfold_less`) below a divider, expanding the sparse micronutrient list — same two-version idea as before (issue #54), just relocated out of the header.
- **Meal sections (`MealSectionCard`)** — each meal leads with a `_MealAvatar` (circular icon tinted by `mealTypeColor`, icon from `mealTypeIcon`, both in `meal_type_icon.dart`) and shows its calorie total as a `_CaloriePill` in the same accent; the whole card stays tap-to-collapse.
- **Logged foods (`FoodLogTile`)** — the row is tap-to-edit (`showEditFoodLogSheet`, see issue #60), keeping the trailing delete `IconButton` as the only other action. Editing covers **quantity and meal type** — the same two fields `LogFoodSheet` captures, so the edit form mirrors the create form; the food itself isn't swappable (delete and re-add instead) and the date is changed by moving the whole day, not one entry. `EditFoodLogSheet` is a separate widget from `LogFoodSheet` rather than one sheet doing both: they hang off different cubits (`DietCubit` vs `FoodSearchCubit`), and abstracting over that would cost more than the duplicated quantity field. It follows `LogFoodSheet`'s shape otherwise — `DietCubit.updateLog` **returns its `Result`** (rather than emitting `DietError`) so the sheet shows the failure inline and pops on success, the same split `FoodSearchCubit.logFood` uses.
- **FAB** — a `FloatingActionButton.extended` (`dietAddFood` label), not a bare icon, so the primary action reads clearly.

`VTMacroRing` and the colorable `VTProgressBar` are deliberately generic (a ring/bar against any goal, any color) so the workout and water features can reuse them for their own headline metrics rather than each rolling its own progress visual.

## Nutrition label scanning

Reading a nutrition label from a photo goes to **Claude, through the `scan-nutrition-label` Supabase Edge Function** (issue #74) — not to on-device OCR. `google_mlkit_text_recognition` used to do this and is gone, along with `TextRecognitionService`, `NutritionOcrDataSource` and the ~125-line regex `NutritionLabelParser`: MLKit gave us text lines with bounding boxes, and everything hard (which number is protein, per-serving → per-100g, kJ → kcal, any language) was the parser's problem. A model that reads the label solves those directly, and the parser and its 10 tests were deleted rather than ported.

**The domain boundary didn't move**: `CustomFoodCubit` → `ScanNutritionLabelUseCase` → `DietRepository.scanNutritionLabel(imagePath)` → `SupabaseNutritionScanDataSource` is the same chain as before, and `ScannedNutritionFacts` is unchanged. Only the datasource swapped, which is the point of the layering.

**The Edge Function exists because of the API key.** An Anthropic key shipped in a mobile binary is extractable by anyone who downloads the app, so the key lives as a Supabase secret and the app only ever calls the function. The function verifies the caller's JWT, which the app's anonymous session already supplies. Creating it and setting the secret is a runbook in [`docs/nutrition-scan-setup.md`](docs/nutrition-scan-setup.md) — scanning fails until both are done (the form still works; scan only pre-fills fields you can always type).

**The function's source is deliberately not in this repo** — it lives only in the Supabase project, and is edited and deployed from the dashboard. Only the client half is tracked here (`SupabaseFunction.scanNutritionLabel` + `SupabaseNutritionScanDataSource`), so don't go looking for a `supabase/functions/` directory or offer to `supabase functions deploy` it. The cost is that the current prompt and schema exist nowhere in version control; the benefit is that no credential-adjacent server code rides along in the app repo. A second function should follow the same split.

`SupabaseFunction` (`core/services/supabase/supabase_function.dart`) pairs the Dart case with the deployed function's name exactly like `SupabaseTable`/`SupabaseBucket` do, and `SupabaseService.invoke` takes the enum — a new function means an enum case, not a raw string at the call site.

The function pins **`claude-opus-4-8` with structured outputs** (`output_config.format` + a JSON schema): the schema is what guarantees the response parses into `ScannedNutritionFacts`, so there's no defensive parsing on the Dart side. Every nutrient is nullable and the prompt says to return null rather than guess — a wrong macro silently poisons a shared catalog row, so a blank field the user fills in is the better failure. The per-100g conversion, unit handling and "is this even a label" judgement all live in that prompt now, which is the real tradeoff: **it's no longer covered by Dart tests.** `DietRepository.scanNutritionLabel` is a thin pass-through, and the two repository tests that used to exercise the parser through it were removed.

Scanning now needs network and costs roughly $0.01–0.02 per scan; the old OCR was free and offline. That was the deliberate trade for a scan that actually works on a real label.

## Meal photo scanning

Photographing a whole plate and logging its items (issue #149) is the **same Edge-Function split as the nutrition-label scan**, one step up: instead of reading one label into one food, the `scan-meal` function asks Claude to identify *every* item on the plate, estimate each portion in grams, and return per-100g macros for each. `CustomFoodCubit → ScanNutritionLabelUseCase` has a direct sibling: `MealScanCubit → ScanMealUseCase → DietRepository.scanMeal → SupabaseMealScanDataSource`, invoking `SupabaseFunction.scanMeal` ('scan-meal') with the same `{imageBase64, fileExtension}` body. `ScannedMeal` (a list of `ScannedMealItem`) is the response entity, parsed by `ScannedMeal.fromMap`; **every macro defaults to 0 on the Dart side** (`ScannedMealItem.fromMap`'s `?? 0`), so the schema can omit a number without a null branch downstream, mirroring `ScannedNutritionFacts`. The function's source lives only in the Supabase dashboard (runbook in [`docs/meal-scan-setup.md`](docs/meal-scan-setup.md)), same as the label scanner — don't look for a `supabase/functions/` dir. Reuses the same `ANTHROPIC_API_KEY` secret; only the deployment name differs.

**A scanned item logs through the ordinary save-then-log path — no new write path, no schema change.** `ScannedMealItem.toFood()` builds a `FoodSource.custom` food (the AI estimate joins the shared catalog exactly like a hand-entered custom food; `foods.source` gains no new value, so no migration). `LogScannedMealUseCase` loops the included items, doing the `saveFood`-then-`logFood` step per item that `LogFoodUseCase` does for one — sequential writes, not a transaction, the `SaveRecipeUseCase`/`copyFoodLogs` shape (a mid-way failure can leave an orphan catalog row, harmless). Everything below the log (macros, adherence, calendar, history) is untouched because grams were always the truth: `quantityGrams` is the on-plate estimate, and the per-100g macros ride on the food row.

**The review step is load-bearing, because the macros are guesses.** `MealScanPage` (`AppRoute.mealScan`, reached from a camera icon in the diet `AppBar`, `cubitParam`-seeded with the day being viewed like `CopyMealsPage`) shows each detected item as a `ScannedMealItemCard` — an include checkbox, an editable grams field (seeded from `estimatedGrams`), and a live calorie badge — over a `MealScanTypeSelector` (the meal it lands in, defaulted by time of day). `MealScanEntry` is the per-row edit state (item + `gramsText` + `isIncluded`), the presentation twin of `CustomFoodState`'s nutrient map; `MealScanState.canLog` gates the "Add to diary" button on at least one included item with a valid amount. `MealScanFoundNothing` (empty `items`) is a warning toast + a "no food detected" empty state inviting another photo — the "return nothing rather than guess" failure the label scanner's null-not-guess rule also follows, since a hallucinated plate would poison the shared catalog. On success the page pops `true`; `DietPage` refreshes and shows a success toast, the `CopyMealsPage` return convention.

## Custom food page

Adding a food the catalog doesn't have is a **full page** (`CustomFoodPage`, `AppRoute.customFood`), not a bottom sheet (see issue #71) — the form is too tall for a sheet once it carries a photo, a name, a brand, a scan action and five nutrient fields. `FoodSearchPage`'s app-bar action does `await context.pushRoute<Food>(.customFood)` and, on a non-null return, opens `showLogFoodSheet` — the page itself never logs anything, it only builds an unsaved `Food` and pops it back (`Navigator.of(context).pop(food)` from the `CustomFoodReady` presentation event, matching how every other page pops). Persisting still happens in `LogFoodUseCase`'s save-then-log, exactly as before.

**The page holds no logic — `CustomFoodCubit` does.** It owns validation (`CustomFoodState.isComplete`), number parsing (comma-or-dot, via `nutrientChanged`), the photo pick, the label scan, the image upload, and building the `Food`. The five nutrients are a `Map<CustomFoodNutrient, double>` on the state rather than five nullable fields: an absent key *is* "empty" (which `copyWith` could never express for a nullable `double`), and one enum drives the fields, the scan mapping (`CustomFoodNutrient.valueOf(facts)`), the labels and the macro accent colors — adding a nutrient to the form is one enum case.

`CustomFoodForm` is the only stateful piece, owning the `TextEditingController`s. It pushes every keystroke into the cubit and **syncs back from state in `didUpdateWidget`**, comparing each controller's *parsed* value against the state's — so a scan that fills the fields flows through state like any other change, while a half-typed `"5."` (parses to null, same as the state) never gets clobbered mid-typing. There's deliberately no "fields were scanned" presentation event; state is the single path.

Visually it follows the diet page's vocabulary: a stretchy `SliverAppBar` whose background is `CustomFoodPhotoHeader` (photo or an `add_a_photo` placeholder over a `primaryContainer→surface` gradient, `StretchMode.zoomBackground` plus `BouncingScrollPhysics` so it zooms on overscroll on both platforms), a `CustomFoodScanCard` using the leading circular accent avatar, nutrient fields dotted in their macro accent, a live `CustomFoodEnergySplitCard` (protein/carbs/fat energy shares at 4/4/9 kcal per gram, `AnimatedSize`d in once a macro is typed), and staggered `VTAppearEffect`s. The page has **no title** — the photo is the hero. Loading (upload/scan) is the global overlay, never a state field; errors, the incomplete-form warning and the "couldn't read the label" case are all presentation events the page turns into toasts (the l10n strings live in the page, not the cubit).

## Workout

The workout feature (issue #8, sliced into #93–#97; this is #93's half) is built from a **shared exercise catalog** plus **per-user sessions**, which is the same split the diet feature uses (`foods` shared, `food_logs` private) — and for the same reason: an exercise is a fact about the world, a set is a fact about you.

`exercises` mirrors `foods`' terms exactly: anyone authenticated reads every row, insert/update/delete stay scoped to the row's own `user_id`, and the bulk-imported rows have a **null `user_id`** (written through the service-role key, which bypasses RLS) so nobody can edit them through the app. `exercises.times_logged` is the popularity counter search ranks by, kept in sync by the `bump_exercise_times_logged` trigger — a `security definer` copy of `bump_food_times_logged`, for the same reason (the logging user doesn't own the imported exercise row). It counts **`workout_exercises`, not `workout_sets`**: the signal is "how many sessions used this exercise", so a fourth set isn't a second vote.

**Names and instructions are locale-keyed `jsonb` (`{"en": ..., "pt": ...}`), not a `name`/`name_pt` column pair** — same reasoning as `foods.micronutrients`: the source dataset is English-only and the import translates it, so adding a locale is an import re-run plus an ARB file, never a migration. `Exercise.nameFor(locale)` falls back to English, so an untranslated row still renders. Catalog search doesn't query those blobs directly: `exercises.search_text` is a **generated column** folding every locale's name into one lowercase, accent-free string, so `ilike` finds "Tríceps Testa" whether the user types "triceps", "tríceps", or the English name. It reads every value of the blob rather than named locales (via the immutable `exercise_search_text`/`immutable_unaccent` wrappers a generated column requires), which is what keeps a new locale migration-free. `AccentFolding.fold` (`core/text/`) folds the query the same way on the Dart side — fold one side only and accented searches silently miss.

**The catalog comes from [free-exercise-db](https://github.com/yuhonas/free-exercise-db)** (873 exercises, public domain, every one photographed), imported by `tool/import_exercise_catalog.dart` — the same shape as `tool/import_food_catalog.dart`: service-role upsert keyed on a natural id (`slug`), `user_id` omitted rather than nulled so a re-run never reassigns a row's owner. It mirrors the photos into the `exercise-images` bucket instead of hot-linking raw.githubusercontent.com (rate-limited, unversioned, could vanish — and `food-images` already set the precedent), and translates names + instructions to pt via Claude with structured outputs. Two things learned the hard way and encoded there: **every HTTP call needs an explicit timeout** (`package:http` waits forever, and a stalled connection hung a real run at 136/873 with no output), and **a failed upsert must abort the run** rather than log and continue, or the tool cheerfully reports `Done. Imported 873` over an empty catalog.

A session is three tables — `workouts` (the day) → `workout_exercises` (what, in order) → `workout_sets` (reps + `weight_kg`) — with RLS walking up to the owning workout rather than duplicating `user_id`, the shape `recipe_ingredients` established. Two deliberate choices:

- **`workout_exercises.exercise_id` is `on delete restrict`, not cascade** (unlike `food_logs` → `foods`). Silently erasing sessions from someone's history because a custom exercise's author deleted it would destroy real training data.
- **`weight_kg` allows 0 rather than being null-for-bodyweight.** A pull-up and a 0kg bar are both "no external load", and one numeric keeps every volume sum total-able with no null branch. The cost is that bodyweight work contributes 0 volume — which is exactly why set count is a first-class metric next to tonnage, not decoration.

**Volume is Σ (reps × load) per session** — the tonnage. `WorkoutVolume` is the mixin that computes it, folding over a `List<WorkoutSet>` so one exercise and a whole workout total up through the same code (`Workout.sets` flattens its exercises' sets) — the generalisation `MacroTotals`/`FoodPortion` already made for diet. #96 defines the history surface on top of it.

Loads are stored in **kilograms**, the base unit, exactly as `quantityGrams`/`amountMl` are. `UnitSystem` gained a `LoadConversion` (kg↔lb) extension **separate from the existing `WeightConversion` (g↔oz)** — two independent conversions in one file, as `VolumeConversion` already established. They are not interchangeable: food quantity is grams/ounces, barbell load is kilograms/pounds.

UI: `WorkoutPage` (`/workout`) is a day view with a date selector, a volume/sets/reps summary, and one `WorkoutExerciseCard` per exercise. Its FAB pushes `ExerciseSearchPage` (`/workout/exercise-search`), which **pops the picked `Exercise` back rather than logging it** — the `IngredientPickerPage` convention, keeping the search single-purpose. `ExerciseDetailPage` (`/workout/exercise-search/detail`) is read-only and **has no cubit**: whoever pushed it already holds the `Exercise`, so re-fetching would be ceremony (`DietDayPage`'s reasoning). `LogSetSheet` is one widget doing both create and edit — worth it here, unlike `LogFoodSheet` vs `EditFoodLogSheet`, because both paths hang off the same `WorkoutCubit`; it returns its `Result` so failures show inline and the sheet pops on success.

`VTRemoteImage` (design system) is `VTFoodImage` generalised — a network image with a caller-supplied placeholder icon — since the catalog needed the same behaviour with a different fallback. `VTFoodImage` is now a thin wrapper passing `Icons.restaurant_outlined`, so diet's call sites are untouched. **Every network image in the app goes through `cached_network_image_ce`** (issue #59), not raw `Image.network` — the `_ce` fork is used over the mainline `cached_network_image` for the same reason `hive_ce`/`hive_ce_flutter` are (it's the family that stays compatible with the project's `hive_ce` toolchain). `VTRemoteImage` and `VTPhotoHeader` are the only two widgets that build a `CachedNetworkImage`; both fall back to their existing placeholder via `errorBuilder`, so a new call site caches for free just by using them rather than reaching for `Image.network`. Body regions get their own fixed `VTColors.bodyRegion*` accents (the `macro*` idea applied to muscle groups); don't reach for `warning`/`macroCarbs` for a region — they're the same hex, so two regions would collide.

## The rest timer, and the exercise workspace

The workout screen's two gaps were connected (issue #163): there was **no rest timer at all**, and `ExerciseDetailPage` was a **dead end** — photos and instructions, but you had to back out of it to log anything.

**`ExerciseWorkoutPage` is the workspace.** It replaces the read-only detail page as what an exercise card opens: photos, sets, the rest timer, the instructions and Finish, on one screen. `ExerciseWorkoutCubit` takes the `WorkoutExercise` through `VTPage`'s `cubitParam` (the `RecipeFormCubit` shape) and **updates its own state from each use case's return value** rather than reloading — `LogSetUseCase`, `UpdateSetUseCase` and `SetWorkoutExerciseCompletedUseCase` all hand back the persisted entity, so a round trip would be ceremony. `WorkoutPage` reloads the day on pop.

**The exercise page owns adding a set; the day list keeps repeat.** Once the page exists, an "Add set" button on the card is a second door to the same room — so the card offers only **repeat** (one tap, no sheet) and editing an existing set. Anything that needs the sheet happens on the page. Finishing an exercise pops back to the day view, since finishing is the end of what that page is for.

**`RestTimerCubit` is a lazy singleton provided at the root**, next to `AppCubit` and `PremiumCubit` — not a widget and not a page cubit, because a rest has to keep counting while you move between the day view and the exercise page. Three rules, all pinned by `rest_timer_cubit_test.dart` and all mutation-verified:

- **`start` replaces, it never stacks.** Logging a second set mid-rest restarts the clock rather than adding to it.
- **Extending grows `total` as well as `remaining`.** Growing only `remaining` makes the progress bar jump backwards, since it is the ratio of the two.
- **A finished rest clears itself** (and fires `VTHaptics.success()`), so nothing lingers at 0:00. The last three seconds tick with `VTHaptics.selection()`, so you can feel it end without watching.
- **Shortening past zero ends the rest** rather than going negative.

**The rest length is a device-local preference**, not a constant — `WorkoutLocalDataSource`'s `workout.restSeconds` behind `GetRestDurationUseCase`/`SaveRestDurationUseCase`, the same shape as the water and sleep goals. `start()` with no argument reads it, so changing it applies from the next set. The timer turns `VTColors.coral` in its final ten seconds, and carries the exercise's name so a rest you come back to says what it belongs to.

**The countdown is the colour, not a bar.** `VTRestTimer` has no progress track — the card itself lerps **green → amber → red** as the rest runs down, which reads at a glance without occupying a row. The mid stop is `VTColors.warningStrong`, a deeper sibling of `warning`, because **`warning` itself is only 2.17:1 against white and cannot carry the ink**. Every point along the ramp clears 4.5:1, asserted at 101 samples by `vt_rest_timer_test.dart` — the same class of bug as the accent-avatar floor, caught the same way. `VTRestTimer` otherwise carries the countdown in tabular figures, −30s/+30s, the rest length and Skip. It docks in `WorkoutPage`'s `bottomNavigationBar` and sits inline on the exercise page. A test that exercises the timer needs `TestWidgetsFlutterBinding.ensureInitialized()`, because the completion haptic reaches a platform channel.

## Workout page visual language

`WorkoutPage` is built from the same vocabulary as the diet page (issue #108) rather than a parallel look — the pieces below are what "match the diet page's polish bar" actually meant in practice:

- **`WorkoutSummaryCard`** mirrors `MacroSummaryCard`: a `VTMacroRing` left, metrics right, accent badges under a divider. **The ring counts finished exercises, not tonnage** — `VTMacroRing` is "a value against a goal", and volume/sets/reps have no goal to ring against, while the session does ("2/5, 3 to go"). It turns `VTColors.success` once `state.isFinished`, so the ring is the same payoff signal as `WorkoutFinishedCard`. `WorkoutState.completedExercises`/`exercises` exist for it.
- **One accent per card, not three.** The three metrics would naturally take `macro*`-style fixed accents, but `VTColors.bodyRegion*` are the *same hexes* as `macro*` and the region badges sit on the same card — a blue "Sets" dot next to a blue "Back" badge is a collision. So volume (the headline) is `colorScheme.primary` and sets/reps stay neutral via `WorkoutMetric`'s nullable `color`. Any new figure on a card that shows regions should follow that rule.
- **A set row is an accent index chip + reps + a load `VTBadge`** (`WorkoutSetRow`, tinted by the exercise's region accent), not one flat `"10 reps · 40 kg"` string — which is why `workoutSetSummaryWeighted` is gone. Bodyweight renders as the same badge in a neutral tint rather than leaving the column blank, so the badges read as one column.
- `WorkoutSummaryCard` is pinned by a widget test at **320px in both locales** asserting no overflow (`workout_summary_card_test.dart`). The ring and the metric column split one row, and pt strings are longer than en — this is the class of break that never shows up on a wide test surface.

**A finished exercise reads as an achievement, not as a disabled row** — it's the payoff for checking the thing off. Three signals carry it, and **none of them touches the text**: the card recedes with a success tint, `WorkoutExerciseThumbnail` drains the photo to greyscale, and the name is struck through.

- **Obfuscate the photo, never the text.** `Opacity` over the card is the obvious way to "dim the finished ones" and it's wrong twice over: it drags `workoutCompletedSummary` to **2.7:1** even at a gentle 0.7 (AA needs 4.5), and that summary is the one thing a collapsed card exists to keep saying.
- **The photo greys out rather than being stamped.** An icon over the image covers the one thing that identifies the exercise, and a check over the catalog's mostly-light gym shots is a coin toss on contrast anyway. `ColorFiltered` with the **luminance weights** (0.2126/0.7152/0.0722 — the same ones a contrast ratio is built from) makes the grey match how bright the photo actually read in colour.
- **The tint blends against `colorScheme.surface`, not against the card colour.** That's what makes one expression right in both themes: it washes lighter on light and darker on dark, receding either way. Blending against the card instead darkens light mode toward the caption colour, which measured **4.32:1 — under AA at only 10% tint**, because a caption on a card starts at 4.78:1 and has almost no headroom. Any new "tinted card state" has to be re-measured, not eyeballed.
- **A Flutter-debug-style `Banner` was considered and dropped on geometry.** `banner.dart` fixes `_kOffset = 40`, an 80px-wide strip and a 10.2px font, so on a 44px thumbnail it falls outside the box entirely; on the card it fits, but `topEnd` runs straight over the reopen button's 48px tap target and `topStart` crosses the photo it was meant to spare.
- **Filters (all/todo/done) were considered and dropped.** A workout is 4–8 exercises and finished ones already collapse to ~76px, so there's no wall to hide; filtering answers a *legibility* problem with *navigation*, hides what the summary ring is counting, and ends the session on an empty list instead of the finished card. Auto-sorting finished ones to the bottom was dropped too: `position` is the order you train in (see Workout routines), so resorting the screen fights the plan.

## Design system: onSurfaceVariant

`onSurfaceVariant` used to be set to the **same hex as `onSurface`** in both schemes, so `VTTextStyles.caption`/`overline` — and every "muted" `copyWith(color: colorScheme.onSurfaceVariant)` at a call site — rendered at full body contrast. Nothing receded, which is most of what read as "raw", and it silently no-op'd intent already in the code (`WorkoutExerciseCard` greys a completed exercise that way). It's now `VTColors.onSurfaceVariantLight`/`Dark`, verified at **4.78:1 (light) / 6.82:1 (dark) on a card** — comfortably past WCAG AA's 4.5:1 for small text, and well below body text's ~13:1 so the hierarchy is real. Keep any replacement inside those bounds: a "muted" token that fails AA is a bug, and one that matches `onSurface` isn't a variant at all.

## Workout: repeating sets and finishing exercises

Two things the logging screen learned from real use (issue #102).

**An exercise with no sets can't be finished.** "Done" means you did it, and allowing it would let a workout of untouched exercises be marked complete and congratulate the user for lifting nothing — and since #94 pre-fills a routine's exercises, that's one tap away. The check button stays visible but disabled with a tooltip saying what's missing, rather than vanishing; removing the exercise is the honest way to clear it. Reopening is always allowed, whatever the set count. `WorkoutCubit.setExerciseCompleted` takes the `WorkoutExercise` rather than its id precisely so it can enforce this itself instead of trusting the UI.

**Repeat** duplicates the exercise's last set in one tap — sets of an exercise are usually identical (4×10 at the same load), so the common case shouldn't cost a sheet and two numbers. It has **no use case of its own**: `WorkoutCubit.repeatLastSet` reads the values off state it already holds and calls the ordinary `LogSetUseCase`. The button only appears once there's a set to repeat, and the cubit guards the same rule rather than trusting the UI to.

**`workout_exercises.completed_at`** (nullable) is what marks an exercise done; it collapses the card, and once every exercise of the day has one, `Workout.isComplete` shows the end-of-workout message. Three decisions worth keeping:

- **Persisted, not screen state.** Closing the app must not lose your progress, and the end-of-workout message needs something durable to test — otherwise it appears and vanishes on no criterion.
- **Never derived from "has ≥1 set".** Those are different facts: one set of a planned four isn't done, and — since #94 — a workout started from a routine is *born* with the previous session's sets already filled in, so deriving it would declare the workout finished before the user lifts anything. `Workout.isComplete` has a test pinning exactly that.
- **Reversible, and the collapsed card still carries its summary.** Unmarking writes `null` back to the same column, so a mistaken tap costs one tap. The collapsed card keeps the exercise name and its set count, because "tidy the screen" must not mean "lose the information".

`Workout.isComplete` is false for an empty workout: nothing was finished, so there's nothing to congratulate.

## Workout routines

A routine is a named, ordered list of exercises ("Treino A — Peito e tríceps"); a user's routines form a **cycle**, and the app suggests which one is next (issue #94). `routines` + `routine_exercises` are private per user like `recipes`, with `routine_exercises` walking up to its parent's owner instead of duplicating `user_id`.

**The rotation is position-based, never calendar-based**, and lives on `RoutineCycle` (`domain/workout/entities/`) as a pure `next` getter over `routines` + `lastRoutineId` — no dates anywhere in it. The next routine is simply the one after the last routine-backed workout, wrapping at the end. Training Mon/Wed/Fri, twice in a day, or after a three-week layoff all advance it identically, and **missing a day never skips a routine** the way binding routines to weekdays would. Binding to weekdays was the alternative the issue raised and it's dropped for exactly that reason. With no usable history it suggests the first routine, which is also the fallback when the last-used routine has since been deleted.

`workouts.routine_id` is **nullable and `on delete set null`**. Nullable because a one-off workout belongs to no routine and stays perfectly valid — the FAB path from #93 is untouched, and only routine-backed workouts participate in the cycle. `set null` rather than cascade because deleting a routine must not delete the training history performed under it; the workout survives and just stops naming where it came from, the same call recipes made by leaving their foods row behind.

**The cycle order is editable, and that's load-bearing**: `position` decides what's suggested next, so `RoutinesPage` is a `ReorderableListView` and `reorderRoutines` rewrites positions (optimistically, reverting on failure — the same call favouriting makes). A `position` only assigned at creation would freeze a mis-built A/B/C forever.

Three `ReorderableListView` rules this feature learned the hard way — all three shipped broken first, so copy them:
- **Key by the item's own id, never the index.** `ValueKey('$id-$index')` changes for every row the moment one is dragged, so the list loses track of its widgets and the drag doesn't stick.
- **Use `onReorderItem`, not the deprecated `onReorder`.** It hands back a `newIndex` already adjusted for the removed item, so the cubit inserts at that index directly — re-deriving it (`newIndex > oldIndex ? newIndex - 1 : newIndex`) double-corrects and lands a downward drag one slot short.
- **Always pass `buildDefaultDragHandles: false` and supply a real handle** — `ReorderableDragStartListener(index: index, child: const VTDragHandle())`. The default handles only render on **desktop**; on a touch platform the list is drag-on-long-press with no affordance whatsoever, so it looks immovable. It also matters that the grip is off the row: these rows are tap-to-edit `VTCard`s, and a long press there is ambiguous between "edit" and "drag" — the tap recognizer usually wins. An `Icons.drag_handle` that isn't wrapped in a listener is worse than no icon: it advertises a gesture that does nothing.

Reorder is **verified by a widget test that performs the drag** (`routines_page_test.dart`), not by inspection — every one of the bugs above looked correct in the source. Two things that test had to get right: drag the handle in **small steps** (`moveBy(Offset(0, 10))` in a loop, pumping between), because the list tracks the pointer as it travels and one large `moveBy` skips the overlap it reorders on; and measure the tile with `tester.getSize` rather than guessing a pixel distance.

A routine holds **distinct** exercises — `RoutineFormCubit.addExercise` no-ops on one already in the list. Repeating an exercise is what sets are for, and distinctness is what makes the id a usable reorder key.

**Starting a routine pre-fills each exercise with the sets from the last time it was trained** (`StartWorkoutFromRoutineUseCase`), which is why it absorbed the core of issue #95 (the standalone-add path and the visible hint came later — see below). Those sets are **recorded, not suggested**: the user adjusts what changed and deletes what they didn't do. The trade is deliberate — the common case (same session, maybe one number up) becomes near-zero typing, at the cost of a workout abandoned right after starting leaving real-looking sets behind. An exercise never trained before starts empty.

`getLastSetsByExercise` is queried **from the `workouts` side, not `workout_exercises`**, because PostgREST's `order` on an embedded resource sorts the *embedded* rows, not the parents — ordering `workout_exercises` by `workouts(performed_date)` silently sorts nothing. Walking workouts newest-first and taking the first hit per exercise is what actually gives "last time"; it's bounded by `_lastSetsWorkoutLookback` so a user with years of history doesn't drag their whole log over the wire.

`WorkoutCubit.loadDate` loads the cycle alongside the day but **never lets it block**: a cycle failure leaves `state.cycle` empty, which just hides the suggestion card. An error dialog over a day that loaded fine would be worse than a missing suggestion. The consequence for tests: any test that loads a day must stub `GetRoutineCycleUseCase` (factories never stub — see Testing).

## Workout: remembering the last session (rest of #95)

#94 pre-filled sets for the *routine* path; #95 finished the job for the standalone one and added the visible reference. Both keep #94's rule: pre-filled sets are **recorded, not suggested**.

**Adding an exercise by the FAB now pre-fills it too.** `AddExerciseToWorkoutUseCase` is the standalone twin of `StartWorkoutFromRoutineUseCase` — it does the same `getLastSetsByExercise` + `logSetsBulk` for one exercise. It fetches last sets *before* creating/adding, so a lookup failure doesn't leave a fresh empty workout behind, and because the exercise isn't in today's workout yet, the newest hit is genuinely the previous session (no exclude-today needed on this path). An exercise never trained before adds empty; a failed `logSetsBulk` leaves it added-but-empty, the same partial-write trade the routine path documents.

**The "last time: 4×10 · 40kg" hint** on `WorkoutExerciseCard` is a stable reference to the *previous* session, shown alongside today's (editable) sets — so the user can tell the pre-filled numbers came from last time rather than being a fresh suggestion. Its data is a separate `WorkoutState.lastSetsByExercise`, loaded in `loadDate` exactly like the cycle: non-blocking, a failure just hides the hint. This is why `getLastSetsByExercise` gained an optional **`before`** — passed `before: date` here so "last time" excludes the day on screen; **without it the hint would just echo today's own sets**. The two existing callers (routine start, standalone add) pass no `before`. `GetLastSetsByExerciseUseCase` is the cubit's use case for it; the routine/standalone use cases reach the repo method directly since they already orchestrate. `WorkoutSetsSummary.format` renders the line — grouping identical sets as `n×reps · load` (the common case, per repeating-sets above) and falling back to `n sets · top load` when they differ, rather than picking one set's reps and lying about the rest. Consequence for tests: any test loading a day **with exercises** must also stub `GetLastSetsByExerciseUseCase` (empty-day loads early-return before it, so those don't).

## Navigation

No bottom nav bar — `HomePage` (`/`) is the single entrypoint: a settings action in the `AppBar` plus a grid of `VTFeatureTile`s (one per feature) that push their feature route. Each feature page owns its own `Scaffold`/`AppBar`; there is no shared shell.

Routes are never referenced by raw path string outside of `app_router.dart`/`routing/routes/`. `AppRoute` (`lib/app/presentation/routing/app_route.dart`) is an enum pairing each route's name with its path. `VTRoute` (`lib/app/presentation/routing/vt_route.dart`) is an abstract class exposing `route` (the `AppRoute`), `builder`, and an optional `redirect` (a real `GoRouterRedirect`, `null` by default) as getters to override, plus a concrete `toGoRoute()` converting itself to a `GoRoute`. Each route is its own concrete class in `lib/app/presentation/routing/routes/` (`DietRoute`, `WaterRoute`, `HomeRoute`, ...), one file per route, overriding only what it needs — most just override `route`/`builder`; `HomeRoute` is the one that also overrides `redirect`. The onboarding gate is that one route-specific `redirect`, not a router-wide one: `HomeRoute` takes a `HasSeenOnboardingUseCase` constructor dependency and redirects to `AppRoute.onboarding.path` unless it returns true, since `home` is the app's sole entrypoint (`initialLocation`) — there's no need to also guard every other route individually. `AppRouter._routes` (`app_router.dart`) is a `List<VTRoute>` instantiating each of these classes (`HomeRoute(hasSeenOnboardingUseCase: G())`, the rest with no arguments), turned into `GoRouter`'s `routes:` with `.map((route) => route.toGoRoute()).toList()`. Elsewhere, navigate with the `BuildContext` extension in `lib/app/core/navigation/navigation_extensions.dart`: `context.pushRoute(.diet)` / `context.goRoute(.settings)`, optionally with `extra:` for arguments. Add a new feature by adding a case to `AppRoute`, a new `XRoute` class in `routing/routes/`, and its entry in `AppRouter._routes`, plus a tile to `HomePage`.

## User profile

A signed-up user has a **display name** and an **avatar** — either a chosen preset or an uploaded photo — shown on the home app-bar and the profile header (issue #117). The name/avatar are captured at sign-up and editable later from Profile → Edit.

**Stored on the Supabase auth user's `user_metadata`, not a `profiles` table** — `display_name`, `avatar_id`, `avatar_url`, written through `auth.updateUser(data: {...})` (the same call `signUp` already made) and read back off `currentUser.userMetadata`. Datasources never touch the SDK's `User` type: `SupabaseService.currentUserMetadata` exposes the map, the way `currentUserEmail` does. `AuthenticatedUser` gained the three nullable fields plus an `initial` (name-or-email first letter, the last-resort avatar).

**The avatar is a preset id *or* a photo url, never both.** `SupabaseAuthDataSource._profileData` always writes all three keys, setting the unused avatar field to an explicit `null` — that null is what clears the other when you switch, since Supabase merges the provided keys. `AuthCubit._resolveDraftAvatar` enforces the same at the source: a draft photo is uploaded (`UploadAvatarUseCase` → the `avatars` bucket, mirroring food images) and becomes a url with a null id; a draft preset passes through with a null url. Display precedence lives in one place, `VTProfileAvatar`: photo (or `previewBytes` while picking) → preset (`VTAvatarCatalog.byId`) → initial → person icon.

**"Emoji" in the issue became preset avatars** (owner clarified mid-build): `VTAvatarCatalog` is a curated list of `VTAvatarOption`s (a glyph on a two-colour gradient) keyed by a **stable string id** — never the list index, since reordering must not change what an existing profile resolves to. `VTAvatarPickerSheet` is the gallery. Adding an option is one catalog entry, no asset pipeline and no dependency, the same call the charts and the (rejected) full emoji keyboard made.

**Auth is two pages, not one toggled form** (`SignInPage`/`SignUpPage`, routes `.signIn`/`.signUp`) — sign-up carries the extra name+avatar fields, so a chip toggle was dropped for `AuthCubit.signIn`/`signUp` (there's no `isSignUpMode` anymore). The cross-links use a **forwarding push**: "log in" from sign-up pushes `.signIn` and, if that returns `true`, pops itself `true` too, so the original caller (profile/onboarding) still learns the user authenticated. `AuthState` holds a **draft avatar** (`draftAvatarId`/`draftAvatarBytes`) separate from the persisted `user`; `copyWith` can't clear nullable draft fields, so `withPresetAvatar`/`withPhotoAvatar`/`withoutAvatar` rebuild instead. `EditProfilePage` reuses `AuthCubit` and the shared `AvatarPicker`, seeding the draft from the current user via `seedAvatarFrom`.

## Premium

The two Anthropic-backed features — the meal photo scan and the nutrition label scan — sit behind a premium entitlement (issue #153); everything else stays free, since nothing else has a per-use cost. Runbook in [`docs/premium-setup.md`](docs/premium-setup.md), which also explains how store subscriptions work end to end.

**Entitlement is a `subscriptions` table, never `user_metadata`.** `user_metadata` is client-writable through `auth.updateUser` (it is where `display_name`/`avatar_id` live, and `SupabaseAuthDataSource` already writes it), so an `is_premium` flag there could be set by any user against themselves — the profile fields are safe there precisely because nothing is gated on them. `subscriptions` (user_id as PK — what you are entitled to *now*, not a purchase history) therefore has **exactly one RLS policy, `for select`**. The absence of insert/update/delete policies is the security property: under RLS an operation with no matching policy is denied, so only the service-role key can write it (a human in the SQL editor today, a store webhook later). Do not "complete the set".

**The Edge Function is the gate; the client lock is UX.** `scan-meal` and `scan-nutrition-label` each read the caller's id from the *verified JWT* (never the request body) and return **402** before calling Anthropic if the caller is not entitled. `MealScanAction` and `CustomFoodScanCard` showing a lock and routing to the paywall is what stops a paid tap becoming a confusing error — a modified client skips it and gets a 402, spending nothing. On the Dart side a 402 becomes `PremiumRequiredError` (`core/error/`, mapped in both scan datasources via `FunctionException.isPremiumRequired`) rather than a plain `VTError`, so the cubits emit their own `XPremiumRequired` presentation event and the page opens the paywall instead of toasting a failure — the safety net for a client whose local check went stale mid-session.

**`PremiumStatus.isActive` checks the expiry alongside the status**, because the row is only corrected when the store tells us: an active status with a past `expires_at` is a subscription that lapsed an hour ago. `SubscriptionStatus.entitles` counts `cancelled` as entitling — turning off auto-renew is not asking for the rest of the paid period back. The same two rules are duplicated in the Edge Function gate; if one moves, move the other.

**`PremiumCubit` is a plain `Cubit` provided once at the root** in `main.dart` alongside `AppCubit` (a `MultiBlocProvider`), not a `PresentationCubit` resolved per page: the entitlement gates affordances on several unrelated screens, and a factory would re-read it over the network on every page build and flicker the locks in. It refreshes from its own constructor (so there is no `onInit` to call), and **a failed read leaves the user free** — the safe direction, and deliberately with no error toast, since a startup complaint about a subscription most users do not have is pure noise. Consequence for tests: any widget test pumping a gated page must stand the provider up — `test/fixtures/premium_fixture.dart`'s `withTestPremium(...)` for a `MaterialApp`-based test, `registerTestPremiumCubit()` for one that pumps the real `VittaApp`. It stubs, so it is a fixture rather than a `CubitsFactories` entry, and it has to stub for the same reason `AppCubit` does — the cubit reads its use case from its own constructor.

**Buying goes through `PurchaseService`** (`core/services/purchases/`), the thin adapter over `purchases_flutter` — the `NotificationService`/`HealthService` boundary, so nothing above `core/services/` imports the SDK. It speaks the app's own `PremiumOffer` and `PurchaseOutcome` rather than RevenueCat's `Package` and `PurchaseResult`, and `PremiumCubit` drives it **directly rather than through a use case**, the way `AuthCubit` drives `ImagePickerService`: buying is a device/store interaction, not persistence. Three things encoded there:

- **The price is the store's localized string (`priceString`), never a number formatted here** — the storefront, its currency and its tax treatment are the store's to decide, and a price formatted in the app is wrong the moment someone buys from a country nobody thought about. `PremiumState.offer` is null until the store answers, and the paywall then says "unavailable" rather than naming a figure.
- **Cancelling is an outcome, not an error.** The store sheet reports a user backing out as a `PlatformException`; `PurchaseOutcome.cancelled` turns that into a return value, so a deliberate action never produces an error toast.
- **RevenueCat is identified at purchase time, not at sign-in.** `purchase`/`restore` take the Supabase `auth.uid()` and call `logIn` first. Identity only matters for attributing a purchase, so doing it here keeps `AuthCubit` out of the store entirely — and without it the purchase carries RevenueCat's anonymous id, which the webhook deliberately ignores, so the entitlement would never reach a `subscriptions` row. `AuthenticatedUser` gained an `id` for this.

**A successful purchase entitles optimistically.** `subscriptions` is only written when the webhook lands, which is a round trip away — waiting would leave the user staring at a locked feature they just paid for. So the cubit emits `active` at once and refreshes from the server behind it; if the server disagrees it wins, and the scans would fail server-side anyway. This is optimism about *timing*, not about entitlement.

**`REVENUECAT_API_KEY` is deliberately optional** (`Env.revenueCatApiKey` uses `maybeGet`, and `PurchaseService` no-ops without it), unlike the Supabase and Sentry keys which fail the build when empty. A build with no purchases configured still launches and works — everything except buying is unaffected — so a missing key degrades rather than crashing at startup.

**`revenuecat-webhook` is the only writer of `subscriptions`** (issue #155, runbook in [`docs/revenuecat-webhook-setup.md`](docs/revenuecat-webhook-setup.md)) — it maps RevenueCat's event types onto the four statuses and upserts the row with the service-role key. It's the first Edge Function **the app doesn't call**, so it has no `SupabaseFunction` enum case and must be deployed with **`verify_jwt` disabled** (RevenueCat holds no Supabase session); it authenticates on a shared secret compared in constant time instead. Three rules encoded there, each of which is a real failure and not a hypothetical: **an event we don't model still answers 200** (a non-200 is retried five times and then RevenueCat stops delivering *altogether*, so refusing unmodelled traffic eventually silences the integration); **`app_user_id` must be a UUID** (a purchase made before `logIn(uid)` carries RevenueCat's own `$RCAnonymousID:…`, which would violate the FK to `auth.users` and burn the retries); and **`subscriptions.last_event_at` drops any event not newer than the stored one**, since webhooks are retried and can arrive out of order — a redelivered `RENEWAL` landing after a `CANCELLATION` would otherwise silently re-entitle someone.

**RevenueCat's `CANCELLATION` is not our `cancelled`.** It fires both for "auto-renew turned off" and for a refund, told apart only by `cancel_reason` — and `CUSTOMER_SUPPORT` (the refund) maps to **`expired`**, not `cancelled`, because `SubscriptionStatus.entitles` counts `cancelled` as entitling until `expires_at` and a refunded user must lose access immediately rather than keep a period they were paid back for. Every other reason is genuinely `cancelled`.

`PaywallPage` (`AppRoute.premium`) hangs off `AuthCubit` because whether to show a purchase CTA or a sign-up CTA is a question about the user, and reads the root `PremiumCubit` for the entitlement itself. **Premium requires a real account**: a store subscription belongs to an Apple/Google account and must survive reinstall, which an anonymous per-device session cannot do — so an `AnonymousUser` gets a sign-up CTA, the same `is AuthenticatedUser` gate the delete-account tile uses. Until store products exist the purchase button is a disabled "coming soon" rather than a button that does nothing. `PremiumFeature` is the enum of what a subscription unlocks, with its labels/icon in `premium_feature_labels.dart` (the `nutrientLabel` pattern, keeping the domain Flutter-free) — adding a premium feature is one enum case. `VTColors.premium` and `VTPremiumBadge` are its own token and pill rather than reusing `macroCarbs`/`warning`, which are the same hexes as the `bodyRegion*` accents and would collide on the diet page.

## Deleting an account

The profile page's "Delete account" tile (issue #120) permanently deletes the signed-in user and all of their data. **The tile only shows for an `AuthenticatedUser`** — an anonymous session has no account to delete, so it's gated behind `if (state.user is AuthenticatedUser)`. **The delete runs through the `delete-account` Supabase Edge Function**, the same client-half-only split the nutrition scan uses (`SupabaseFunction.deleteAccount` → `SupabaseService.invoke` → `SupabaseAuthDataSource.deleteAccount`; the function's source lives only in the Supabase dashboard, runbook in [`docs/delete-account-setup.md`](docs/delete-account-setup.md)). **It has to be an Edge Function because deleting an `auth.users` row needs the service-role key** (`auth.admin.deleteUser`), which the app must never hold; the function reads the caller's id from the verified JWT (never from the request body, or any user could delete any account) so a user can only delete themselves.

**Deleting the auth user is the whole deletion** — every user-scoped table is `references auth.users(id) on delete cascade`, so the cascade wipes `food_logs`/`water_logs`/`sleep_logs`/`body_weight_logs`/`recipes`/`food_favorites`/`workouts`/`routines` (and their child rows) plus the user's own `foods`/`exercises` catalog rows in one go. There's no per-table delete to keep in sync, and the bulk-imported catalog (null `user_id`) is untouched. `DeleteAccountUseCase` follows the delete with a **fresh anonymous sign-in** (delete → `signOut` → `signInAnonymously`, the same "reset to a clean guest session" shape `SignOutUseCase` uses) so the app is left usable rather than on a dead session. `AuthCubit.deleteAccount` emits `AuthAccountDeleted` on success (a success toast), and `ProfilePage` gates the whole thing behind `DeleteAccountDialog` — a destructive-styled confirm whose copy states it can't be undone (a legitimate `showDialog`, like the goal editors, because it's a real choice, not an OK-only barrier). Storage objects (avatars, contributed catalog photos) are **not** cascade-deleted; catalog photos are deliberately kept since they back shared rows.

## Reminders

A personal reminders list (issue #118, the "todo" of the issue title — the **user-facing verbiage and every code identifier is "reminder"**, not todo): create a reminder for today or any date, tick it done, filter the day by all/completed/active, repeat it, and get a local notification near its time. `reminders` (user_id + title + due_date + optional `notes`/`remind_at`/`recurrence` + nullable `completed_at`) is private per user exactly like `water_logs`, behind `SupabaseReminderDataSource`/`ReminderRepository`. Same feature shape as water/sleep: entities mapped by hand via `Reminder.fromMap`, `ReminderCubit` (`PresentationCubit`) + `VTPage`, its own `AppRoute.reminders`/`ReminderRoute`, a home `VTFeatureTile`, l10n. The page is a `ReminderDateSelector` (prev/next/pick, the `DietDateSelector` shape) over a `VTSegmentedTabs` filter over a `ReminderTile` list; the FAB opens `showReminderFormSheet` (title, notes, date, a remind-me time toggle, and a `ReminderRecurrence` chip row) which also handles edit. The app-bar calendar action opens `ReminderHistoryPage` (`/reminders/history`) — the same history surface diet/water/sleep have: a `ReminderHistoryCubit` (its own use case `GetRemindersInRangeUseCase`, grouped into `ReminderHistoryState.remindersInMonth`) drives a `VTCalendarMonthGrid` that dots each day with reminders (green when all done, primary otherwise). Only days with reminders are tappable (`isDayEnabled`); tapping one sets `ReminderHistoryState.selectedDay` and its reminders render **inline under the calendar** in a read-only `ReminderDaySection` (reusing `ReminderTile` with its callbacks left null — the "pass no callback to disable" convention). `selectedDay` clears on a month change by emitting a fresh state (`copyWith` can't null a field back out). **`VTCalendarDayCell` gained a fix for this:** it used to gate taps on `isFuture || !hasValue`, but reminders live in the future — so it now gates on `hasValue` alone (and a future day with a value keeps its accent colour instead of dimming). The diet/water/sleep calendars are unaffected: a future day there never has a value, so it stays untappable and dimmed exactly as before.

**Recurrence is "next occurrence on complete", client-side (no server cron).** `ReminderRecurrence` (`none`/`daily`/`weekly`/`monthly`) pairs each case with its DB `wireValue` (`none` ⇄ null) the way `MealType` does, and owns `nextDate(from)` (monthly clamps a short month — Jan 31 → Feb 28). Every occurrence carries the recurrence, so `ReminderRepository.completeReminder` marks the row done and, only on the incomplete→complete transition of a recurring one, inserts the next occurrence (`Reminder.nextOccurrenceSchedule()` shifts `remind_at` by the same span) and returns a `ReminderCompletion { reminder, nextOccurrence? }` — sequential writes, the `SaveRecipeUseCase` shape. `completed_at` (nullable timestamptz) is the done flag; toggling back off just nulls it (it doesn't delete an already-spawned occurrence — a minor, accepted wart). `ReminderFilter` (all/completed/active) is a pure enum with a `matches(Reminder)` predicate held in cubit state, like `DietHistoryState.trendRange`; labels for both enums live in l10n via `presentation/pages/reminder/widgets/reminder_labels.dart` (the `nutrientLabel`/`dietModalityLabel` "keep the domain Flutter-free" pattern).

**Reminders fire a LOCAL notification, not a push — so this does not depend on #88.** A reminder's time is known on-device, so `flutter_local_notifications` schedules it directly; a remote push (#88) is only for server-initiated alerts. `NotificationService` (`core/services/notifications/`) is the thin vendor adapter — the `HealthService`/`ImagePickerService` boundary, so nothing above `core/services/` imports the package — exposing `init()` (timezone via `flutter_timezone` + plugin init + the Android channel), `requestPermission()`, `scheduleReminder({id,title,body,dateTime})` (`zonedSchedule`), and `cancel`/`cancelAll`. It's a lazy singleton `init()`-ed in `bootstrap()` before `appRunner`. **`ReminderCubit` is what coordinates it** (schedule on create/update, cancel on complete/delete, schedule the spawned occurrence) — the one device service a cubit drives rather than a datasource, exactly like `AuthCubit`↔`ImagePickerService`; the domain stays notification-free. The notification id is `Reminder.notificationId` (`id.hashCode & 0x7fffffff`, stable so scheduling and cancelling hit the same slot), and its title/body are the reminder's own `title`/`notes`, so no l10n has to reach into the cubit. Platform config lives in `AndroidManifest.xml` (POST_NOTIFICATIONS + exact-alarm perms + the two plugin receivers), `android/app/build.gradle.kts` (core-library desugaring, required by `zonedSchedule`), and `AppDelegate.swift` (the `UNUserNotificationCenter` delegate). Like `SupabaseService`, `NotificationService` is only ever **mocked** in tests — platform channels can't run under `flutter test`, so real delivery is verified on a device.

## App-level state

The user's cross-cutting preferences — locale override (`AppSettings.locale`, null = follow system), `themeMode`, and `unitSystem` (`core/units/unit_system.dart`'s `UnitSystem.metric`/`.imperial`, default metric) — live in `AppSettings` (`lib/app/domain/settings/entities/app_settings.dart`), a domain entity, not a presentation-only type. `SettingsRepository` (`lib/app/data/settings/settings_repository.dart`) wraps `SettingsLocalDataSource` (see Local storage) the same way every other feature repository wraps its datasource; `GetAppSettingsUseCase`/`SaveAppSettingsUseCase` (`lib/app/domain/settings/use_cases/`) are the only way anything reads or writes it. `AppCubit` (`lib/app/cubit/app_cubit.dart`) is `PresentationCubit<AppSettings, AppPresentationEvent>` — its Bloc state *is* the domain entity directly (no separate wrapper), constructed via `GetAppSettingsUseCase` and persisted via `SaveAppSettingsUseCase` on every change. It's a GetIt singleton provided once at the root in `main.dart` via `BlocProvider.value(value: G<AppCubit>(), ...)` wrapping `MaterialApp.router`. `SettingsPage` is its only UI consumer, changing locale/theme/units through `RadioGroup`s. **Any other cubit that only needs to *read* settings (not display/edit them) depends on `GetAppSettingsUseCase` directly instead of reading `AppCubit`** — e.g. `WaterCubit.unitSystem` and `FoodSearchCubit.unitSystem` are plain getters calling `_getAppSettingsUseCase().unitSystem`, and `WaterPage`/`LogFoodSheet` read `cubit.unitSystem` off the cubit they already depend on rather than also reaching for `context.read<AppCubit>()`. This keeps `AppCubit` consumption limited to the one page that's actually about editing settings.

`UnitSystem` has two independent conversion extensions in the same file: `WeightConversion` (grams/ounces — food quantity consumed, `LogFoodSheet`) and `VolumeConversion` (milliliters/fluid ounces — water intake, `WaterPage`/`AddWaterSheet`/`EditWaterGoalDialog`). Both convert to their metric base unit before it ever reaches the domain (`FoodLog.quantityGrams`, `WaterLog.amountMl`) — the base unit stays the source of truth end-to-end, same principle as storing locale/theme as their real types rather than display strings. `WeightConversion` deliberately does not touch a food's per-100g macros (`CustomFoodPage`, `dietNutritionPer100gTitle`, ...) — "per 100g" is a nutrition-label convention independent of the reader's preferred unit system, not a quantity to convert.

## Internationalization

**Every `plural` message needs an explicit `=0` case.** Without one, `Intl.pluralLogic` falls through to the locale's CLDR rule — and Portuguese classifies zero as the **`one`** category (`_pt_rule`: `if (_i == 0 || _i == 1) return ONE`). So `{n, plural, =1{1 série feita} other{{n} séries feitas}}` renders **"1 série feita" for zero**: not a grammar wobble, a wrong number. It's invisible in English, where 0 maps to `other` and reads correctly — so it survives review and ships. `test/app/l10n/plural_zero_test.dart` pins this for both locales.

Standard Flutter `gen-l10n` (not a custom solution). ARB files live in `lib/l10n/arb/` (`app_en.arb` is the template, plus `app_pt.arb`). Never hardcode user-facing strings — add a key to both ARB files and read it via `context.l10n` (`BuildContextLocalizationExt`, `core/localization/localization_extensions.dart`), not `AppLocalizations.of(context)` directly — the same file also has `context.materialLocalizations` for the rare case that needs Flutter's own localized strings (date/time formatting) instead of the app's. Run `flutter gen-l10n` (or `flutter pub get`, which triggers it) after editing an ARB file. Add more locales the same way dofus_buddy added `fr` — a new ARB file plus the corresponding `RadioListTile` in `SettingsPage`.

## Code style

- No comments. Code must be self-explanatory through naming; if it isn't, restructure it rather than annotate it.
- No variables named `result` (or similarly generic — `data`, `response`, `value` as a local var name). Name it after what it actually holds. A local variable holding a not-yet-unwrapped `Result<F, S>` gets a `Result` **suffix** instead — `dailyMacrosResult`, `loggedResult`, `deletedResult` — not a bare/generic name.
- `Result<F, S>` (`lib/app/core/error/result.dart`) is consumed via `.when(failure, success)`, not a `switch`/pattern match — `T when<T>(T Function(F error) failure, T Function(S value) success)`, positional (not named) with `failure` always first: `dailyMacrosResult.when((error) => emit(XError(message: error.message)), (value) => emit(XLoaded(...)))`. When one branch is synchronous and the other returns a `Future` (e.g. failure just calls `emit(...)`, success awaits a reload), wrap the sync branch to match rather than mixing return types — `Future.sync(() => emit(XError(...)))` for a `void`-returning cubit method, `Future.value(Failure(error))` for one returning `Result` itself — see `WaterCubit.deleteLog()`/`LogFoodUseCase.call()`. Prefer this over `Future.sync` when the two branches would otherwise need unrelated wrapping just to satisfy `.when()`'s single return type: extract the value both branches actually agree on instead (e.g. `final error = result.when((error) => error, (_) => null);` then branch with a plain `if` outside `.when()`) — see `DietCubit.deleteLog()`.
- One widget per file. Never declare a private `_Widget` helper class alongside another widget in the same file — extract it to its own file under the feature's `widgets/` folder as a public (non-underscore) widget, and import it. A page file holds only its page widget; each supporting piece (`ProfileHeader`, `ProfileMenuTile`, `MacroPill`, ...) is its own file. The `show...` launcher function for a modal/sheet stays in the same file as the widget it launches (`showFoodDetailsDialog` with `FoodDetailsDialog`) — that's a function, not a second widget.
- Use the dot-shorthand operator wherever the target type is inferable (`.center`, `.w700`, `.light`).
- `flutter analyze` must report zero issues (including `info`-level lints) before considering a change done.
- Lint rules are the explicit list in `analysis_options.yaml`, carried over from the dofus_buddy project.

## Testing

Mirrors `lib/`. No `late` variables, ever — no `setUp`-populated shared state either. Every test builds its own instances inline, at the top of the `test`/`build:` body that uses them, via the factories/fixtures below. This is stricter than "avoid `late`": a `final` declared in `setUp()` and closed over by every test in the file is the same shared-mutable-state problem `late` causes, just spelled differently.

- `test/mocks/`: one file per layer (`repositories_mocks.dart`, `use_cases_mocks.dart`, `datasources_mocks.dart`, `services_mocks.dart`, ...), each a one-line `class MockX extends Mock implements X {}`.
- `test/factories/`: one `abstract class XFactory`/`XFactories` per file, `static` build methods only — never instantiate the class, and it reads better in autocomplete (`FoodFactory.build(...)`, `CubitsFactories.buildDietCubit(...)`) than a pile of same-named top-level functions. **No stubbing inside a factory, ever** — a factory either takes an already-configured instance or defaults to a bare, unstubbed `MockX()`; all `when()`/`verify()` setup happens in the test that needs it. Entity factories in `test/factories/entities/` (`FoodFactory`, `FoodLogFactory`, ...); use case factories in `UseCasesFactories` (`test/factories/use_cases_factories.dart`); cubit factories in `CubitsFactories` (`test/factories/cubits_factories.dart`) — only add a cubit factory once the cubit takes constructor dependencies worth defaulting. Repositories are normally thin pass-throughs not worth a dedicated test, but one with real branching logic of its own (`DietRepository.searchFoods`'s catalog-then-Open-Food-Facts fallback) gets tested the same way a cubit does — by mocking its own dependencies (here, the datasources) rather than the repository itself; `RepositoriesFactories` (`test/factories/repositories_factories.dart`) is that shape's factory file, following the same defaults-to-a-bare-mock convention. Note the consequence: if a cubit's constructor calls a mocked dependency synchronously (`AppCubit` calls `getThemeMode()` at construction), the bare-default path isn't usable as-is — the test must always pass a pre-stubbed mock, same as every other case.
- `test/fixtures/`: builders that need real I/O a mock can't stand in for (e.g. `local_storage_fixture.dart`'s `openTestHiveBox()`/`buildTestLocalStorageService()`, which open a real Hive box against a temp directory). Register cleanup with `addTearDown()` *inside* the fixture function itself — it hooks into whichever test is currently running, so callers never juggle a paired teardown.
- Cubits: `bloc_test`'s `blocTest`, building every mock and the cubit inside `build:` (not a separate `setUp:` — that would need `late` to share the mock between the two callbacks). If a mock must also be reached from `verify:`, declare it as a `final` local right above that one `blocTest(...)` call, not hoisted to the file/group level. A `PresentationCubit`'s state-transition tests stay in `blocTest` as usual (its `expect:` no longer includes a loading state, since there isn't one); its loading events get their own `blocPresentationTest` (from `bloc_presentation_test`, same file, same `build:` shape) asserting `expectPresentation: () => [isA<XShowLoading>(), isA<XHideLoading>()]` for that cubit's own presentation event type — see `diet_cubit_test.dart`/`water_cubit_test.dart`/`food_search_cubit_test.dart`.

## Releases (TestFlight for iOS, Firebase App Distribution for Android)

`git tag v1.2.3 && git push origin v1.2.3` is the whole release ritual (issue #81, extended for Android in #69). The **same tag** fires two independent workflows: `.github/workflows/testflight.yml` (iOS) and `.github/workflows/firebase-android.yml` (Android). Each platform uses the tool that's simplest *for it* — TestFlight needs no per-device registration (any Apple ID installs), while Firebase App Distribution is a two-step build-and-upload for Android but would need ad-hoc signing + tester UDID registration on iOS, so iOS deliberately stays on TestFlight and only Android went to Firebase. Both read the version from the tag the same way.

`.github/workflows/testflight.yml` runs on `macos-15` and hands off to `ios/fastlane`'s `beta` lane — all the iOS build logic lives in the Fastfile, the workflow only supplies secrets and the tag.

**The tag is the source of truth for the version**, not `pubspec.yaml`: the lane parses `v1.2.3` → `1.2.3` and rejects anything else outright, so a typo'd tag fails in seconds instead of uploading a mislabelled build. pubspec's `version:` stays a default for local builds and is deliberately never read by the lane. The build number comes from `latest_testflight_build_number + 1` — asking TestFlight is the only source that can't collide with a build already up there, whichever tag is being cut.

The version reaches the binary through Flutter, not Xcode: `flutter build ios --release --no-codesign --build-name/--build-number` writes `ios/Flutter/Generated.xcconfig`, which `Info.plist`'s `$(FLUTTER_BUILD_NAME)`/`$(FLUTTER_BUILD_NUMBER)` read. That's why the lane shells out to Flutter first and lets `build_app` do the archiving — the archive is produced once, by fastlane.

**Signing is `match`** (certs/profiles in a separate private git repo, encrypted), and CI runs it `readonly: true` — CI must never mint a certificate, since that silently invalidates the existing one. The checked-in Xcode project keeps `CODE_SIGN_STYLE = Automatic` so local `flutter run` works untouched; the lane flips it to manual with the match profile just before archiving and leaves the checkout dirty, which is free on a throwaway CI runner.

`.env` is a **bundled asset** (see `pubspec.yaml`) and gitignored, so the workflow writes it from secrets — a build without it ships an app that can't reach Supabase. A missing secret reads as an empty string rather than an error and `Env` would only blow up once a tester launches the build, so the step fails fast on an empty one instead.

`ITSAppUsesNonExemptEncryption = false` in `Info.plist` matters more than it looks: without it every upload stops in TestFlight waiting for the export compliance form to be answered by hand before testers can install. Vitta only talks HTTPS, which is exempt.

**One-time setup** (Apple app record, the private `match` certificates repo, the App Store Connect API key, and the nine GitHub secrets) is a runbook in [`docs/testflight-setup.md`](docs/testflight-setup.md) — the pipeline fails until every step there is done, and none of it can live in the repo.

The **Android** pipeline (`firebase-android.yml`, issue #69) runs on `ubuntu-latest` and is deliberately *not* fastlane — Android + Firebase is `flutter build apk` then one `firebase appdistribution:distribute` (Firebase CLI), so wrapping it in a Ruby/Gemfile/plugin toolchain the way the iOS lane needs would be ceremony, not simplicity. It mirrors the iOS lane's decisions where they apply: the tag is the source of truth for `--build-name` (rejected in-shell if it isn't `vX.Y.Z`), `.env` is written from the same secrets with the same fail-fast-on-empty guard, and it distributes to the Firebase tester group aliased **`testers`**. `--build-number` (the Android `versionCode`) is `github.run_number` rather than TestFlight's `latest_build_number + 1`, since Firebase has no equivalent "ask the store" call — a run number is monotonic and enough for in-place updates.

**Android signing is a committed debug keystore, deliberately not a release key.** `android/app/debug.keystore` is checked into the repo (well-known public `android` password — nothing secret) and `android/app/build.gradle.kts` points the debug signing config at it, with the `release` build type reusing that same config. This is the whole reason there are no signing secrets: Firebase can only update a tester's install in place when consecutive builds share a signature, and a CI runner otherwise auto-generates a fresh debug keystore per run whose changing signature makes Android reject the update (a reinstall every drop). A committed keystore gives every build — local `flutter run --release` and CI — one stable signature with zero setup. The tradeoff accepted: it's not a Play Store upload key, so a real release key has to be generated if a Play pipeline is ever added. A release-keystore-from-secrets approach was built first and dropped as unnecessary ceremony for beta distribution.

**One-time setup** is just the Firebase side (project + Android app registration, a service-account JSON with App Distribution Admin, and two new GitHub secrets — `FIREBASE_ANDROID_APP_ID` and `FIREBASE_SERVICE_ACCOUNT`; the three Supabase/Sentry ones are reused from TestFlight), a runbook in [`docs/firebase-distribution-setup.md`](docs/firebase-distribution-setup.md). Same `com.rodrigobastosv.vitta` id as iOS (`namespace` + `applicationId` in `android/app/build.gradle.kts`, plus the `MainActivity.kt` package path). There is still no Google **Play** pipeline — Firebase App Distribution is beta delivery to invited testers, not a store listing.
