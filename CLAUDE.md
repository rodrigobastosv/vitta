# Vitta

Lifestyle companion app: diet and workout tracking. Flutter, SDK ^3.10.7.

## Backend

Supabase (Postgres). Chosen over Firebase because the diet domain is inherently relational — a day's macro totals are a `SUM()`/`GROUP BY` over `food_logs` joined to `foods`, which Postgres does natively; Firestore would push that aggregation into the client. `supabase_flutter` talks to it directly from the app (no custom backend server).

- `supabase/schema.sql`: source of truth for the `foods` and `food_logs` tables, run manually in the Supabase SQL editor (no migration tooling yet). Both tables have row-level security scoped to `auth.uid()`, so each user only ever sees their own rows.
- Auth is anonymous (`signInAnonymously`, wired in `lib/app/bootstrap.dart`) so RLS has a stable `user_id` without building a login UI yet. Swapping in real accounts later means adding email/OAuth sign-in on top of the same anonymous-session-per-device bootstrap — existing anonymous data can be linked via Supabase's `linkIdentity`.
- `main()` only calls `bootstrap(appRunner: () => runApp(const VittaApp()))` — infrastructure bootstrap (dotenv, Sentry init, Supabase init, anonymous sign-in, `setupDependencies()`) is deliberately kept out of `AppCubit`/widget code so it stays a single, synchronously-ordered sequence and `AppCubit` stays pure presentation state. `appRunner` is threaded through rather than `main()` calling `bootstrap()` then `runApp()` as two separate statements, so every init step (not just the widget tree) runs inside Sentry's zone — see Error tracking below.
- Credentials live in a gitignored `.env` (`SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEY`, `SENTRY_DSN`), loaded via `flutter_dotenv` and read through `lib/app/core/env/env.dart`. Copy `.env.example` to `.env` and fill in a real project's values — the placeholder committed values won't connect to anything.

## Error tracking

`sentry_flutter`, initialized inside `bootstrap()` (`lib/app/bootstrap.dart`) via `SentryFlutter.init((options) => options.dsn = Env.sentryDsn, appRunner: ...)` — the package's own `runZonedGuarded`/`FlutterError.onError` wiring is what actually captures uncaught exceptions, so nothing else in the app needs to touch `runZonedGuarded` directly. `appRunner` wraps the rest of bootstrap (Supabase init, Hive init, `setupDependencies()`) and finally calls the `appRunner` passed in from `main()`, so a startup crash in any of those steps is captured too, not just runtime errors after the widget tree is up.

**Errors surfaced through `Result`/`VTError` (network failures, validation, ...) are not reported to Sentry.** Those are expected, already-handled failure paths shown to the user via `VTErrorDialog` — forwarding every one to Sentry would mostly report noise (e.g. a user opening Diet with no internet connection). Sentry is scoped to genuinely unexpected crashes only. If a specific `VTError` case turns out to be worth tracking, capture it explicitly at that call site (`Sentry.captureException(error.cause)`) rather than wiring every `Failure` generically.

## Logging

Navigation and user-action logging goes through one centralized `LoggingService` (`core/services/logging/`, see issue #46) so a new log sink is added in exactly one place. The service holds a `List<LogDestination>` and fans every `LogEntry` (a `category` like `'navigation'`/`'action'`, a `message`, a `LogLevel`, and a `data` map) out to all of them — currently `ConsoleLogDestination` (`debugPrint`, gated on `kDebugMode`, so log lines show in the `flutter run` **terminal** — `dart:developer`'s `log` was tried first but only surfaces in DevTools/the IDE console, not the terminal) and `SentryLogDestination` (a Sentry **breadcrumb**, so the last actions/navigations ride along on any crash report — this is the one place breadcrumbs are added, complementing the crash-only `Sentry.captureException` above). Adding a remote/analytics sink later means one new `LogDestination` implementation added to the list, nothing else. `LogDestination` is an abstract class — the one other deliberate interface besides the vendor adapters, because "send logs to different places" is a real polymorphism point.

**Call sites never see `LoggingService` — they use the static `Log` facade** (`core/services/logging/log.dart`): `Log.action('<snake_case_event>', data: {...})` and `Log.navigation(...)`, so nothing has to inject a logger through its constructor. `Log` holds one swappable `LoggingService` (defaulting to a no-op with zero destinations, so code that logs is safe in unit tests that don't care); `setupDependencies` wires the real one once via `Log.service = LoggingService(destinations: const [...])`. It's the one sanctioned global-static in the app — deliberately not a GetIt registration, since a logger is a pure cross-cutting sink with no swap boundary worth a `G<>()` lookup.

**Navigation is logged automatically** by `LoggingNavigatorObserver` (a `NavigatorObserver` wired into `GoRouter(observers: ...)` in `app_router.dart`), which calls `Log.navigation(action: 'push'|'pop'|'replace', route: route.settings.name)` on every route change — no per-page code. **User actions are logged explicitly** from the cubit that performs them, calling `Log.action('<snake_case_event>', data: {...})` on the success path — the write actions across the app are instrumented this way: `AuthCubit` (`sign_in`/`sign_up`/`sign_out`), `FoodSearchCubit` (`food_logged`), `DietCubit` (`food_log_deleted`/`food_log_updated`), `WaterCubit` (`water_logged`/`water_log_deleted`/`water_goal_changed`), `SleepCubit` (`sleep_logged`/`sleep_log_deleted`), `MacroGoalsCubit` (`macro_goals_saved`), `OnboardingCubit` (`onboarding_completed`), and `AppCubit` (`theme_changed`/`locale_changed`/`unit_system_changed`). Instrument a new notable action the same way — event name in `snake_case`, on the success branch, never from the widget layer. Tests that assert a log fired swap in a mock with `useMockLog()` (`test/fixtures/logging_fixture.dart`, which resets `Log.service` on teardown) and `verify` on it; tests that don't care need no setup because the default `Log.service` is a harmless no-op.

## Food data

`foods` is a catalog shared across every user, not a per-user table — RLS allows any authenticated user to read every row (see `Anyone can read foods` in `supabase/schema.sql`), while insert/update/delete stay scoped to the row's own `user_id`. `DietRepository.searchFoods` tries the shared catalog first (`SupabaseDietDataSource.searchCatalog`, an `ilike` on `name`) and falls back to the [Open Food Facts](https://world.openfoodfacts.org) public API (`OpenFoodFactsDataSource`, `lib/app/data/diet/datasources/http/`) whenever the catalog is **sparse**, not only when it's empty: below `_sparseCatalogThreshold` (5) catalog hits it also queries OFF and merges the two, deduping by `barcode` (catalog rows first since they're popularity-ranked, then OFF rows not already present) — a Portuguese term the OFF *import* barely covers (its country filter is US-heavy) still gets augmented by OFF's live search, which spans the whole OFF database. An empty catalog returns the OFF result directly (including its error); a non-empty-but-sparse catalog whose OFF augmentation fails still returns the catalog hits rather than erroring — there's still no proprietary/mirrored copy of the full OFF export (several GB, would dwarf Supabase's free-tier storage), so the catalog only grows organically from what real users actually log. A search result (from either source) is only persisted (into the shared `foods` table) once a user actually logs it; `LogFoodUseCase` does the save-then-log in one step for any `Food` without an `id`, and `SupabaseDietDataSource.saveFood` looks up an existing row by `barcode` first so the same product logged by different users reuses one row instead of duplicating it (see the unique partial index on `barcode`). Users can also add a fully custom food (name + optional brand + macros per 100g, optionally a photo) through the same flow — see the Custom food page below; custom foods become part of the shared catalog immediately, with no moderation step. A photo uploads to the `food-images` Supabase Storage bucket (public read, authenticated write — see `supabase/schema.sql`) via `SupabaseService.storage(bucket)`, the one place a datasource touches Storage directly instead of `.from(table)`; the resulting public URL is what gets stored in `foods.image_url` and rendered by the design system's `VTFoodImage` (falls back to a placeholder icon when there's no image or it fails to load).

Catalog search ranks by **popularity** (see issue #56): `searchCatalog` orders its `ilike` matches by `foods.times_logged desc` (then `name` as a stable tiebreak), so the most-logged products surface first. `times_logged` is a denormalized counter on the shared `foods` row — it has to live there because `food_logs`' RLS scopes each user to their own rows, so "how many times *anyone* logged this" can't be aggregated client-side. A `security definer` trigger on `food_logs` (`bump_food_times_logged` in `supabase/schema.sql`) keeps it in sync (+1 on insert, −1 on delete, floored at 0) — deliberately not on update, since editing a log's quantity or meal (`SupabaseDietDataSource.updateFoodLog`) leaves it the same one log, and the food it points at can't be swapped; it's `security definer` because the logging user doesn't own the (often imported, unowned) food row, so `foods`' update policy would otherwise block the bump. The signal is in-app logging only — deliberately *not* seeded from OFF's `unique_scans_n` — so it starts at 0 for the bulk-imported catalog and grows purely with real usage.

Macros (calories/protein/carbs/fat/fiber) are dedicated `numeric` columns since they're mandatory and used in checks, but **vitamins and minerals are a keyed `jsonb` blob (`foods.micronutrients`), not one column per nutrient** — there are dozens, coverage is sparse (most OFF rows only carry macros), and a new nutrient shouldn't need a schema migration. Each is a case on the `Nutrient` enum (`lib/app/domain/diet/entities/nutrient.dart`) pairing its `wireKey` (the JSONB/DB key), its `offKey` (the OFF `nutriments` key its per-100g value is read from — like `SupabaseTable`/`MealType` pair a case with its wire value), and a `NutrientUnit` (mg/µg) that only governs display. Everything is stored per-100g **in grams**, the same base unit as macros (`NutrientUnit.fromGrams` converts for display); `Food.micronutrientsPer100g` is a `Map<Nutrient, double>` parsed from the blob (unknown keys ignored, so the app tolerates nutrients it doesn't model yet), `FoodLogEntry`/`MacroTotals` scale and sum it exactly like a macro. `MacroSummaryCard` has two versions (see issue #54): it defaults to the compact macros-only view and, only when some micronutrient is actually present (`> 0`), shows an `unfold_more`/`unfold_less` icon button to the right of the calorie total that toggles the complex version — expanding a "Vitamins & minerals" section that renders only the present nutrients, which keeps the sparse data from showing as a wall of blanks. Per-nutrient **goals are deliberately not modelled yet** — `MacroGoals` still covers only calories + the four macros. Custom foods (`CustomFoodPage`) capture macros only; their micronutrient map is empty until OFF data fills it. Add a nutrient by adding an enum case plus its `nutrientLabel` l10n key — no new column, request field, or parse branch.

The diet calendar colors each logged day's dot by how close that day's calories are to the goal (see issue #51), not just "has a log". `DailyMacros.adherenceTo(MacroGoals)` returns a `GoalAdherence` (`met`/`close`/`off`) — it ratios the day's `totalCalories` against `calorieGoal` and classifies it via `GoalAdherence.forRatio` (green band ±10%, yellow band ±25%, else red — the bands are the tunable constants). **Adherence is calories-only by design** — the four macros are shown on `MacroSummaryCard` but don't factor into the day's color (a `combineWorst` fold across all macros was tried and dropped; per-macro goals aren't a strong enough signal to gate the color on). `GoalAdherence.color` maps `met`/`close`/`off` to `VTColors.green`/`warning`/`error` (used by both the calendar dots and `MacroSummaryCard`'s calorie ring — the ring falls back to `colorScheme.primary` on an empty day so a day with no log isn't painted red). This is why the calendar loads full per-day macros for the month: `DietCubit.loadMonthMacros` → `GetMacrosInRangeUseCase` → `DietRepository.getMacrosInRange` (groups `SupabaseDietDataSource.getLogsInRange`'s joined rows into a `Map<DateTime, DailyMacros>`). That chain is named for the range it takes (`from`/`to`), not for a month — the history page (see below) reuses it for a rolling 7/30/90-day window, so anything month-shaped is the caller's choice of arguments.

## Architecture

Clean Architecture, without repository/datasource interfaces and without a separate data-model layer — one concrete class per responsibility, no `Impl` suffixes:

```
lib/app/
  core/            DI (get_it), navigation extensions, error (Result/VTError), http, env, services/
  data/<feature>/   Repository (concrete class), datasources/
  domain/<feature>/ Entities, use cases (plain classes with a `call` method)
  design_system/    VT-prefixed tokens, themes and components
  presentation/     Pages (Cubit + VTPage), routing
```

`data/diet/` and `domain/diet/` are the first feature built out this way, composing two datasources behind one `DietRepository`: `SupabaseDietDataSource` (persistence) and `OpenFoodFactsDataSource` (remote search, via `core/http`'s `VTHttpClient`). Entities double as the wire format for Supabase rows via a `fromMap(Map<String, dynamic> row)` factory constructor on the entity itself (`Food.fromMap`, `FoodLog.fromMap`, ... — no `@JsonSerializable`/`build_runner`, Supabase's client already returns decoded maps, so generated (de)serialization would just add ceremony); the datasource calls `EntityName.fromMap(row)` and otherwise just orchestrates the Supabase/HTTP call, with no parsing logic of its own. `data/water/`/`domain/water/` follow the same shape with a single `SupabaseWaterDataSource`/`WaterRepository` — no remote catalog to search, so it's simpler: log an amount directly, no save-then-log step. Follow this same shape for the next feature (workout): one repository per feature composing whatever datasources it needs, entities mapped by hand via their own `fromMap`.

A feature's `datasources/` folder is flat (single file, e.g. `data/auth/datasources/supabase_auth_datasource.dart`) when there's only one datasource. Once a feature composes more than one kind, split into one subfolder per kind — `supabase/`, `http/`, `local/` — so the datasource type is visible from the path: `data/diet/datasources/{supabase/supabase_diet_datasource.dart, http/open_food_facts_datasource.dart}`, `data/water/datasources/{supabase/supabase_water_datasource.dart, local/water_local_datasource.dart}`. A vendor-specific `requests/` folder (wire-format request bodies, e.g. `CreateFoodRequest`) nests under that vendor's subfolder (`data/diet/datasources/supabase/requests/`), not directly under `datasources/`. This doesn't apply to a local-only feature with no other datasource kind (`OnboardingLocalDataSource`, `SettingsLocalDataSource`) — those keep the flatter `data/<feature>/<feature>_local_datasource.dart` shape from the Local storage section below, since there's nothing to disambiguate from.

A domain use case depends directly on the concrete repository class from `data/`, e.g.:

```dart
class LogFoodUseCase {
  LogFoodUseCase({required DietRepository dietRepository}) : _dietRepository = dietRepository;
  final DietRepository _dietRepository;
  Future<Result<VTError, FoodLog>> call({required Food food, ...}) => _dietRepository.logFood(...);
}
```

One deliberate exception to "no interfaces": the thin adapter that directly wraps a third-party SDK gets its own concrete class under `core/services/<vendor>/`, so datasources depend on it rather than on the vendor's client type directly (`core/services/storage/local_storage_service.dart`'s `LocalStorageService`; `core/services/supabase/supabase_service.dart`'s `SupabaseService`). Repositories/datasources/use cases above that boundary stay concrete-only as usual. `core/services/image_picker/`'s `ImagePickerService` is the one such adapter a **cubit** depends on rather than a datasource (picking a photo is a device interaction, not persistence) — it returns a plain `PickedImage` (path + bytes + extension) and takes the app's own `ImagePickerSource` enum, which pairs its case with `image_picker`'s `ImageSource` the same way `SupabaseTable` pairs a case with its wire name, so nothing above `core/services/` imports `package:image_picker`. Choosing camera-vs-gallery stays UI (`showImageSourceSheet`); the cubit only receives the chosen source.

**A Page/widget never resolves a datasource, repository, or use case itself** — no `G<XLocalDataSource>()`, `G<XRepository>()`, or `G<XUseCase>()` inside a `build()` method or a callback. The only thing a widget resolves via `G<...>()` is its own Cubit (in a `BlocProvider`'s `create:`), and the only thing it calls on that Cubit is one of its methods. The Cubit is what depends on the use case(s), which depend on the repository, which depends on the datasource(s) — every hop stays in that order, with nothing skipped. `domain/onboarding/`'s `OnboardingPage → OnboardingCubit → CompleteOnboardingUseCase → OnboardingRepository → OnboardingLocalDataSource` is the reference shape, including for a feature with no remote datasource to justify a repository: it's still a thin single-datasource wrapper, kept because nothing above `data/` is allowed to reach past the repository boundary, not because it composes anything. The one deliberate exception is routing-level infrastructure — `HomeRoute`'s redirect (the onboarding gate, see Navigation below) depends on `HasSeenOnboardingUseCase` directly via `G<>()`, the same way `bootstrap.dart` calls services directly; neither is a Page, both sit outside the widget tree as wiring. (`WaterCubit` predates this rule and still depends on `WaterLocalDataSource` directly for its daily-goal read/write, with no use case/repository in between — don't copy that shape for a new feature; `OnboardingCubit` is the one to follow.)

## Dependency injection

`GetIt.instance` is aliased as `G` (`lib/app/core/di/dependencies.dart`). Register dependencies in `setupDependencies()`, called once from `bootstrap()`. Resolve with `G<Type>()`.

## Local storage

`hive_ce` (+ `hive_ce_flutter` for `Hive.initFlutter()`) for anything that must survive an app restart but doesn't belong in Supabase (device-local preferences, not user data), sitting behind `core/services/storage/local_storage_service.dart`'s `LocalStorageService` (`get<T>`/`put<T>`/`delete`, backed by a Hive `Box<dynamic>`) — datasources never touch `package:hive_ce` directly, only `LocalStorageService`'s constructor does. One shared `app` `Box<dynamic>`, opened once in `bootstrap()` and wrapped in a single `LocalStorageService` registered in DI (opening a box is async, DI registration isn't) — not one box/service per feature. Each feature still gets its own local datasource class (concrete, no interface, same shape as a feature's Supabase datasource) depending on `LocalStorageService`; keys are prefixed with the feature name (e.g. `SettingsLocalDataSource`'s `settings.locale`, `settings.themeMode` in `data/settings/settings_local_datasource.dart`) so datasources can't collide with each other in the shared box. Primitives only (`String`, `int`, `bool`, ...) — no `TypeAdapter`/`build_runner`. `SettingsLocalDataSource` sits behind `SettingsRepository`/`GetAppSettingsUseCase`/`SaveAppSettingsUseCase` (see App-level state below) — nothing above `data/` touches it directly. A new feature that needs local-only storage gets its own `data/<feature>/<feature>_local_datasource.dart` depending on `LocalStorageService`, with its own key prefix — don't add unrelated state to `SettingsLocalDataSource`. Route it through a repository and use case(s) per the Architecture section's layering rule (`data/onboarding/`'s `OnboardingLocalDataSource` behind `OnboardingRepository` is the current example) rather than a Cubit depending on the local datasource directly.

Supabase gets the same treatment: `core/services/supabase/supabase_service.dart`'s `SupabaseService` wraps `SupabaseClient`, exposing `auth`, `currentUserId`, `hasSession`, and `from(table)` — datasources depend on `SupabaseService`, never on `package:supabase_flutter`'s `SupabaseClient` directly (`SupabaseDietDataSource` is the example). This isn't a full abstraction of Postgrest's fluent query builder (that's not a real swap boundary and would just add ceremony) — it centralizes construction and the one piece of logic every datasource needed (`currentUserId`), consistent with `LocalStorageService`.

Table and Storage-bucket names are never raw strings: `from(...)` takes a `SupabaseTable` enum and `storage(...)` takes a `SupabaseBucket` enum (`core/services/supabase/supabase_table.dart`, `supabase_bucket.dart`), each pairing its Dart case with the DB `wireName` the same way `AppRoute` pairs a name with its `path` (`foods('foods')`, `foodLogs('food_logs')`, `foodImages('food-images')`, ...). A new table/bucket means adding an enum case, not hardcoding its name at the call site — the signatures make a raw string impossible to pass. Column names and embedded-resource selects stay plain strings, except where a select re-references a table name it reads it off the enum too (`getDailyLog`'s `'*, ${SupabaseTable.foods.wireName}(*)'`).

Tests exercise `LocalStorageService` against a real Hive box in a temp directory (`Hive.init(tempDir.path)`, no platform channels needed) rather than mocking it — see `test/app/core/services/storage/local_storage_service_test.dart`, `test/app/data/settings/settings_local_datasource_test.dart`, and `test/widget_test.dart` (all via the `test/fixtures/local_storage_fixture.dart` fixture). `SupabaseService` is mocked instead (`test/mocks/services_mocks.dart`) since standing up a real Supabase client in tests isn't practical.

## State management

`flutter_bloc` Cubits. A Cubit that loads data extends `PresentationCubit<S, P>` (`lib/app/presentation/general/presentation_cubit.dart`) instead of `Cubit<S>` directly — it mixes in `bloc_presentation`'s `BlocPresentationMixin<S, P>`, giving it `emitPresentation(...)` for one-off side effects that aren't part of state. `P` is **not shared globally** — every cubit gets its own `sealed class XPresentationEvent` next to its state/cubit files (`diet_presentation_event.dart`, `water_presentation_event.dart`, ...), so a cubit is free to grow presentation events beyond loading (a snackbar, a one-off navigation) without touching anyone else's type. Pages that own such a cubit use `VTPage<C, S, P>` (`lib/app/presentation/general/vt_page.dart`), which does three things at once: resolves the cubit via `G<C>()` (a page never calls `G<C>()` itself, and never passes a `create:` callback — `VTPage` is the one place that does DI resolution for pages), wires `BlocBuilder` (handing cubit + state to `builder:`, so pages never call `context.read` for their own cubit), and wires `BlocPresentationListener`, forwarding every event to an **optional** `onPresentation: (context, event) => ...` (see Loading below for the one thing every current page uses it for). A cubit with no async load at all (`AppCubit`) doesn't need `VTPage` — `SettingsPage` just uses a plain `BlocBuilder<AppCubit, AppState>` since `AppCubit`'s state is available synchronously from construction.

**`XState` is a single, immutable, `Equatable` class per cubit holding only real data — never a `sealed class` with one subtype per "mode", and never an error field.** What used to be separate variants (`XLoaded`/`XError`, or `FoodSearchIdle`/`FoodSearchLoaded`/`FoodSearchError`) are now just data fields on one class (`DietState.dailyMacros`, `SleepState.logs`, ...) — an error is never part of state, see Errors below for where it actually lives. `FoodSearchState.results` is nullable — `null` means idle/no search yet, a non-null empty list means a search ran and found nothing; the two are different states with the same field. Every `XState` gets a `copyWith({...})` for updating one or two fields without reconstructing the rest. Pages read `state.field` directly, or pattern-match with `switch (state) { XState(results: final results?) => ..., ... }` when there's more than a plain if/else worth of cases (see `FoodSearchPage`) — never a `switch` over subtypes. This only applies to **Cubit state**; a domain entity that's genuinely a closed set of mutually exclusive shapes (`User` — `AuthenticatedUser` vs `AnonymousUser`, `lib/app/domain/auth/entities/user.dart`) still models that as a `sealed class`, since that's a real sum type, not a "same data plus an error flag" shape — the distinction is state (one cubit's evolving snapshot, data only) vs. entity (a domain concept with fundamentally different shapes).

Any cubit `VTPage` creates that needs to kick off work on mount (typically an initial load) overrides `PresentationCubit.onInit()` — e.g. `DietCubit.onInit() => loadToday();`. Nothing calls `onInit()` itself; `VTPage` calls it automatically, but only from a widget that's a **descendant** of its `BlocPresentationListener`, not from `BlocProvider`'s `create:`. This matters: `emitPresentation` writes to a plain broadcast stream with no replay, so if `onInit()` ran before the listener subscribed, the very first `showLoading` a cubit emits on mount would be emitted into a stream nobody's listening to yet and silently dropped — the overlay would just never appear for the initial load. Flutter guarantees a widget's `initState` completes before any descendant mounts, so calling `onInit()` from a small private `StatefulWidget` placed as `BlocPresentationListener`'s `child` (see `_VTPageOnInit` in `vt_page.dart`) is what makes the ordering safe.

## Loading

No cubit models loading as a state field — it's a pair of presentation events instead, shown as one global overlay instead of each page switching on a loading state. Every feature's `XPresentationEvent` declares its own plain `XShowLoading`/`XHideLoading` classes (no shared base class to extend — `VTPage` doesn't auto-detect loading events centrally, each page wires it explicitly). A cubit emits these around an async load: `emitPresentation(DietShowLoading())` before the await, `emitPresentation(DietHideLoading())` after, then `emit(...)` the real state — see `DietCubit.loadToday()`/`WaterCubit.loadToday()`/`FoodSearchCubit.search()`. Because there's no loading state, a `PresentationCubit`'s **initial state is a "loaded but empty" value** (e.g. `DietCubit`'s initial state is `DietState(dailyMacros: const DailyMacros(entries: []))`, not a null/loading placeholder). Every page that owns such a cubit passes `onPresentation` to `VTPage` and switches on its own event type to call `context.showLoading()`/`context.hideLoading()` explicitly (`lib/app/core/loading/loading_extensions.dart`, wrapping `loader_overlay`'s `context.loaderOverlay`) — see `DietPage`/`WaterPage`/`SleepPage`/`FoodSearchPage`/`AuthPage`/`ProfilePage`'s `onPresentation:` switch. `main.dart` wraps `MaterialApp.router`'s `builder:` in a single `LoaderOverlay` for the whole app — pages/cubits never touch `loader_overlay` directly, only through `context.showLoading()`/`hideLoading()`. The overlay's visual is `VTLoadingOverlayIndicator` (design system, see below).

## Errors

Like loading, an error is never part of `XState` — it's a presentation event, shown as a dialog instead of each page switching on an error state. Every feature's `XPresentationEvent` declares its own `XError` class (`DietError`, `WaterError`, `SleepError`, `FoodSearchError`; `AuthCubit` reuses its existing `AuthActionFailed` for the same role) carrying `message`. A cubit emits it in place of a state change: `emitPresentation(DietError(message: error.message))` — the state stays whatever it already was, so the page keeps showing the last-good (or empty) content underneath the dialog. Every page's `onPresentation` switch adds a case for its error event calling `context.showErrorDialog(message: message, onRetry: context.read<XCubit>().loadX)` (`lib/app/core/error/error_dialog_extensions.dart`) — `onRetry` is omitted where there's nothing sensible to retry (`FoodSearchPage`, since the cubit doesn't hold onto the last query). `DietError` carries a `date` alongside `message` since `DietCubit` can be loading any day, not just today (`DietPage`'s `DietDateSelector` lets the user page through history) — `onRetry` calls `cubit.goToDate(date)` with the date that actually failed, not whatever `state.date` happens to be by the time the user taps retry. The dialog itself is `VTErrorDialog` (design system, see below): a message plus an optional retry action and an OK dismiss action, both localized (`l10n.retry`/`l10n.ok`).

## Toasts

A success confirmation follows the same presentation-event shape as loading/errors, just on the happy path: a cubit emits a one-off event (`FoodLogged`, carrying the data the toast renders — `foodName`, `mealType`) after the work succeeds, and the page's `onPresentation` switch calls `context.showToast(...)` (`lib/app/core/toast/toast_extensions.dart`). `showToast` shows a floating, transparent-background `SnackBar` whose `content` is `VTToast` (design system) — a rounded elevated card with a leading circular accent-tinted icon avatar, a title, and a subtitle, so the toast fully owns its look rather than the Material `SnackBar` chrome. `FoodSearchCubit.logFood` still returns its `Result` to `_LogFoodSheet` (which pops on success) *and* emits `FoodLogged` on success — the sheet's inline error handling and the page's toast are two different concerns on the same call. Reach for this (not a dialog) for a non-blocking "it worked" acknowledgement; keep `VTErrorDialog` for failures that need a decision.

## Design system

Everything under `lib/app/design_system` is prefixed `VT` (`VTColors`, `VTSpacing`, `VTRadius`, `VTTextStyles`, `VTTheme`, `VTGap`, `VTCard`, `VTPrimaryButton`, `VTEmptyState`, `VTErrorDialog`, `VTLoadingOverlayIndicator`, `VTFeatureTile`, `VTProgressBar`, `VTMacroRing`, `VTToast`, `VTSearchField`, `VTAppearEffect`, `VTBadge`, `VTBarChart`, `VTDistributionBar`, `VTLegendDot`, `VTPhotoHeader`, `VTImageSourceSheet`) so design-system pieces are always identifiable at a glance. Never use raw Material spacing/colors/text styles in a page when a `VT` token or component exists — add one if it's missing instead of reaching for `SizedBox`/`Colors.*` directly. Component set is sized to what's actually rendered — add a new one only once a page needs it (e.g. a network image or search field component once the diet feature has food images/search).

Palette: forest green primary (health, nutrition), coral-orange secondary (energy, warmth), warm-neutral surfaces. Typography: Poppins for headings, Inter for body text (both via `google_fonts`).

The screens are meant to feel modern and visually appealing, not just functional — the diet page (see below) is the reference bar for that polish; new features should aim for the same level. Beauty comes from the design system, never from raw Material widgets in a page: when a screen needs a richer visual (a progress ring, a colored badge, a gradient header), add a reusable `VT`-prefixed component for it rather than one-off styling inline, the same rule as spacing/color tokens. The visual vocabulary already established — favor these over inventing a parallel look: circular progress (`VTMacroRing`) for a single headline metric against a goal, `VTProgressBar` (now colorable via its `color`) for secondary/list metrics, soft rounded cards (`VTCard`), a leading circular icon avatar tinted with a category's accent color at ~16% alpha for section headers, and pill-shaped badges (rounded-full container, accent color at ~16% alpha with the accent as text/icon color) for compact figures like a meal's calorie total.

## Charts

Charts are hand-rolled `VT` components under `design_system/components/charts/`, not a charting package (`fl_chart` was considered and dropped for issue #55) — `VTMacroRing` already set the precedent that a `CustomPainter` we own beats a dependency whose look isn't ours, and the two shapes the app needs are simple enough that the package's tooltips/axes would be paid for in theming fights.

- **`VTBarChart`** — vertical bars over time. Every bar is a `VTBarChartBar` holding a list of `VTBarChartSegment`s (`value` + `color`), so a plain bar (one segment) and a stacked bar (protein/carbs/fat) are the same widget on one scale; `VTBarChartBar.empty()` is a day with no data and paints a faint stub rather than a zero-height bar, which keeps gaps legible. An optional `referenceValue` draws a dashed goal line, and the scale always includes it so the goal can't fall off the top.
- **`VTDistributionBar`** — one rounded horizontal bar of shares (segments again), for "what is this made of". Used by `CustomFoodEnergySplitCard` (protein/carbs/fat energy split), `MealSplitCard` (calories per meal) and `RecipeTotalsCard`.
- **`VTLegendDot`** — the colour swatch + label that goes under a chart (`isDashed` for a reference line rather than a series).

Both take `VTBarChartSegment`, so a colour+value pair is the one chart primitive. Bars animate in via `TweenAnimationBuilder` like the rest of the design system. Reach for these before writing a one-off `CustomPainter` in a page.

## Recipes

A recipe is a set of foods eaten together (issue #63). The modelling decision that makes everything else small: **a recipe is stored as an ordinary `foods` row** with `source: 'recipe'`, whose per-100g macros are the roll-up of its ingredients. Logging a recipe is therefore the plain `logFood` path — search, the log sheet, editing, the calendar, adherence and the history charts all keep working untouched, and "add a recipe to my day" needs no new write path at all. The two alternatives were dropped for good reasons: expanding a recipe into one `food_log` per ingredient loses the recipe's identity in the day view, and hanging a nullable `recipe_id` off `food_logs` would force `food_id` nullable and give `FoodLogEntry`/`DailyMacros`/`MacroTotals`/history two cases to handle forever.

`recipes` (user_id + food_id) and `recipe_ingredients` (recipe_id + food_id + quantity_grams) only record **what the recipe was built from**, so it can be listed and its ingredients shown; they carry no macros of their own — the foods row is the single source of truth for the name and the numbers. `recipe_ingredients` has no `user_id`: its RLS policy checks the parent recipe's owner instead of duplicating the column.

**Recipes are personal**, unlike the shared catalog they read from: `searchCatalog` adds `.or('source.neq.recipe,user_id.eq.<me>')`, so a row is kept when it isn't a recipe *or* it's mine. That one predicate is the whole visibility rule — recipes still live in the shared `foods` table, they're just filtered out of everyone else's search.

**Deleting a recipe deliberately leaves its foods row behind.** Past `food_logs` reference it and must keep resolving; the leftover food is no different from any other food you logged once and stopped using.

`SaveRecipeUseCase` is the one place with real orchestration, and it covers **both create and edit** (`recipe: null` means create) — the form is one page doing both, and the ingredient-saving below would otherwise be duplicated across two use cases. An ingredient picked from an Open Food Facts search has no `id` yet, so each id-less ingredient is `saveFood`ed first (the same save-then-log idea as `LogFoodUseCase`); then creating saves the rolled-up `RecipeDraft.toFood()` and writes `recipes` + `recipe_ingredients`, while editing `updateFood`s the recipe's existing foods row in place and replaces its ingredient rows. These are sequential writes, not a transaction — there's no RPC in the project and this matches the existing save-then-log shape, so a mid-way failure can leave an orphan foods row (harmless: it's an unreferenced catalog entry).

**Editing a recipe rewrites what past days show.** Because the recipe *is* its foods row, changing its ingredients changes that row's per-100g macros, and every `food_log` pointing at it — including yesterday's — is re-read through the new numbers. This is the same wart comparable trackers have, and it's the direct cost of the model that makes everything else free. The alternative (a fresh foods row per edit, old logs frozen against the old one) was rejected: the superseded row would still be a recipe-food owned by you, so it would sit in your own search as a duplicate with no recipe behind it.

`RecipeDraft` (name + ingredients, no id) is what the form edits and what computes the roll-up; `Recipe` (id + food + ingredients) is what comes back persisted. Both mix in `MacroTotals`.

**`MacroTotals` was generalised for this**: it now folds over `FoodPortion` (a mixin pairing a `Food` with `quantityGrams`, and deriving calories/protein/carbs/fat/fiber/micronutrients from it) rather than `FoodLogEntry` specifically, so a recipe ingredient and a logged food total up through the exact same code. `FoodLogEntry` and `RecipeIngredient` both mix in `FoodPortion`; `DailyMacros` still narrows `entries` back to `List<FoodLogEntry>` via a covariant override. `MacroTotals` also gained `totalGrams`, which is what the issue's "somar ... gramas" asks for and is meaningful for a day too.

UI: a book icon in the diet `AppBar` opens `RecipesPage` (`/diet/recipes`, list + delete + FAB); tapping a recipe or the FAB opens `RecipeFormPage` (`/diet/recipes/new`), which creates or edits depending on the `Recipe?` in its `RecipeFormExtra`. The form's "add ingredient" pushes `IngredientPickerPage` (`/diet/recipes/new/ingredient`), which reuses `FoodSearchCubit` and `FoodSearchResultTile` but pops a `RecipeIngredient` instead of logging — `FoodSearchPage` stays single-purpose rather than growing a "pick" mode. `RecipeTotalsCard` shows the live roll-up as the draft grows. A recipe can carry a photo, picked and uploaded exactly like a custom food's: the cubit uploads the bytes and puts the resulting URL on the draft, so the domain only ever sees `RecipeDraft.imageUrl` and the raw `Uint8List` stays in `RecipeFormState`.

**`VTPage` gained an optional `cubitParam`** for this: `RecipeFormCubit` needs the recipe being edited at construction, and it's handed to `G<C>(param1: ...)` against a `registerFactoryParam` registration. Passing `param1: null` to a plain `registerFactory` is a no-op, so every other page is unaffected. Reach for this rather than a `create:` callback or seeding a cubit from a widget's `initState` — `VTPage` stays the one place that resolves a page's cubit.

## Favorite foods

A heart on `FoodSearchResultTile` favourites a food, and `FoodSearchPage` splits into **Search / Favorites tabs** (issue #78). `FoodSearchState.tab` holds the selection and `changeTab` sets it — cubit state, not a `TabController`, the same shape as `DietHistoryState.trendRange`. Both tabs stay alive across a switch: results and favourites are separate fields, so flipping tabs never re-queries.

**Favouriting a `Food` is the whole feature — recipes came free.** A recipe *is* a `foods` row (`source: 'recipe'`, see Recipes above), so it favourites through the same insert and lands in the same list as any other food. No second mechanism, no `favorite_recipes` table.

`food_favorites` is a join table (`user_id` + `food_id`, unique together) rather than a flag on `foods`, because `foods` is a shared catalog — "is this a favourite" is per-user and can't live on the row everyone reads. It's private like `recipes`: RLS scopes it to `auth.uid()`, and the `on delete cascade` on both FKs means deleting a food (or a user) takes its favourites with it.

**The heart is optimistic**: `toggleFavorite` flips `favorites` before the request and `_revert`s to the previous list (plus an error dialog) if it fails. Favouriting is cheap and reversible, so making the user watch a spinner — two round-trips for an unsaved Open Food Facts result — buys nothing.

**`FavoriteFoodUseCase` does save-then-favourite** and returns the persisted food: a result straight out of an Open Food Facts search has no `id` yet, and the id is what the favourite points at — the same save-then-act step `LogFoodUseCase` does before logging. That id-less window is why `FoodSearchState.isFavorite` matches an id-less food **by value** and everything else **by id** (`FoodSearchCubit._sameFood`), and why `_favorite` swaps the saved copy into both `results` and `favorites` once it lands. It also creates one real race: un-hearting *during* the save leaves `_unfavorite` with no id to delete by, so it no-ops and `_favorite` settles the server when its id arrives.

`FoodSearchState.favorites` is the single source of truth for both the favourites tab and every result's heart; `loadFavorites()` only runs from `onInit`.

The heart is **optional on the tile** (`onToggleFavorite` nullable, the "pass no callback to disable" convention `MealSectionCard` and `DietCalendarDayCell` already use): `IngredientPickerPage` reuses the same tile to build a recipe, where favouriting isn't the job.

`VTSegmentedTabs` (design system) is the tab strip: a pill track with an indicator that slides between segments via `AnimatedAlign`, rather than Material's `SegmentedButton`, which snaps and brings its own chrome. It's generic over the value type, so `TrendRangeSelector` — still on `SegmentedButton` — could move onto it later.

## Recent searches

The Search tab's idle state lists past queries instead of the "search for a food" prompt, which now only shows for someone who has never searched (issue #86). Tapping one replays it; each row has an × and the header a Clear.

**Device-local, no table** — they're a typing shortcut, not user data worth a round-trip. `RecentSearchesLocalDataSource` (`data/diet/datasources/local/`) sits on `LocalStorageService` under the `diet.recentSearches` key, a sibling of `DietGoalsLocalDataSource`. Reads go through `get<List<dynamic>>(...)?.cast<String>()`: Hive stores the element type but hands the list back untyped, so asking it for a `List<String>` directly throws.

**Only a search that found something is kept.** A typo or a dead end is exactly what the list should spare you from replaying, so `FoodSearchCubit.search` records the query after a hit and skips it on an empty result or an error.

The ordering policy — most recent first, case-insensitive de-duplication (the newly typed spelling wins), capped at `_maxRecentSearches` — lives on `DietRepository`, next to `searchFoods`'s catalog/OFF merge, since both are read-then-decide over a datasource. Each mutating method **returns the updated list** so the cubit doesn't re-read to find out where the query landed.

`onInit` emits recent searches (a synchronous local read) *before* awaiting favourites, so the list is there instantly rather than behind a network call. That's two emits, and the first one is delivered even when it equals the initial state — `Cubit.emit` only de-duplicates after its first emission.

`VTSearchField` gained a `text` property synced in `didUpdateWidget` (the `CustomFoodForm` pattern): replaying a recent search has to fill the box, or you'd see results for "banana" over an empty field. Only a *change* to `text` is pushed in, so typing is never clobbered — `FoodSearchState.query` is set when a search runs, never on keystrokes.

## Copying meals

The diet `AppBar`'s copy icon opens `CopyMealsPage` (`/diet/copy`, issue #61): pick a source day on a calendar, tick which of its meals to bring over, and they're appended to the day the diet page is currently showing. The target day rides in on `CopyMealsExtra` and reaches the cubit through `VTPage`'s `cubitParam` (a `registerFactoryParam`, same as `RecipeFormCubit`) — the page never picks the date itself, it copies into whatever day you were already looking at.

**Copying appends, it never replaces.** Meals already logged on the target day stay put and the copied entries land alongside them, so nothing is destroyed by a mistaken tap; deleting the surplus is the user's job. This is also why there's no confirmation dialog.

A copy is **plain `food_logs` inserts**, not a new kind of row: `SupabaseDietDataSource.copyFoodLogs` rebuilds a `CreateFoodLogRequest` per entry against the target date and bulk-inserts them in one round-trip. The copies therefore reference the same `foods` rows, and the `bump_food_times_logged` trigger counts each one — logging via copy is still logging. Nothing marks a log as "copied": once inserted it's indistinguishable from a hand-logged one, which is what makes editing/deleting/adherence keep working untouched.

`CopyMealsState.macrosByDate` **accumulates across months** rather than being replaced per month (`{...state.macrosByDate, ...loaded}`), which is what lets a day picked in June survive browsing to July — the button names the source date, so an off-screen selection still reads clearly. It also means the source day's meals come free from the month load already in memory; picking a day fires no extra query. The target day is deliberately not tappable as its own source (`CopyMealsMonthGrid` passes a null `onTap` for it).

The calendar reuses `MonthNavigator` and `DietCalendarDayCell` from `pages/diet_history/widgets/` — the cell gained an optional `isSelected` (filled primary circle) and a **nullable** `onTap`, following `MealSectionCard`'s "pass no callback to disable" convention. Its existing "not tappable unless the day has a log" rule is exactly what a source picker wants, so days with nothing logged can't be chosen. The weekday header is the one piece not reused: history's carries a week-average column this page has no use for.

## Diet history page

The diet page's `AppBar` calendar icon opens `DietHistoryPage` (`/diet/history`, issue #55) — a **read-only** analysis surface, deliberately separate from `DietDateSelector`'s `showDietCalendarSheet`, which is a date *picker* for choosing which day to edit. Two calendars, two jobs: one navigates, one explains.

It's one scrolling page (tabs were tried and dropped), stacking a calendar card over a trends section:

- **Calendar** — a month grid built as `DietCalendarWeekRow`s rather than a `GridView`, because each week row ends in a `WeekAverageBadge`: the week's mean calories, tinted by `GoalAdherence` like every other calorie figure. The average counts **only days that actually have food logged** (`CalorieAverage.fromLoggedDays`) — an untracked day means "no data", not "ate nothing", and averaging zeros in would slander every week you forgot to log. A week with nothing logged shows `-`. Day cells are tappable only when that day has a log.
- **Trends** — a `TrendRange` (7/30/90 days) drives `CaloriesTrendCard` (daily bars tinted by adherence against a dashed goal line), `MacrosTrendCard` (stacked protein/carbs/fat), and `MealSplitCard` (`MealCalorieSplit`, calories per meal as a `VTDistributionBar` plus per-meal share and daily average). The range is independent of the displayed month — progression is the point, and paging month-by-month buries it. `DietHistoryCubit` therefore runs two separate range queries (month for the calendar, rolling window for the trends) against the same `GetMacrosInRangeUseCase`.

`DietDayPage` (`/diet/history/day`) is what a calendar day opens. It takes the day's `DailyMacros` and `MacroGoals` through `DietDayExtra` and **has no cubit at all** — the history page already holds every day of the month in memory, so re-fetching one of them would be ceremony. It reuses `MacroSummaryCard` and `MealSectionCard` from `pages/diet/widgets/` so the day reads identically to the live diet page; `MealSectionCard`/`FoodLogTile` take **nullable** `onEditEntry`/`onDeleteEntry`/`onAddFood`, and passing none is what makes them read-only. `initiallyExpanded: true` opens every meal, since the point of the page is seeing everything at once. Importing another page's widgets is the deliberate exception here — the macro/meal cards are diet-domain widgets shared by the diet pages, and neither duplicating them nor hoisting domain-specific cards into the design system would be an improvement.

## Diet page visual language

The diet page (`DietPage`) is the app's showcase screen — the intended look is a modern nutrition tracker (MyFitnessPal / Lifesum family), and it sets the polish bar every other feature should match. Its structure:

- **Macro hero (`MacroSummaryCard`)** — a `VTMacroRing` calorie ring (progress = calories / goal, colored `colorScheme.primary`) sits left with the consumed total, `dietCaloriesOfGoal`, and a `dietCaloriesLeft`/`dietCaloriesOver` delta stacked in its center; the four macros sit right as color-coded `_MacroProgressRow`s (colored dot + label + `progressLabel` + a tinted `VTProgressBar`). Each macro has a fixed accent from `VTColors` — `macroProtein` (coral), `macroCarbs` (amber), `macroFat` (blue), `macroFiber` (teal); reuse these exact colors anywhere macros are shown so the color↔macro mapping stays learnable. The "Vitamins & minerals" toggle is a centered `TextButton.icon` (`unfold_more`/`unfold_less`) below a divider, expanding the sparse micronutrient list — same two-version idea as before (issue #54), just relocated out of the header.
- **Meal sections (`MealSectionCard`)** — each meal leads with a `_MealAvatar` (circular icon tinted by `mealTypeColor`, icon from `mealTypeIcon`, both in `meal_type_icon.dart`) and shows its calorie total as a `_CaloriePill` in the same accent; the whole card stays tap-to-collapse.
- **Logged foods (`FoodLogTile`)** — the row is tap-to-edit (`showEditFoodLogSheet`, see issue #60), keeping the trailing delete `IconButton` as the only other action. Editing covers **quantity and meal type** — the same two fields `LogFoodSheet` captures, so the edit form mirrors the create form; the food itself isn't swappable (delete and re-add instead) and the date is changed by moving the whole day, not one entry. `EditFoodLogSheet` is a separate widget from `LogFoodSheet` rather than one sheet doing both: they hang off different cubits (`DietCubit` vs `FoodSearchCubit`), and abstracting over that would cost more than the duplicated quantity field. It follows `LogFoodSheet`'s shape otherwise — `DietCubit.updateLog` **returns its `Result`** (rather than emitting `DietError`) so the sheet shows the failure inline and pops on success, the same split `FoodSearchCubit.logFood` uses.
- **FAB** — a `FloatingActionButton.extended` (`dietAddFood` label), not a bare icon, so the primary action reads clearly.

`VTMacroRing` and the colorable `VTProgressBar` are deliberately generic (a ring/bar against any goal, any color) so the workout and water features can reuse them for their own headline metrics rather than each rolling its own progress visual.

## Nutrition label scanning

Reading a nutrition label from a photo goes to **Claude, through a Supabase Edge Function** (`supabase/functions/scan-nutrition-label/`, issue #74) — not to on-device OCR. `google_mlkit_text_recognition` used to do this and is gone, along with `TextRecognitionService`, `NutritionOcrDataSource` and the ~125-line regex `NutritionLabelParser`: MLKit gave us text lines with bounding boxes, and everything hard (which number is protein, per-serving → per-100g, kJ → kcal, any language) was the parser's problem. A model that reads the label solves those directly, and the parser and its 10 tests were deleted rather than ported.

**The domain boundary didn't move**: `CustomFoodCubit` → `ScanNutritionLabelUseCase` → `DietRepository.scanNutritionLabel(imagePath)` → `SupabaseNutritionScanDataSource` is the same chain as before, and `ScannedNutritionFacts` is unchanged. Only the datasource swapped, which is the point of the layering.

**The Edge Function exists because of the API key.** An Anthropic key shipped in a mobile binary is extractable by anyone who downloads the app, so the key lives as a Supabase secret and the app only ever calls the function. The function verifies the caller's JWT, which the app's anonymous session already supplies. Deploying it and setting the secret is a runbook in [`docs/nutrition-scan-setup.md`](docs/nutrition-scan-setup.md) — none of it can live in the repo, and scanning fails until both steps are done (the form still works; scan only pre-fills fields you can always type).

`SupabaseFunction` (`core/services/supabase/supabase_function.dart`) pairs the Dart case with the deployed function's name exactly like `SupabaseTable`/`SupabaseBucket` do, and `SupabaseService.invoke` takes the enum — a new function means an enum case, not a raw string at the call site.

The function pins **`claude-opus-4-8` with structured outputs** (`output_config.format` + a JSON schema): the schema is what guarantees the response parses into `ScannedNutritionFacts`, so there's no defensive parsing on the Dart side. Every nutrient is nullable and the prompt says to return null rather than guess — a wrong macro silently poisons a shared catalog row, so a blank field the user fills in is the better failure. The per-100g conversion, unit handling and "is this even a label" judgement all live in that prompt now, which is the real tradeoff: **it's no longer covered by Dart tests.** `DietRepository.scanNutritionLabel` is a thin pass-through, and the two repository tests that used to exercise the parser through it were removed.

Scanning now needs network and costs roughly $0.01–0.02 per scan; the old OCR was free and offline. That was the deliberate trade for a scan that actually works on a real label.

## Custom food page

Adding a food the catalog doesn't have is a **full page** (`CustomFoodPage`, `AppRoute.customFood`), not a bottom sheet (see issue #71) — the form is too tall for a sheet once it carries a photo, a name, a brand, a scan action and five nutrient fields. `FoodSearchPage`'s app-bar action does `await context.pushRoute<Food>(.customFood)` and, on a non-null return, opens `showLogFoodSheet` — the page itself never logs anything, it only builds an unsaved `Food` and pops it back (`Navigator.of(context).pop(food)` from the `CustomFoodReady` presentation event, matching how every other page pops). Persisting still happens in `LogFoodUseCase`'s save-then-log, exactly as before.

**The page holds no logic — `CustomFoodCubit` does.** It owns validation (`CustomFoodState.isComplete`), number parsing (comma-or-dot, via `nutrientChanged`), the photo pick, the label scan, the image upload, and building the `Food`. The five nutrients are a `Map<CustomFoodNutrient, double>` on the state rather than five nullable fields: an absent key *is* "empty" (which `copyWith` could never express for a nullable `double`), and one enum drives the fields, the scan mapping (`CustomFoodNutrient.valueOf(facts)`), the labels and the macro accent colors — adding a nutrient to the form is one enum case.

`CustomFoodForm` is the only stateful piece, owning the `TextEditingController`s. It pushes every keystroke into the cubit and **syncs back from state in `didUpdateWidget`**, comparing each controller's *parsed* value against the state's — so a scan that fills the fields flows through state like any other change, while a half-typed `"5."` (parses to null, same as the state) never gets clobbered mid-typing. There's deliberately no "fields were scanned" presentation event; state is the single path.

Visually it follows the diet page's vocabulary: a stretchy `SliverAppBar` whose background is `CustomFoodPhotoHeader` (photo or an `add_a_photo` placeholder over a `primaryContainer→surface` gradient, `StretchMode.zoomBackground` plus `BouncingScrollPhysics` so it zooms on overscroll on both platforms), a `CustomFoodScanCard` using the leading circular accent avatar, nutrient fields dotted in their macro accent, a live `CustomFoodEnergySplitCard` (protein/carbs/fat energy shares at 4/4/9 kcal per gram, `AnimatedSize`d in once a macro is typed), and staggered `VTAppearEffect`s. The page has **no title** — the photo is the hero. Loading (upload/scan) is the global overlay, never a state field; errors, the incomplete-form warning and the "couldn't read the label" case are all presentation events the page turns into `VTErrorDialog`s (the l10n strings live in the page, not the cubit).

## Navigation

No bottom nav bar — `HomePage` (`/`) is the single entrypoint: a settings action in the `AppBar` plus a grid of `VTFeatureTile`s (one per feature) that push their feature route. Each feature page owns its own `Scaffold`/`AppBar`; there is no shared shell.

Routes are never referenced by raw path string outside of `app_router.dart`/`routing/routes/`. `AppRoute` (`lib/app/presentation/routing/app_route.dart`) is an enum pairing each route's name with its path. `VTRoute` (`lib/app/presentation/routing/vt_route.dart`) is an abstract class exposing `route` (the `AppRoute`), `builder`, and an optional `redirect` (a real `GoRouterRedirect`, `null` by default) as getters to override, plus a concrete `toGoRoute()` converting itself to a `GoRoute`. Each route is its own concrete class in `lib/app/presentation/routing/routes/` (`DietRoute`, `WaterRoute`, `HomeRoute`, ...), one file per route, overriding only what it needs — most just override `route`/`builder`; `HomeRoute` is the one that also overrides `redirect`. The onboarding gate is that one route-specific `redirect`, not a router-wide one: `HomeRoute` takes a `HasSeenOnboardingUseCase` constructor dependency and redirects to `AppRoute.onboarding.path` unless it returns true, since `home` is the app's sole entrypoint (`initialLocation`) — there's no need to also guard every other route individually. `AppRouter._routes` (`app_router.dart`) is a `List<VTRoute>` instantiating each of these classes (`HomeRoute(hasSeenOnboardingUseCase: G())`, the rest with no arguments), turned into `GoRouter`'s `routes:` with `.map((route) => route.toGoRoute()).toList()`. Elsewhere, navigate with the `BuildContext` extension in `lib/app/core/navigation/navigation_extensions.dart`: `context.pushRoute(.diet)` / `context.goRoute(.settings)`, optionally with `extra:` for arguments. Add a new feature by adding a case to `AppRoute`, a new `XRoute` class in `routing/routes/`, and its entry in `AppRouter._routes`, plus a tile to `HomePage`.

## App-level state

The user's cross-cutting preferences — locale override (`AppSettings.locale`, null = follow system), `themeMode`, and `unitSystem` (`core/units/unit_system.dart`'s `UnitSystem.metric`/`.imperial`, default metric) — live in `AppSettings` (`lib/app/domain/settings/entities/app_settings.dart`), a domain entity, not a presentation-only type. `SettingsRepository` (`lib/app/data/settings/settings_repository.dart`) wraps `SettingsLocalDataSource` (see Local storage) the same way every other feature repository wraps its datasource; `GetAppSettingsUseCase`/`SaveAppSettingsUseCase` (`lib/app/domain/settings/use_cases/`) are the only way anything reads or writes it. `AppCubit` (`lib/app/cubit/app_cubit.dart`) is `PresentationCubit<AppSettings, AppPresentationEvent>` — its Bloc state *is* the domain entity directly (no separate wrapper), constructed via `GetAppSettingsUseCase` and persisted via `SaveAppSettingsUseCase` on every change. It's a GetIt singleton provided once at the root in `main.dart` via `BlocProvider.value(value: G<AppCubit>(), ...)` wrapping `MaterialApp.router`. `SettingsPage` is its only UI consumer, changing locale/theme/units through `RadioGroup`s. **Any other cubit that only needs to *read* settings (not display/edit them) depends on `GetAppSettingsUseCase` directly instead of reading `AppCubit`** — e.g. `WaterCubit.unitSystem` and `FoodSearchCubit.unitSystem` are plain getters calling `_getAppSettingsUseCase().unitSystem`, and `WaterPage`/`LogFoodSheet` read `cubit.unitSystem` off the cubit they already depend on rather than also reaching for `context.read<AppCubit>()`. This keeps `AppCubit` consumption limited to the one page that's actually about editing settings.

`UnitSystem` has two independent conversion extensions in the same file: `WeightConversion` (grams/ounces — food quantity consumed, `LogFoodSheet`) and `VolumeConversion` (milliliters/fluid ounces — water intake, `WaterPage`/`AddWaterSheet`/`EditWaterGoalDialog`). Both convert to their metric base unit before it ever reaches the domain (`FoodLog.quantityGrams`, `WaterLog.amountMl`) — the base unit stays the source of truth end-to-end, same principle as storing locale/theme as their real types rather than display strings. `WeightConversion` deliberately does not touch a food's per-100g macros (`CustomFoodPage`, `dietNutritionPer100gTitle`, ...) — "per 100g" is a nutrition-label convention independent of the reader's preferred unit system, not a quantity to convert.

## Internationalization

Standard Flutter `gen-l10n` (not a custom solution). ARB files live in `lib/l10n/arb/` (`app_en.arb` is the template, plus `app_pt.arb`). Never hardcode user-facing strings — add a key to both ARB files and read it via `context.l10n` (`BuildContextLocalizationExt`, `core/localization/localization_extensions.dart`), not `AppLocalizations.of(context)` directly — the same file also has `context.materialLocalizations` for the rare case that needs Flutter's own localized strings (date/time formatting) instead of the app's. Run `flutter gen-l10n` (or `flutter pub get`, which triggers it) after editing an ARB file. Add more locales the same way dofus_buddy added `fr` — a new ARB file plus the corresponding `RadioListTile` in `SettingsPage`.

## Code style

- No comments. Code must be self-explanatory through naming; if it isn't, restructure it rather than annotate it.
- No variables named `result` (or similarly generic — `data`, `response`, `value` as a local var name). Name it after what it actually holds. A local variable holding a not-yet-unwrapped `Result<F, S>` gets a `Result` **suffix** instead — `dailyMacrosResult`, `loggedResult`, `deletedResult` — not a bare/generic name.
- `Result<F, S>` (`lib/app/core/error/result.dart`) is consumed via `.when(failure, success)`, not a `switch`/pattern match — `T when<T>(T Function(F error) failure, T Function(S value) success)`, positional (not named) with `failure` always first: `dailyMacrosResult.when((error) => emit(XError(message: error.message)), (value) => emit(XLoaded(...)))`. When one branch is synchronous and the other returns a `Future` (e.g. failure just calls `emit(...)`, success awaits a reload), wrap the sync branch to match rather than mixing return types — `Future.sync(() => emit(XError(...)))` for a `void`-returning cubit method, `Future.value(Failure(error))` for one returning `Result` itself — see `WaterCubit.deleteLog()`/`LogFoodUseCase.call()`. Prefer this over `Future.sync` when the two branches would otherwise need unrelated wrapping just to satisfy `.when()`'s single return type: extract the value both branches actually agree on instead (e.g. `final error = result.when((error) => error, (_) => null);` then branch with a plain `if` outside `.when()`) — see `DietCubit.deleteLog()`.
- One widget per file. Never declare a private `_Widget` helper class alongside another widget in the same file — extract it to its own file under the feature's `widgets/` folder as a public (non-underscore) widget, and import it. A page file holds only its page widget; each supporting piece (`ProfileHeader`, `ProfileMenuTile`, `MacroPill`, ...) is its own file. The `show...` launcher function for a modal/sheet stays in the same file as the widget it launches (`showFoodDetailsDialog` with `FoodDetailsDialog`) — that's a function, not a second widget.
- Use the dot-shorthand operator wherever the target type is inferable (`.center`, `.w700`, `.light`).
- `flutter analyze` must report zero issues (including `info`-level lints) before considering a change done.
- Lint rules are the explicit list in `analysis_options.yaml`, carried over from the dofus_buddy project.

## Testing

Mirrors `lib/`. No `late` variables, ever — no `setUp`-populated shared state either. Every test builds its own instances inline, at the top of the `test`/`build:` body that uses them, via the factories/fixtures below. This is stricter than "avoid `late`": a `final` declared in `setUp()` and closed over by every test in the file is the same shared-mutable-state problem `late` causes, just spelled differently.

- `test/mocks/`: one file per layer (`repositories_mocks.dart`, `use_cases_mocks.dart`, `datasources_mocks.dart`, `services_mocks.dart`, ...), each a one-line `class MockX extends Mock implements X {}`.
- `test/factories/`: one `abstract class XFactory`/`XFactories` per file, `static` build methods only — never instantiate the class, and it reads better in autocomplete (`FoodFactory.build(...)`, `CubitsFactories.buildDietCubit(...)`) than a pile of same-named top-level functions. **No stubbing inside a factory, ever** — a factory either takes an already-configured instance or defaults to a bare, unstubbed `MockX()`; all `when()`/`verify()` setup happens in the test that needs it. Entity factories in `test/factories/entities/` (`FoodFactory`, `FoodLogFactory`, ...); use case factories in `UseCasesFactories` (`test/factories/use_cases_factories.dart`); cubit factories in `CubitsFactories` (`test/factories/cubits_factories.dart`) — only add a cubit factory once the cubit takes constructor dependencies worth defaulting. Repositories are normally thin pass-throughs not worth a dedicated test, but one with real branching logic of its own (`DietRepository.searchFoods`'s catalog-then-Open-Food-Facts fallback) gets tested the same way a cubit does — by mocking its own dependencies (here, the datasources) rather than the repository itself; `RepositoriesFactories` (`test/factories/repositories_factories.dart`) is that shape's factory file, following the same defaults-to-a-bare-mock convention. Note the consequence: if a cubit's constructor calls a mocked dependency synchronously (`AppCubit` calls `getThemeMode()` at construction), the bare-default path isn't usable as-is — the test must always pass a pre-stubbed mock, same as every other case.
- `test/fixtures/`: builders that need real I/O a mock can't stand in for (e.g. `local_storage_fixture.dart`'s `openTestHiveBox()`/`buildTestLocalStorageService()`, which open a real Hive box against a temp directory). Register cleanup with `addTearDown()` *inside* the fixture function itself — it hooks into whichever test is currently running, so callers never juggle a paired teardown.
- Cubits: `bloc_test`'s `blocTest`, building every mock and the cubit inside `build:` (not a separate `setUp:` — that would need `late` to share the mock between the two callbacks). If a mock must also be reached from `verify:`, declare it as a `final` local right above that one `blocTest(...)` call, not hoisted to the file/group level. A `PresentationCubit`'s state-transition tests stay in `blocTest` as usual (its `expect:` no longer includes a loading state, since there isn't one); its loading events get their own `blocPresentationTest` (from `bloc_presentation_test`, same file, same `build:` shape) asserting `expectPresentation: () => [isA<XShowLoading>(), isA<XHideLoading>()]` for that cubit's own presentation event type — see `diet_cubit_test.dart`/`water_cubit_test.dart`/`food_search_cubit_test.dart`.

## Releases (TestFlight)

`git tag v1.2.3 && git push origin v1.2.3` is the whole release ritual (issue #81). `.github/workflows/testflight.yml` fires on `v*` tags, runs on `macos-15`, and hands off to `ios/fastlane`'s `beta` lane — all the build logic lives in the Fastfile, the workflow only supplies secrets and the tag.

**The tag is the source of truth for the version**, not `pubspec.yaml`: the lane parses `v1.2.3` → `1.2.3` and rejects anything else outright, so a typo'd tag fails in seconds instead of uploading a mislabelled build. pubspec's `version:` stays a default for local builds and is deliberately never read by the lane. The build number comes from `latest_testflight_build_number + 1` — asking TestFlight is the only source that can't collide with a build already up there, whichever tag is being cut.

The version reaches the binary through Flutter, not Xcode: `flutter build ios --release --no-codesign --build-name/--build-number` writes `ios/Flutter/Generated.xcconfig`, which `Info.plist`'s `$(FLUTTER_BUILD_NAME)`/`$(FLUTTER_BUILD_NUMBER)` read. That's why the lane shells out to Flutter first and lets `build_app` do the archiving — the archive is produced once, by fastlane.

**Signing is `match`** (certs/profiles in a separate private git repo, encrypted), and CI runs it `readonly: true` — CI must never mint a certificate, since that silently invalidates the existing one. The checked-in Xcode project keeps `CODE_SIGN_STYLE = Automatic` so local `flutter run` works untouched; the lane flips it to manual with the match profile just before archiving and leaves the checkout dirty, which is free on a throwaway CI runner.

`.env` is a **bundled asset** (see `pubspec.yaml`) and gitignored, so the workflow writes it from secrets — a build without it ships an app that can't reach Supabase. A missing secret reads as an empty string rather than an error and `Env` would only blow up once a tester launches the build, so the step fails fast on an empty one instead.

`ITSAppUsesNonExemptEncryption = false` in `Info.plist` matters more than it looks: without it every upload stops in TestFlight waiting for the export compliance form to be answered by hand before testers can install. Vitta only talks HTTPS, which is exempt.

**One-time setup** (Apple app record, the private `match` certificates repo, the App Store Connect API key, and the nine GitHub secrets) is a runbook in [`docs/testflight-setup.md`](docs/testflight-setup.md) — the pipeline fails until every step there is done, and none of it can live in the repo.

Android carries the same `com.rodrigobastosv.vitta` id (`namespace` + `applicationId` in `android/app/build.gradle.kts`, plus the `MainActivity.kt` package path) purely for consistency — there is no Play pipeline yet, so nothing consumes it.
